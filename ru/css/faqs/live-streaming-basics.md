# Основы прямого вещания

### Что такое трансляция, прямое вещание и видео по требованию?

- **Трансляция**: процесс передачи хостом локального видео и аудио на серверы Tencent Video Cloud. В некоторых сценариях это называется «RTMP publishing».
- **Прямое вещание**: во время прямого вещания видеопотоки генерируются в реальном времени. Это работает только если кто-то осуществляет трансляцию. URL прямого вещания становится недействительным в момент, когда хост прекращает трансляцию. Поскольку прямые потоки воспроизводятся в реальном времени, зрители не могут видеть полосу прогресса во время прямого вещания.
- **Видео по требованию (VOD)**: VOD позволяет воспроизводить файлы в облаке. Файл может быть воспроизведен в любой момент, если он не удален поставщиком (например, Tencent Video). Поскольку видео уже хранятся на сервере, зрители могут видеть полосу прогресса во время воспроизведения.

### Какие требования к доменному имени воспроизведения в CSS?

Доменное имя может содержать до 45 символов и не может содержать прописные буквы. Дополнительную информацию см. в разделе [Добавление доменного имени](https://www.tencentcloud.com/document/product/267/35970).

### Могу ли я использовать одно доменное имя для воспроизведения и трансляции? Могу ли я использовать доменные имена второго уровня?

Вы должны использовать разные доменные имена для воспроизведения и трансляции, но вы можете использовать одно доменное имя второго уровня, чтобы указать, что поток одинаковый.
Например, вы можете использовать `123.abc.com` для трансляции и `456.abc.com` для воспроизведения.

### Какие протоколы трансляции поддерживаются?

Протоколы потоковой трансляции, поддерживаемые CSS, включают: RTMP, SRT, WebRTC и RTMP over SRT.

- **Протокол RTMP:** RTMP постепенно вытесняется другими низкозадержливыми протоколами в области прямого вещания, но это наиболее распространенный протокол, используемый для потоковой трансляции (передачи данных от хостов на серверы). В настоящее время отечественные сервисы облачного видео используют RTMP как основной протокол трансляции, а SDK потоковой трансляции (например, TXLivePusher) также называется RTMP SDK.
- **Протокол SRT:** TS over SRT напрямую передает TS потоки, содержащие данные аудио/видео, используя протокол SRT. Существующая система прямого вещания используется для воспроизведения. TS over SRT используется как стандартный формат трансляции для оборудования Haivision и OBS.
- **Протокол WebRTC:** SDK TXLivePusher используется для трансляции потоков в CSS. Он может передавать аудио/видео, которые браузер захватывает с камеры, совместного использования экрана или локального файла, на серверы прямого вещания через WebRTC. Одновременно он позволяет осуществлять локальное смешивание собранного контента перед отправкой его на серверный сервер. Вы можете использовать протокол WebRTC для прямого вещания в веб-среде. На стороне ПК вы также можете использовать инструмент OBS для прямого вещания по WebRTC. Дополнительную информацию см. в разделе [Прямое вещание OBS WebRTC](https://www.tencentcloud.com/document/product/267/57042). Преимущество использования Web для прямого вещания по WebRTC заключается в том, что не требуется установка дополнительного программного обеспечения, и вы можете работать прямо в браузере.
- **RTMP over SRT**: Транспортный уровень заменяет TCP на SRT. Благодаря низкой задержке SRT и высокой устойчивости к потере пакетов, он значительно снижает влияние RTMP на качество передачи при плохой сети и колебаниях, улучшая общее качество просмотра прямого вещания.

### Какие протоколы воспроизведения поддерживаются?

Распространенные протоколы прямого вещания включают RTMP, HTTP-FLV, HLS и WebRTC.

- **RTMP** можно использовать как для прямой трансляции, так и для прямого воспроизведения. Он работает путем разделения больших видео и аудиофрагментов на короткие фрагменты и их транспортировки в виде небольших пакетов данных через Интернет. RTMP поддерживает шифрование и, таким образом, обеспечивает конфиденциальность. Однако сложные процессы разделения и сращивания добавляют неопределенность в стабильность передачи данных в сценариях высокой одновременности.
- **HTTP-FLV** разработан компанией Adobe Systems и представляет собой довольно простой формат видео. Он работает путем добавления заголовка к большим видео и аудиофрагментам. Эта простота делает его отличным в контроле задержки и производительности при высокой одновременности. Единственным недостатком является то, что HTTP-FLV плохо поддерживается на мобильных браузерах, но это идеальный вариант для мобильных приложений.
- **HLS** выпущен компанией Apple. Он разделяет видеопотоки на сегменты длительностью 5-10 секунд и управляет ими с помощью плейлистов M3U8. Протокол обеспечивает плавное воспроизведение по мере загрузки клиентами пакетов данных 5-10 секунд. Однако он имеет высокую задержку около 10-30 секунд. В отличие от HTTP-FLV, HLS хорошо поддерживается на iPhone и большинстве браузеров Android и поэтому часто используется для совместного использования URL в QQ и WeChat Moments.
- **WebRTC** расшифровывается как Web Real-Time Communication. Это API, который позволяет осуществлять вызовы в реальном времени с аудио/видео на веб-браузерах. Поддерживаемый Google, Mozilla и Opera, спецификации WebRTC были опубликованы Консорциумом World Wide Web (W3C) 1 июня 2011 года. WebRTC используется LEB, который является версией LVB с ультра-низкой задержкой и предлагает потоковую передачу с миллисекундной задержкой. Это подходит для сценариев с высокими требованиями к задержке, таких как онлайн-образование, трансляция спортивных событий и онлайн-викторины.

| Протокол | Преимущества | Недостатки | Задержка воспроизведения |
| --- | --- | --- | --- |
| HTTP-FLV | Зрелый, подходит для сценариев высокой одновременности | Требуется интеграция SDK. | 2-3s |
| RTMP | Относительно низкая задержка | Плохая производительность в сценариях высокой одновременности | 1-3s |
| HLS (M3U8) | Хорошо поддерживается на мобильных браузерах | Высокая задержка | 10-30s |
| WebRTC | Самая низкая задержка | Требуется интеграция SDK. | < 1s |

### Какой формат у URL воспроизведения?

```
rtmp://domain/AppName/StreamName?txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)http://domain/AppName/StreamName.m3u8?txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)http://domain/AppName/StreamName.flv?txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)https://domain/AppName/StreamName.m3u8?txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)https://domain/AppName/StreamName.flv?txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)webrtc://domain/AppName/StreamName?txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)
```

- **Префикс**
RTMP: **rtmp://**
HTTP-FLV: **http://** или **https://**
HLS: **http://** или **https://**
WebRTC: **webrtc://**
- **Имя приложения (**`AppName`**)**
Имя приложения указывает путь хранения файла прямого вещания. По умолчанию это **live**.
- **Имя потока (**`StreamName`**)**
Имя потока (`StreamName`) уникально идентифицирует прямой поток.
- **Ключ аутентификации и другие пользовательские параметры**
Ключ аутентификации: **txSecret=Md5(key+StreamName+hex(time))&txTime=hex(time)**.

### Какие распространенные методы трансляции?

- **Камера на устройствах Android/iOS**: используйте стороннее программное обеспечение или [MLVB SDK](https://www.tencentcloud.com/products/mlvb) для захвата видео с камеры и передачи видеопотока на ваш URL трансляции.
- **Камера или программа записи экрана на ПК**: используйте стороннее программное обеспечение для захвата видео с камеры или записи экрана, а затем передачи данных на ваш URL трансляции. Такое стороннее программное обеспечение включает [OBS (рекомендуется)](https://www.tencentcloud.com/document/product/267/31569), XSplit, FMLE и т. д.
- **Устройство видеозахвата**: подключите видеокамеру высокой четкости с выходом HDMI или SDI к кодировщику и передавайте RTMP потоки приложениям прямого вещания. Вам нужно установить адрес публикации RTMP кодировщика на ваш URL трансляции.
Если вы используете веб-камеру, которая поддерживает RTMP, вы также можете установить адрес публикации RTMP веб-камеры на ваш URL трансляции.
- **Преобразование видеофайлов в видеопотоки**: читайте видеофайлы и передавайте их как RTMP потоки на ваш URL трансляции RTMP. Это можно достичь с помощью команды `FFmpeg`, которая работает на Windows, Linux и macOS.

### В чем разница между прерыванием потока и отключением потока?

- **Прерывание потока**: если прямой поток прерван, трансляция остановится, и зрители не смогут смотреть поток. Однако хост может снова запустить трансляцию, чтобы возобновить прямое вещание.
- **Отключение потока**: если прямой поток отключен, трансляция остановится, и зрители не смогут смотреть поток. Хост не может снова запустить трансляцию. Вы можете отключить поток на странице управления потоком консоли CSS. Отключенные потоки можно найти в списке отключенных потоков. Вы можете нажать **Enable**, чтобы включить отключенный поток.

### Могу ли я включить текстовый чат для прямого вещания?

Да. Вы можете использовать компонент Instant Messaging (IM) для реализации текстового чата для прямого вещания. Кроме того, IM предоставляет комментарии на экране, лайки, подарки, уведомления о повторе и другие интерактивные функции. Вы также можете использовать функцию управления комнатой для реализации совместного телевещания, управления идентификаторами пользователей и разрешением на отключение звука членов и других функций.

### Является ли CSS программным обеспечением?

Нет. CSS предоставляет API для разработки приложений прямого вещания.

### Как узнать количество зрителей прямого вещания?

Вы можете вызвать API CSS v3.0 [DescribeStreamPlayInfoList](https://www.tencentcloud.com/document/product/267/37297) для получения количества онлайн-зрителей.

### Какие протоколы потоковой передачи поддерживаются для взаимного преобразования и доступа ретрансляции?

Взаимное преобразование и доступ ретрансляции поддерживаются для общих протоколов потоковой передачи, таких как RTP, SRT, RTMP и UDP.

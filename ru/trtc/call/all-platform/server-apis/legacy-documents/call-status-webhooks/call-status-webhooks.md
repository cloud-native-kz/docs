# Webhooks статуса вызовов

Для обеспечения детального управления услугами вызовов TRTC Call (TUICallKit) предоставляет callbacks статуса вызовов. Ваш серверный компонент может использовать эти callbacks для отслеживания результатов вызовов пользователей в реальном времени, таких как пропущенные вызовы, отклонения и т.д., и на основе этого выполнять аналитику данных в реальном времени и другие операции. Для методов конфигурации вызовов см. [Список API конфигурации Callback](https://www.tencentcloud.com/document/product/647/60135#).

## Условия использования

- Только приложения (SDKAppId), которые активировали **Версию группового вызова** TRTC Call, могут использовать callback статуса вызовов. Вы также можете бесплатно активировать пробную версию для тестирования. Для описания версий и инструкций по активации см. [Активация сервиса](https://www.tencentcloud.com/document/product/647/59832#).
- В настоящее время callbacks статуса вызовов доступны только в определенных версиях TUICallKit на различных платформах, как указано в таблице ниже:

| Платформа/Framework | Номер версии |
| --- | --- |
| Android/iOS/Flutter/uni-app (клиент) | ≥ 1.7.1 |
| Web | ≥ 1.4.6 |
| WeChat Mini Program | ≥ 1.5.1 |

> **Примечание:** Соответствующую информацию о вызовах можно просмотреть в консоли только после обновления всех участвующих платформ/frameworks до упомянутых выше версий.

## Примечания

- Для включения callback необходимо сконфигурировать URL callback и включить переключатель для этой команды. Пожалуйста, обратитесь к [Создание конфигурации Callback](https://www.tencentcloud.com/document/product/647/60134#).
- Направление callback идет от серверной части Callkit к серверной части приложения через HTTP POST запрос.
- После получения запроса callback серверная часть приложения должна проверить, что SDKAppID, содержащийся в URL запроса, соответствует своему собственному SDKAppID.

## Сценарии, которые могут инициировать этот callback

Действия, выполняемые пользователями приложения через клиент во время вызова, например завершение вызова и т.д.

## Время callback

После завершения вызова.

## Возможные результаты callback

| Статус callback | Указание результата |
| --- | --- |
| Пропущенный вызов: истечение времени ожидания получателя перед ответом | not_answer |
| Отклоненный вызов: получатель отклонил вызов | reject |
| Занято: линия занята | call_busy |
| Отмена: вызывающий отменил вызов до соединения | cancel |
| Завершение: вызов соединен и завершен нормально | normal_end |
| Прерывание: вызов прерван из-за сетевых или других проблем | interrupt |

## Описание API

### URL запроса

В следующем примере настроенный в приложении URL callback — это `https://www.example.com`.
**Пример:**

```
$http://www.example.com?sdkappid=$sdkappid&command=$command&contenttype=json&clientip=$clientip&optplatform=$optplatform
```

### Параметры запроса

| Параметр | Описание |
| --- | --- |
| http | Протокол запроса — HTTPS или HTTP, метод запроса — POST |
| www.example.com | URL callback |
| sdkappid | SDKAppID, назначенный консолью Instant Messaging при создании приложения |
| command | Пожалуйста, обратитесь к: [Список команд callback](https://www.tencentcloud.com/document/product/647/60130#0ef77eb4-15d0-4bac-9448-57533360b554) |
| contenttype | Фиксированное значение: json |
| clientip | IP адрес клиента, например 127.0.0.1 |
| optplatform | Платформы клиента могут включать iOS, Android, Web, miniProgram |

Специфическое содержание callback включено в тело HTTP запроса. Подробнее см. примеры callback ниже.

### Пример Callback

Пример webhook запроса:

```
POST /?sdkappid=8888888&command=call_end&contenttype=json&clientip=127.0.0.1&optplatform=iOS HTTP/1.1Host: www.example.comContent-Length: 337{    "UserId": "Alice",	"RoomId": "Alice's Room",    "TotalNum": 2,    "MediaType": "audio",    "CallType": "single",    "CallId": "aheahfo-eqwnknoihfsd-qweqf",    "Role": "caller",    "CallResult": "normal_end",	"EventTime": 170485688,    "StartCallTs": 1704856873,    "AcceptTs": 1704856876,    "EndTs": 1704856885}
```

### Поля пакета запроса

| Поле | Тип | Описание |
| --- | --- | --- |
| UserId | String | ID пользователя, выполняющего операцию |
| RoomId | String | ID комнаты, в которой выполняется операция |
| TotalNum | Integer | Количество участников вызова |
| MediaType | String | Тип медиа: `Audio` Голосовой вызов, `Video` Видеовызов |
| CallType | String | Тип вызова: `Single` Один на один, `Group` Групповой |
| CallId | String | Уникальный ID вызова |
| Role | String | Роль: `Caller` ID пользователя-инициатора, `Callee` ID пользователя-получателя |
| CallResult | String | Результат вызова, заполняется только при событии завершения, в противном случае пусто: `Cancel` Отмена: вызывающий отменил вызов перед соединением, `Reject` Отклонение: получатель отклонил вызов, `Not_answer` Пропущенный вызов: получатель не ответил вовремя, `Normal_end` Завершение: вызов соединен и завершен нормально, `Call_busy` Занято: линия была занята во время вызова, `Interrupt` Прерывание: вызов прерван из-за сетевых или других проблем |
| EventTime | Integer | Временная метка операции (уровень секунды) |
| StartCallTs | Integer | Временная метка инициирования вызова (уровень секунды), возвращается только при normal_end |
| AcceptTs | Integer | Временная метка ответа на вызов (уровень секунды), возвращается только при normal_end |
| EndTs | Integer | Временная метка завершения вызова (уровень секунды), возвращается только при normal_end |

> **Примечание:** Поле CallResult показывает все типы только для индивидуальных вызовов, для групповых чатов только `normal_end`.

### Пример ответа Callback

Пакет ответа callback отправляется после синхронизации данных серверной частью приложения.

```
{    "ErrorCode": 0,    "ErrorMessage": "Success"}
```

### Описание полей пакета ответа

| Поле | Тип | Описание |
| --- | --- | --- |
| ErrorCode | Integer | 0 указывает на успех, все остальные значения указывают на ошибку |
| ErrorMessage | String | Ваш сервер может отправить пакет ответа с информацией об ошибке, как определено |

> **Примечание:** Рекомендуется, чтобы ваш сервер оперативно ответил на правильный пакет ответа после получения пакета запроса правильно, в противном случае это вызовет повторную попытку системы. Механизм повторных попыток следующий.

## Механизм повторных попыток

- Когда инициируется повторная попытка

ErrorCode = 0 считается успешным callback, в противном случае инициируется повторная попытка.

Ошибка HTTP запроса к вашему appServer также инициирует повторную попытку.

- Интервал повторных попыток

Интервал повторных попыток: `0s, 1s, 1s, 2s, 3s, 5s`. Если все попытки не удаются, процесс будет прерван.

## Код ошибки

| Код ошибки | Описание |
| --- | --- |
| 0 | Запрос успешен |
| 50001 | Текущее приложение должно приобрести пакет TUICallKit Group Call Version для использования |
| 999 | Callback не существует |
| 70001 | Недействительные параметры запроса, пожалуйста, проверьте, не отсутствуют ли обязательные параметры или введены неправильно |
| 70002 | UserSig недействителен |
| 70003 | UserSig истек |
| 70004 | Запрашивающий пользователь не является суперадминистратором |
| 70005 | Частота запросов ограничена |
| Неизвестный код ошибки | Внутренняя ошибка системы, пожалуйста, [Отправьте тикет](https://console.intl.cloud.tencent.com/workorder/category) для связи с техническим специалистом |

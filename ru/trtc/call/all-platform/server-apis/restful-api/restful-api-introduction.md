# Введение в RESTful API

REST API — это серверный HTTP-API управления пакета CallKit SDK. Его основная цель — предоставить разработчикам простой портал управления.

В целях безопасности REST API предоставляет только API на основе HTTPS.

## Предварительные условия

Для вызова REST API необходимо приобрести или получить требуемый пакет для CallKit SDK.

> **Предупреждение:** Если вы используете устаревшие интерфейсы `TUICallKit.call()` или `TUICallKit.groupCall()` для инициирования вызова, пожалуйста, ознакомьтесь с каталогом **Документация по устаревшим функциям**. Если у вас есть вопросы, вы можете связаться с: info_rtc@tencent.com.

## Метод вызова

### URL запроса

Формат URL REST API выглядит следующим образом:

```
https://xxxxxx/$ver/$servicename/$command?sdkappid=$SDKAppID&identifier=$identifier&usersig=$usersig&random=99999999&contenttype=json
```

Значение и допустимые значения каждого параметра приведены ниже (имена и значения параметров чувствительны к регистру):

| Параметр | Значение | Описание |
| --- | --- | --- |
| https | Протокол запроса | Протокол запроса — HTTPS, метод запроса — POST. |
| xxxxxx | Домен запроса | Выделенное имя домена, соответствующее стране/региону, где находится SDKAppID: Китай: `console.tim.qq.com` Сингапур: `adminapisgp.im.qcloud.com` |
| ver | Номер версии протокола | Зафиксирован как `v4` |
| servicename | Имя внутреннего сервиса. Разные имена сервисов соответствуют разным типам сервисов. | `v4/call_engine_http_srv/get_call_info`, где `call_engine_http_srv` — это `servicename`. Для получения более подробной информации см. [Список REST API](https://www.tencentcloud.com/document/product/647/60702#). |
| command | Слово команды, комбинируется с servicename для определения определенной бизнес-функциональности. | Пример: `v4/call_engine_http_srv/get_call_info`, где `get_call_info` — это `command`. Для получения более подробной информации см. [Список REST API](https://www.tencentcloud.com/document/product/647/68927#). |
| sdkappid | Приложение получает идентификатор приложения в консоли IM. | Получается при подаче заявки на доступ. |
| identifier | Имя пользователя должно быть учетной записью администратора приложения при вызове REST API. | см. [Администратор приложения](https://trtc.io/document/33517) |
| usersig | Пароль, соответствующий имени пользователя. | см. [Генерация UserSig](https://trtc.io/document/34385) |
| random | Параметр случайного числа, указывающий текущий запрос | 32-битное целое число без знака, значение колеблется от 0 до 4294967295 |
| contenttype | Формат запроса | Фиксированное значение — `json`. |

> **Примечание:** Серверу приложения необходимо использовать учетную запись администратора приложения в качестве идентификатора при вызове REST API. Приложение может генерировать UserSig для учетной записи администратора каждый раз при вызове REST API или повторно использовать фиксированный UserSig. Особое внимание следует уделить периоду действия UserSig.

### Формат тела HTTP запроса

- REST API поддерживает только метод POST. Тело его запроса в формате JSON. Пожалуйста, обратитесь к подробному описанию каждого API для получения конкретного формата тела.
- Важно отметить, что тело POST не может быть пусто. Даже если пакет протокола не нужно переносить какую-либо информацию, необходимо переносить пустой объект JSON, то есть `{}`.

### Код возврата HTTP

Если не происходит сетевая ошибка (например, ошибка 502), результат вызова REST API — 200. Фактический код ошибки вызова API и информация об ошибке возвращаются в теле ответа HTTP.

### Формат тела ответа HTTP

Пакет ответа REST API также находится в формате JSON. Его формат имеет следующие особенности:

```
{    "ActionStatus": "OK",     "ErrorInfo": "",     "ErrorCode": 0,    "RequestId": "Id-70e312f1de024af5a36714b7b71da224-O-Seq-63504"    // Другое содержимое ответа REST API}
```

Полезная нагрузка ответа обязательно содержит атрибуты ActionStatus, ErrorInfo, ErrorCode и RequestId, значения которых приведены ниже:

| Поле | Тип | Описание |
| --- | --- | --- |
| ActionStatus | String | Результат обработки запроса. OK означает успешную обработку; FAIL означает неудачную обработку. Если это FAIL, ErrorInfo содержит причину сбоя. |
| ErrorInfo | String | Причина сбоя |
| ErrorCode | Integer | Код ошибки: 0 означает успех; любое другое значение указывает на сбой. Вы можете запросить [Таблицу кодов ошибок](https://www.tencentcloud.com/document/product/647/54901#) для получения конкретных причин. |
| RequestId | String | Уникальный ID запроса. Возвращается для каждого запроса. RequestId требуется для определения проблемы. |

## Часто задаваемые вопросы

### Может ли REST API Request истечь и не получить ответ?

1. Период истечения времени для параметров REST API серверной части комнаты составляет 3 с. Период истечения времени, установленный вызывающей стороной, должен быть дольше 3 с.
2. `telnet console.tim.qq.com 443` Подтвердите, можете ли вы подключиться к порту сервиса.
3. Используйте `curl -I https://console.tim.qq.com` для простого теста, чтобы увидеть, составляет ли код состояния 200.
4. Подтвердите, является ли конфигурация DNS сервера на машине внутренним DNS сервером или общественным DNS сервером. Если это внутренний DNS сервер, пожалуйста, убедитесь, что исходящий трафик DNS совпадает с региональным оператором, где находится исходящий IP-адрес сети этой машины.
5. Рекомендуется, чтобы бизнес-вызывающие стороны использовали режим "постоянное соединение + пул соединений".

> **Примечание:** Поскольку продолжительность TCP соединения HTTPS неустойчивых соединений относительно велика и существуют расходы на TCP + tls handshake для каждого запроса, рекомендуется использовать доступ с постоянным соединением для REST API.

Сценарии использования стандартной библиотеки HTTP: Для HTTP1.0 необходимо указать заголовок запроса Connection: keep-alive. HTTP1.1 по умолчанию поддерживает долгое соединение. В сценариях, где запросы HTTPS инкапсулированы на основе TCP, TCP соединения могут быть повторно использованы для отправки и получения запросов.

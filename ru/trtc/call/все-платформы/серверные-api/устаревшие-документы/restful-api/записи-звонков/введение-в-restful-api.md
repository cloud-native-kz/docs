# Введение в RESTful API

RESTful API является частью интерфейса HTTP Hub Management Interface Hub сервиса TRTC Call, предоставляя разработчикам упрощенную точку управления. Для ознакомления с поддерживаемыми в настоящее время RESTful API сервиса TRTC Call обратитесь к [Списку REST API](#6536fcf5-62c7-4fb8-ad61-dcbd58737bf0).

По соображениям безопасности RESTful API доступен только через интерфейс HTTPS.

## Условия использования

- **REST API в настоящее время находится в бета-версии**. В период бета-тестирования приложения (SDKAppId), для которых включена **версия группового вызова** сервиса TRTC Call, могут использовать REST API. Вы также можете активировать бесплатный пробный доступ к экспериментальной версии. Описание версий и инструкции по активации см. в разделе [Активировать сервис](https://www.tencentcloud.com/document/product/647/59832#).
- В настоящее время RESTful API доступен только в определенных версиях TUICallKit для различных платформ, как показано в таблице ниже:

| Платформа/Фреймворк | Номер версии |
| --- | --- |
| Android/iOS/Flutter/uni-app (клиент) | ≥ 1.7.1 |
| Web | ≥ 1.4.6 |
| WeChat Mini Program | ≥ 1.5.1 |

> **Примечание:** Только после обновления всех участвующих платформ/фреймворков до указанных выше версий в консоли можно просматривать соответствующую информацию о вызовах. В период бета-тестирования RESTful API поддерживает запрос данных за последние 7 дней.

## Способ вызова

### URL запроса

Формат URL RESTful API выглядит следующим образом:

```
https://xxxxxx/$version/$kind/$command?sdkappid=$SDKAppID&identifier=$identifier&usersig=$usersig&random=99999999&contenttype=json
```

Значения и описания каждого параметра приведены ниже (имена параметров и их значения чувствительны к регистру):

| Параметр | Описание | Получение значения |
| --- | --- | --- |
| https | Протокол запроса | Протокол запроса — HTTPS, метод запроса — POST |
| xxxxxx | Зарезервированное имя домена | `callkit-intl.trtc.tencent-cloud.com` |
| version | Номер версии протокола | Фиксированное значение `v1` |
| kind | Классификация управления | Пример: `v1/records/get_record_by_callId`, где `records` — это kind |
| command | Слово `command`, используется в сочетании с `kind` для указания конкретной функции бизнеса | Пример: `v1/records/get_record_by_callId`, где `get_record_by_callId` — это command |
| sdkappid | Идентификатор приложения, полученный в консоли | Получается при подаче заявки на интеграцию |
| identifier | имя пользователя, при вызове RESTful API должна быть учетная запись администратора приложения | Использование учетной записи администратора Chat |
| usersig | пароль, соответствующий имени пользователя | См. раздел [Генерирование UserSig](https://www.tencentcloud.com/document/product/647/39074#) |
| random | Параметр случайного числа, идентифицирующий текущий запрос | 32-битное целое число без знака в диапазоне от 0 до 4294967295 |
| contenttype | Формат запроса | Фиксированное значение: json |

> **Примечание:** После получения или приобретения пакетов в системе аккаунтов Chat будет создана учетная запись администратора с именем `administrator`. Используйте `administrator` для параметра `identifier` в запросах. Если вы отмените или удалите администратора в Chat в разделе [Управление аккаунтами](https://console.tencentcloud.com/im/account-management), укажите правильную учетную запись администратора и соответствующий UserSig. Приложения могут либо генерировать UserSig для учетной записи администратора при каждом вызове RESTful API, либо генерировать фиксированный UserSig для повторного использования. Однако крайне важно уделить внимание периоду действия UserSig. При выполнении операций создания или входа в комнату система автоматически импортирует аккаунты Chat в систему Chat. Обратите на это внимание.

## Формат тела HTTP запроса

RESTful API поддерживает только метод POST, и тело его запроса имеет формат JSON. Для конкретных форматов тела обратитесь к подробному описанию каждого API.

> **Примечание:** Тело POST не может быть пустым. Даже если протокол не требует никаких данных, он все равно должен содержать пустой объект JSON, а именно `{}`.

### Код возврата HTTP

Если не происходит сетевой ошибки (например, ошибка 502), результат вызова RESTful API всегда имеет код 200. Фактический код ошибки и сообщение об ошибке вызова API возвращаются в теле HTTP ответа.

### Формат тела HTTP ответа

Тело ответа RESTful API также имеет формат JSON и соответствует следующим характеристикам:

```
{    "errorCode": 0,     "errorMessage": "Success",    "requestId": "1c8960ac38d61be38b6fb219db0182d1",     "data": {} }
```

Тело ответа должно содержать три атрибута: errorCode, errorMessage, requestId. Их описание приведено ниже:

| Поле | Тип | Описание |
| --- | --- | --- |
| errorCode | String | Код ошибки, 0 означает успех, другие значения означают ошибку |
| errorMessage | String | Сообщение об ошибке |
| requestId | Integer | Уникальный идентификатор запроса |

## Коды распространенных ошибок RESTful API

| Код ошибки | Описание |
| --- | --- |
| 0 | Запрос выполнен успешно |
| 50001 | Текущее приложение должно приобрести пакет версии для группового вызова TUICallKit для использования |
| 70001 | Недействительные параметры запроса, проверьте, не отсутствуют ли обязательные параметры или они введены неправильно |
| 70002 | UserSig недействителен |
| 70003 | UserSig истек |
| 70004 | Запрашивающий пользователь не является администратором |
| 70005 | Частота запросов ограничена |
| 70009 | Ошибка при разборе тела запроса, проверьте, верен ли тип параметра запроса |
| Неизвестный код ошибки | Внутренняя ошибка системы, пожалуйста, [Отправьте заявку](https://console.intl.cloud.tencent.com/workorder/category) для связи с техническим персоналом |

## Часто задаваемые вопросы

### Возможны ли истечения времени ожидания запросов REST API без получения ответа?

Вы можете подтвердить это со следующих аспектов:

1. Время ожидания интерфейса REST на сервере составляет 3 с, время ожидания вызывающей стороны должно быть больше чем 3 с.
2. Используйте `telnet callkit-intl.trtc.tencent-cloud.com 443` для проверки возможности подключения к портуу сервиса.
3. Используйте `curl -I https://callkit-intl.trtc.tencent-cloud.com` для простого теста, чтобы проверить, равен ли код статуса 200.
4. Убедитесь, является ли конфигурация DNS сервера машины внутренним или публичным DNS сервером. Если это внутренний DNS сервер, убедитесь, что выход сети DNS сервера и регион ISP машины совпадают.
5. Рекомендуется использовать бизнес-вызывающей стороне паттерн **долгого соединения + пул соединений**.

# Групповые сообщения

## Варианты использования

### Отправка и получение сообщений в группе

Члены группы отправляют и получают сообщения в группе.

### Администратор приложения отправляет сообщения

Администратор приложения может отправлять сообщения из бэкенда или выдавать себя за других пользователей для отправки сообщений. В последнем случае, даже если администратор приложения или отправитель не является членом группы, сообщения всё равно будут доставлены.

### Администратор приложения имитирует системные сообщения

Администратор приложения может имитировать системное уведомление, отправляя сообщения из бэкенда. Эти сообщения доставляются как системные уведомления указанным онлайн-членам группы, и пользовательские сообщения администратора приложения, полученные в приложении, могут обрабатываться специальным образом.

## Механизм SEQ для групповых сообщений

Chat ведёт SEQ сообщения для каждой группы, начиная с начального значения 1. Каждый раз, когда в групповом чате отправляется обычное сообщение, бэкенд Chat использует текущее значение SEQ как SEQ сообщения и затем увеличивает SEQ на 1.

Идентификатор группы и SEQ вместе составляют уникальный идентификатор сообщения.

> **Внимание:** Chat генерирует инкрементные SEQ только для роамируемых сообщений.

### Типы групповых сообщений

| Тип сообщения | Описание |
| --- | --- |
| Текстовое сообщение | Содержимое сообщения — простой текст. |
| Изображение | Содержимое сообщения включает URL, размеры и размер файла изображения. |
| Сообщение с эмодзи | Сообщения с эмодзи настраиваются разработчиками. |
| Аудиосообщение | Аудиоданные должны включать длительность в секундах. |
| Сообщение о месторасположении | Содержимое сообщения содержит описание, долготу и широту местоположения. |
| Файловое сообщение | Содержимое сообщения включает URL, размер и формат файла. Ограничений на формат файла нет, максимальный поддерживаемый размер файла составляет 100 МБ. |
| Сообщение с коротким видео | Содержимое сообщения включает URL, длительность, размер и формат файла короткого видео. Максимальный поддерживаемый размер файла составляет 100 МБ. |
| Пользовательское | Типы сообщений, настраиваемые разработчиками, такие как красный пакет и камень-ножницы-бумага. |
| Системное уведомление | Этот тип сообщения разделяется на встроенные системные уведомления и системные уведомления, настраиваемые разработчиками. |

| Тип | Описание функции | Сценарий применения |
| --- | --- | --- |
| Отправка обычных групповых сообщений | Члены группы могут отправлять сообщения через API Chat SDK. Вызывая RESTful API, администраторы приложений могут отправлять сообщения в любую группу без присоединения к группе. | Члены группы отправляют сообщения в группе, и администраторы приложений могут отправлять сообщения в любую группу. |
| Отправка системных уведомлений группы | Вызывая RESTful API, администраторы приложений могут отправлять системные уведомления в любую группу без присоединения к группе. Эти системные уведомления получают только онлайн-члены группы и не имеют возможности роаминга. | Администраторы приложений отправляют своевременные уведомления всем или части онлайн-членов группы. |
| Автономная доставка групповых сообщений | Поддерживается мобильными телефонами Apple, Huawei, Xiaomi, OPPO, vivo и Meizu | Автономная доставка групповых сообщений |
| Члены группы получают онлайн-сообщения группы | Члены группы могут получать онлайн-сообщения группы через Chat SDK. | Онлайн-члены группы получают групповые сообщения в реальном времени. |
| Члены группы получают автономные сообщения или исторические сообщения | Члены группы могут запрашивать исторические сообщения через Chat SDK. | Члены группы получают автономные сообщения при переходе в онлайн, и члены группы могут просматривать исторические сообщения группы. |
| Бэкенд приложения получает групповые сообщения | Администраторы приложений могут загружать все сообщения, созданные в течение определённого периода времени, через RESTful API. Администраторы приложений также могут получить историю чата любой группы через RESTful API. Бэкенд приложения может получать групповые сообщения через webhook после отправки сообщений в группе. | Приложения регулярно резервируют копии истории сообщений. Приложениям нужно быстро получить историю сообщений определённых групп. Приложениям нужно получать групповые сообщения в реальном времени. |
| Удаление сообщений | Исторические сообщения можно удалить через RESTful API, чтобы обеспечить, что они не будут распространяться дальше. | Удаление вредоносной информации в группе. |
| Групповые сообщения включают профиль пользователя | Групповые сообщения могут содержать никнейм отправителя, фото профиля, профиль группы, пользовательские поля уровня пользователя и пользовательские поля уровня члена. | Отображение информации об отправителе, такой как никнейм и фото профиля. |
| Контроль отправки групповых сообщений | Вы можете контролировать отправку групповых сообщений путём отключения звука и webhook до отправки группового сообщения. | Предотвращение отправки сообщений членом группы, предотвращение отправки сообщений всеми членами группы и фильтрация или изменение сообщений в бэкенде приложения. |
| Контроль получения групповых сообщений | Вы можете установить различные параметры получения сообщений для отдельных групп: получать и уведомлять, получать без уведомления и блокировать сообщения. Когда выбран параметр **получать без уведомления**, устройства iOS отключат push-уведомления APNs. | Пользователь блокирует сообщения из группы. |
| Контроль частоты групповых сообщений | Контроль частоты отправки групповых сообщений. Значение по умолчанию — 40 сообщений в секунду. Системные сообщения группы, отправляемые администратором приложения через RESTful API, не подлежат контролю частоты. Дополнительные сведения см. в разделе о приоритете сообщений и контроле частоты ниже. | Предотвращение спама сообщений. |

> **Внимание:** Если необходимо включить никнейм отправителя и фото профиля, импортируйте информацию в эти два поля профилей пользователей в Chat. Если необходимо включить пользовательские поля, настройте пользовательские поля в консоли и отправьте заявку на добавление соответствующих полей к сообщениям.

## Контроль отправки групповых сообщений

Отправка групповых сообщений может контролироваться следующими методами:

| Метод контроля | Описание |
| --- | --- |
| Отключение звука члены группы | Предотвращение отправки сообщений членом группы на определённый период времени. Эта функция действует в пределах одной группы. Если отключённый от звука пользователь покидает группу и присоединяется снова, пользователь остаётся отключённым от звука до истечения времени отключения. |
| Webhook перед отправкой группового сообщения | Перед доставкой сообщения членам группы Chat проверяет разрешение у бэкенда приложения. Если разрешение не дано, сообщение не будет доставлено. После получения webhook бэкенд приложения может изменить содержимое сообщения и вернуть его в Chat, который доставит изменённое сообщение. Дополнительные сведения см. в разделе [Before Group Message Is Sent](https://intl.cloud.tencent.com/document/product/1047/34374). После вызова webhook Chat доставит сообщение напрямую, если не получит ответ в течение двух секунд, без повторных попыток. |

## Приоритет сообщения и контроль частоты

### Приоритет группового сообщения

Групповые сообщения разделены на три уровня приоритета. Если группа превышает предел частоты сообщений, бэкенд доставляет сообщения с высоким приоритетом в первую очередь. Поэтому выбирайте подходящие приоритеты в соответствии с важностью сообщений.
Ниже приведены приоритеты от высшего к низшему:

| Приоритет | Рекомендуемый тип сообщения |
| --- | --- |
| Высокий | Сообщение о красном пакете и подарке |
| Нормальный | Текстовое сообщение |
| Низкий | Сообщение «нравится» |

### Контроль частоты групповых сообщений

#### Контроль частоты на основе количества

> **Примечание:** Контроль частоты на основе количества ограничивает максимальное количество сообщений, отправляемых всеми клиентскими API в секунду в одной группе. Значение по умолчанию — 40 сообщений в секунду. Когда количество отправленных сообщений превышает лимит, бэкенд сначала доставляет сообщения с более высоким приоритетом, сообщения с одинаковым приоритетом доставляются в случайном порядке.

Сообщение, ограниченное контролем частоты, не доставляется и не сохраняется в истории сообщений, но отправителю возвращается ответ об успехе. Webhook [Before Group Message Is Sent](https://intl.cloud.tencent.com/document/product/1047/34374) срабатывает, но webhook [After Group Message Is Sent](https://intl.cloud.tencent.com/document/product/1047/34375) не срабатывает.

#### Контроль частоты на основе приоритета

Контроль частоты на основе приоритета ограничивает максимальное количество сообщений с определённым приоритетом, отправляемых в секунду в одной группе. Запрос на отправку сообщения должен сначала пройти проверку контроля частоты на основе количества перед вводом в обработку контроля частоты на основе приоритета.

Все сообщения подлежат ограничению частоты в 40 сообщений в секунду. Вы можете установить три уровня приоритета. Сообщения с приоритетом High являются приоритетными сообщениями, которые вряд ли будут ограничены. Однако, если количество сообщений с высоким приоритетом, отправляемых в секунду, превышает 40, сообщения с высоким приоритетом также будут отброшены.

## Обработка автономных групповых сообщений

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/a2924d6952d211ee974d5254005f490f.png)

#### Автономные групповые сообщения обрабатываются следующим образом:

1. Пользователь A вызывает `sendMessage` для отправки сообщений в группу C, когда пользователь B находится в автономном режиме.

1.1 Группа C добавляется в последние контакты пользователя B с кешированием до 100 сообщений.

1.2 Система обновляет информацию группового сообщения, включая последний SEQ группового сообщения.

1.3 Сообщения хранятся на роамирующем сервере в течение 7 дней.

2. Пользователь B вызывает API `login` для входа в Chat.
3. SDK автоматически извлекает информацию SEQ всех групповых сообщений, включая последний SEQ сообщения и количество непрочитанных сообщений.
4. SDK автоматически извлекает последние контакты и срабатывает API `OnNewMessage`.
5. Пользователи уведомляются через API `OnRefresh` при завершении синхронизации группового сообщения.
6. Пользователи вызывают `getMessage`, и SDK автоматически извлекает сообщения с роамирующего сервера.

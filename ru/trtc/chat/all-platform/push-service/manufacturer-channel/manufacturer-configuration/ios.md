# iOS

Эта статья поможет вам создать и использовать **Push-сертификаты** и **Файлы профиля**, при условии, что у вас есть **учетная запись разработчика** и **сертификаты разработки/распространения**.

Если у вас есть вопросы о "учетной записи разработчика" или "сертификатах разработки/распространения", вы можете [обратиться к соответствующим концепциям в приложениях](https://www.tencentcloud.com/document/product/1047/60548).

## Конфигурация Push-сертификата APNs

Перед интеграцией компонента TIMPush вам необходимо запросить push-сертификат APNs у Apple, а затем загрузить сертификат в консоль Chat. После этого вы можете следовать инструкциям быстрого доступа для интеграции.

Apple в настоящее время предлагает два основных типа сертификатов для конфигурации поставщика: сертификаты p12 и сертификаты p8. Каждый имеет свои преимущества и недостатки, поэтому вы можете выбрать один в зависимости от ваших потребностей.

|  | **Тип сертификата** | **Срок действия и управление** | **Безопасность** | **Dynamic Island** |
| --- | --- | --- | --- | --- |
| **P12 сертификат** | Сертификат P12 — это бинарный файл, содержащий открытый и закрытый ключи, используемый для аутентификации на основе сертификата. Он объединяет сертификат открытого ключа и закрытый ключ в один файл с расширением .p12 или .pfx. | Сертификат P12 обычно имеет период действия один год и должен быть переформирован и повторно развернут по истечении срока действия. Каждое приложение требует отдельного сертификата P12 для обработки push-уведомлений. | Сертификаты: P12 сертификат использует аутентификацию на основе сертификата и требует хранения закрытого ключа на сервере. Это может увеличить риски безопасности, так как закрытый ключ может быть доступен неавторизованным пользователям. | Не поддерживается |
| **P8 сертификат** | P8 сертификат — это Auth Key, используемый для аутентификации на основе токена. Это текстовый файл, содержащий закрытый ключ, с расширением .p8. | P8 сертификат не имеет срока действия, поэтому вам не нужно беспокоиться об истечении срока действия сертификата. Кроме того, использование P8 сертификата упрощает управление сертификатами, так как вы можете использовать один P8 сертификат для предоставления услуги push-уведомлений для нескольких приложений. | P8 сертификат использует аутентификацию на основе токена, что означает, что ваш сервер периодически генерирует JSON Web Token (JWT) для установления соединения с APNs. Этот метод более безопасен, так как не требует хранения закрытого ключа на сервере. | Поддержка Dynamic Island push |

### Метод 1: использование сертификата p12 (традиционный push-сертификат APNs)

### Шаг 1: запрос сертификата APNs

1. Войдите в [Apple Developer Center](https://developer.apple.com/account/), нажмите **Certificates, Identifiers & Profiles** или **Certificates, IDs & Profiles** на боковой панели, чтобы перейти на страницу **Certificates, IDS & Profiles**.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4bcbd12fb2d611efa2e952540075b605.jpeg)

2. Нажмите **+** рядом с Identifiers на правой стороне.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4c92b989b2d611ef9d2952540055f650.png)

3. Вы можете выполнить следующие шаги для создания нового AppID или добавить сервис `Push Notification` к существующему AppID.

> **Примечание:** `Bundle ID` вашего приложения не может использовать подстановочный знак `*`, иначе сервис удаленной отправки будет недоступен.

4. Выберите **App IDs**, нажмите **Continue** для перехода к следующему шагу.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4c08663bb2d611efa2e952540075b605.png)

5. Выберите **App**, нажмите **Continue** для перехода к следующему шагу.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4d027f50b2d611ef9d2952540055f650.jpeg)

6. Сконфигурируйте `Bundle ID` и другую информацию, нажмите **Continue** для перехода к следующему шагу.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4cceec6eb2d611ef9dc0525400329841.png)

7. Выберите **Push Notifications** для активации сервиса удаленной отправки.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4ce0e971b2d611efa2e952540075b605.png)

#### Генерирование сертификата

1. Выберите ваш AppID, выберите **Configure**.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4ca1376fb2d611ef9dc0525400329841.png)

2. Вы можете увидеть в окне **Apple Push Notification service SSL Certificates**, что есть два `SSL сертификата`, один для среды разработки (Development) и один для среды производства (Production), как показано ниже:

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4cce355fb2d611efa2e952540075b605.jpeg)

3. Сначала мы выбираем Development и нажимаем **Create Certificate**; система запросит нас на Certificate Signing Request (CSR).

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4ccbcfd3b2d611ef8b1b525400f69702.png)

4. Откройте **Keychain Access** на вашем Mac, в меню выберите **Keychain Access** > **Certificate Assistant** > **Request a Certificate From a Certificate Authority** (`Keychain Access - Certificate Assistant - Request a Certificate From a Certificate Authority`).

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/9202f78dc1c511ef9d7852540075b605.png)

5. Введите адрес электронной почты пользователя (ваше письмо), Common Name (ваше имя или имя компании), выберите **Save to disk**, нажмите **continue**, и система создаст файл `*.certSigningRequest`.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/a1bf7a64c1c511efac86525400329841.png)

6. Вернитесь на страницу веб-сайта Apple Developer, на которой вы находились на шаге 3 приведенных выше инструкций, нажмите **Choose File** для загрузки файла `*.certSigningRequest`.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4dfee5ddb2d611ef8c01525400fdb830.png)

7. Нажмите **Continue** для генерирования push-сертификата.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4d204532b2d611ef9d2952540055f650.png)

8. Нажмите **Download** для загрузки `Development SSL Certificate` на вашу локальную машину.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4d2cec98b2d611ef8c01525400fdb830.jpeg)

9. Повторите шаги 1-8 выше для загрузки `Production SSL Certificate` на вашу локальную машину.

> **Примечание:** На самом деле этот сертификат — это объединенный сертификат Sandbox и Production, который применяется как к среде разработки, так и к среде производства.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4c372d41b2d611ef9dc0525400329841.png)

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4cb04041b2d611efbfb3525400bdab9d.png)

10. Дважды щелкните загруженный `SSL Certificate` для сред разработки и производства. Система импортирует его в цепочку ключей.
11. Откройте приложение Keychain Access, перейдите в **log in to** > **My Certificates**, щелкните правой кнопкой мыши, чтобы экспортировать недавно созданные среду разработки (`Apple Development IOS Push Service`) и среду производства (`Apple Push Services`) файлы `p12`.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/b9b239abc1c511efb92b525400fdb830.png)

> **Примечание:** При сохранении файла `.p12` обязательно установите пароль. Шаг 2: загрузите сертификат в консоль.

### Шаг 2: загрузите сертификат p12 в консоль Chat

1. Войдите в [Chat консоль](https://console.trtc.io/chat/push-plugin-push-identifier).
2. Нажмите на карточку целевого **приложения** для перехода на страницу базовой конфигурации приложения.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/a7e6f064c1c611ef893e525400bdab9d.png)

3. Нажмите **iOS native offline push settings** справа, нажмите **Add Certificate**.
4. Выберите тип сертификата, загрузите сертификат iOS (p12), установите пароль сертификата и нажмите **Confirm**.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/59652721c1d911efa3e352540099c741.png)

> **Примечание:** При загрузке push-сертификата убедитесь, что Bundle ID совпадает с Bundle ID вашего основного приложения. Рекомендуется называть загруженный сертификат на английском языке (специальные символы, такие как скобки, не допускаются). Вы должны установить пароль для загруженного сертификата. Без пароля push-уведомления не могут быть получены. Для приложения, опубликованного в App Store, окружение сертификата должно быть окружением производства. В противном случае push-уведомления не могут быть получены. Загруженный сертификат .p12 должен быть вашим собственным подлинным и действительным сертификатом.

5. После создания информации о push-сертификате запишите **ID** сертификата.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/7b16f993c1c711ef9ed252540022f050.png)

### Метод 2: использование сертификата p8 (поддержка Dynamic Island Push)

P8 сертификат: P8 сертификат не имеет срока действия, поэтому вам не нужно беспокоиться об истечении срока действия сертификата. Кроме того, использование p8 сертификата может упростить управление сертификатами, так как вы можете использовать один p8 сертификат для предоставления услуг push-уведомлений для нескольких приложений. Кроме того, p8 сертификат поддерживает Dynamic Island push.

### Шаг 1: запрос сертификата APNs

Для создания файла сертификата p8 вам сначала нужно войти в [Apple Developer Center](https://developer.apple.com/).

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4d25e5b1b2d611ef9d2952540055f650.png)

1. Перейдите в Certificates, Identifiers & Profiles: нажмите на **Account** в верхнем правом углу страницы, затем выберите **Certificates, Identifiers & Profiles** из раскрывающегося меню.
2. Создайте новый App ID: нажмите на **Identifiers** в левом меню, затем нажмите на **+** справа для создания нового App ID. Заполните соответствующую информацию и нажмите **Continue**.
3. Создайте новый ключ: В левом меню нажмите на **Keys**, затем нажмите **+** справа для создания нового ключа. Введите имя ключа, затем отметьте **Apple Push Notifications service (APNs)**, нажмите **Continue**.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4c9147c3b2d611efa2e952540075b605.png)

Подтвердите и создайте ключ: На странице подтверждения проверьте информацию вашего ключа, затем нажмите **Register**. Далее вы увидите страницу с предложением загрузить ключ. Нажмите **Download** для сохранения созданного файла .p8 на ваш компьютер.

> **Примечание:** P8 сертификат можно загрузить только один раз, поэтому храните его безопасно. Храните загруженный файл p8 в безопасности, так как вы не сможете загрузить его снова. Вы можете использовать этот P8 сертификат для конфигурирования вашего iOS приложения для получения push-уведомлений.

### Шаг 2: загрузите сертификат p8 в консоль Chat

1. Войдите в [Chat консоль](https://console.trtc.io/chat/push-plugin-push-identifier).
2. Нажмите на карточку целевого приложения для перехода на страницу базовой конфигурации приложения.
3. Нажмите **iOS native offline push settings** справа, нажмите **Add Certificate**.
4. Выберите сертификат .p8
5. Выберите окружение производства и окружение разработки каждого для загрузки файла (вам нужно подготовить те же материалы, только бэкенд назначит для вас окружение APNs push)

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/f8682b2ec1c611ef98e4525400329841.png)

> **Примечание:** **KeyID**: это уникальный идентификатор вашего ключа APNs Auth. Когда вы создаете новый ключ APNs Auth в Apple Developer Center, система генерирует для вас Key ID. Вы можете найти его в разделе "Certificates, Identifiers & Profiles" под "Keys". **TeamID**: это уникальный идентификатор вашей учетной записи разработчика. Вы можете найти его на странице деталей учетной записи в Apple Developer Center. Нажмите "Membership" в верхнем правом углу, и вы найдете свой Team ID в разделе "Membership Details". **BundleID**: это уникальный идентификатор вашего приложения, также известный как Application ID. Вы можете найти его в разделе "Certificates, Identifiers & Profiles" в Apple Developer Center. Выберите "Identifiers", затем найдите соответствующий Bundle ID в списке приложений.

## Создание файла профиля

### Шаг 1: создайте App ID

1. **Войдите в Apple Developer Center:**
- Перейдите в [Apple Developer Center](https://developer.apple.com/account/) и войдите в вашу учетную запись разработчика.
2. **Создайте App ID:**
- В разделе "Certificates, Identifiers & Profiles" выберите "Identifiers".
- Нажмите кнопку "+" для создания нового App ID.
- Выберите "App IDs" и нажмите "Continue".
- Введите имя App ID и Bundle ID (например, **com.yourcompany.yourapp**).
- Убедитесь в выборе соответствующих функций (например, Push Notifications, App Groups и т.д.), затем нажмите "Continue" и подтвердите.

### Шаг 2: создайте App ID для Service Extension (опционально)

- Повторите приведенные выше шаги для создания нового App ID для вашего Service Extension.
- Bundle ID обычно состоит из Bundle ID основного приложения, за которым следует идентификатор расширения, например, com.yourcompany.yourapp.extension.

### Шаг 3: создайте файл профиля

1. **Создайте файл профиля:**
- В разделе "Certificates, Identifiers & Profiles" выберите "Profiles".
- Нажмите кнопку "+" для создания нового файла профиля.
- Выберите "iOS App Development" или "App Store" (в зависимости от ваших потребностей), затем нажмите "Continue".
- Выберите только что созданный App ID (App ID основного приложения), затем нажмите "Continue".
- Выберите ваш сертификат разработки и устройство (если это файл профиля разработки), затем нажмите "Continue".
- Введите имя файла профиля, затем нажмите "Generate".
2. **Создайте файл профиля для Service Extension (опционально):**
- Повторите приведенные выше шаги для создания нового файла профиля для вашего Service Extension, убедившись, что вы выбираете соответствующий App ID.

### Шаг 4: загрузите и установите файл профиля

1. **Загрузите файл профиля:**
- После создания файла профиля нажмите кнопку "Download" для загрузки файла профиля.
2. **Установите файл профиля:**
- Дважды щелкните на загруженном файле профиля. Он автоматически установится в Xcode.

## Приложение: учетная запись разработчика, тип сертификата и файл профиля

**Учетная запись разработчика Apple:**

Создание сертификатов и профилей iOS требует платную учетную запись Apple. Существует три типа учетных записей разработчика Apple: Individual developer, Company developer и Enterprise developer, каждый с разными целями.

- Учетные записи Individual developer и Company developer стоят $99 в год, в то время как учетные записи Enterprise developer стоят $299 в год.
- Разработчики Enterprise обычно используются крупными предприятиями при разработке внутренних приложений и не могут быть опубликованы в App Store.
- Пожалуйста, убедитесь, что у вас уже есть учетная запись Individual developer или Company developer.

**Введение в сертификаты:**

Сертификаты имеют различные типы и служат разным целям. Вот классификация сертификатов iOS:

| **Тип сертификата** | **Различия** |
| --- | --- |
| **Сертификат разработчика** (Developer Certificate) | Используется для подписания и отладки приложений на этапе разработки. Сертификат разработчика выдается Apple Developer Center и требует использования Xcode или других инструментов разработки для применения и управления. |
| **Сертификат распространения** (Distribution Certificate) | Используется для распространения приложений другим пользователям или загрузки их в App Store на рассмотрение. Сертификат распространения выдается Apple Developer Center и требует использования Xcode или других инструментов разработки для применения и управления. |
| **Push-сертификат** (Push Certificate) | Используется для реализации функции удаленной отправки APNs, что позволяет приложению получать push-уведомления с сервера. Push-сертификат выдается Apple Developer Center и требует использования Xcode или других инструментов разработки для применения и управления. |
| **Корпоративный сертификат** (Enterprise Certificate) | Используется для распространения приложений среди внутренних сотрудников или клиентов. Корпоративный сертификат выдается Apple Developer Center и требует использования Xcode или других инструментов разработки для применения и управления. |

**Введение в файл профиля:**

Provisioning Profiles также делятся на два типа: Development и Distribution. Файл конфигурации включает сертификаты, App ID и устройства.

Расширение файла — .mobileprovision. Он играет роль в конфигурации и валидации в системе учетной записи разработчика и необходим для отладки на реальном устройстве и упаковки для выпуска.

| **Файл профиля (**Provisioning Profile**)** | **Функция** |
| --- | --- |
| Development | Provisioning Profile соответствует одному App ID (Bundle ID). Provisioning Profile определяет, какую пару ключей (открытый ключ/закрытый ключ) Xcode использует для подписания приложения, и будет встроена в пакет .ipa при упаковке приложения. Provisioning Profile упаковывает всю эту информацию вместе, что удобно для использования при отладке и упаковке приложения для выпуска. Таким образом, достаточно просто выбрать разные файлы Provisioning Profile в разных ситуациях. Provisioning Profiles также делятся на типы Development и Distribution с периодом действия, аналогичным Certificates. Версия Development Provisioning Profile используется для разработки и отладки, а версия Distribution в основном используется для отправки в App Store на рассмотрение и не указывает устройства для тестирования разработки. |

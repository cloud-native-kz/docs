# Как интегрировать чат в игры

adminapisgp.im.qcloud.comadminapisgp.im.qcloud.comМгновенный обмен сообщениями — частое требование игр, а мгновенный чат стал важной функцией в многопользовательских играх. Игровая платформа включает разнообразные типы групп, пользовательские типы сообщений (такие как дарение игровых предметов и торговля), глобальный доступ и другие сложные требования. В этом документе описаны методы реализации распространённых требований при разработке игрового чата, а также возможные проблемы и соображения, помогая разработчикам быстро разобраться в требованиях и реализовать их.

## Подготовка

### Расчёт UserSig с помощью ключа

В системе аккаунтов Chat пароль, необходимый для входа пользователя, рассчитывается сервером с использованием ключа, предоставленного Chat. Дополнительную информацию см. в разделе [Генерирование UserSig](https://intl.cloud.tencent.com/document/product/1047/34385). На этапе разработки, чтобы не замедлять разработку на клиенте, вы также можете рассчитать `UserSig` в [консоли](https://console.trtc.io/usersig) следующим образом:

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/685065ff0a1f11f0a6d15254007c27c5.png)

### Настройка учётной записи администратора

Управление игровым мгновенным обменом сообщениями может потребовать от администраторов отправки объявлений по электронной почте в игры, управления временными сообщениями команд и т. д., что может быть сделано через [серверные API Chat](https://intl.cloud.tencent.com/document/product/1047/34621). Для вызова этих API необходимо [создать учётную запись администратора Chat](https://console.tencentcloud.com/im/account-management). По умолчанию Chat предоставляет учётную запись с `UserID`, равным `administrator`. Вы также можете создавать несколько учётных записей администраторов по мере необходимости. Обратите внимание, что можно создать максимум пять учётных записей администраторов.

### Настройка адреса обратного вызова и включение обратного вызова

Для реализации игровых команд и других требований используется событие Webhook, в котором бэкенд Chat вызывает бэкенд бизнеса в определённых сценариях. Вам только нужно предоставить HTTP API и настроить его в модуле [**Конфигурация Webhook**](https://console.tencentcloud.com/im/callback-setting) в консоли следующим образом:

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/11b196230a1e11f0a1dd5254005ef0f7.png)

### Интеграция клиентского SDK

После завершения подготовки необходимо интегрировать клиентский SDK Chat в ваш проект. Вы можете выбрать различные варианты интеграции по мере необходимости. Подробные инструкции см. в разделе [Введение в TUIKit](https://intl.cloud.tencent.com/document/product/1047/34547).

Далее описаны распространённые функции Chat, которые необходимо интегрировать в игры, и приводятся лучшие практики с кодами реализации.

## Руководство по разработке функций игрового чат-рума

### Профиль пользователя

#### Обычный профиль пользователя

Обычные профили пользователей, хранящиеся в игровом бизнесе, можно разделить на профили базовой информации и другие профили информации.

| Базовая информация | Другая информация |
| --- | --- |
| Имя пользователя, пол, дата рождения, уровень, роль, номер мобильного телефона и т. д. | Другая информация, необходимая для игр |

#### Хранение профиля

Игровой бизнес имеет большое количество пользователей, и сложно хранить огромные объёмы данных пользователей. Tencent Cloud Chat предлагает полный набор решений для профилей через возможности размещения профилей пользователей. Ниже сравниваются хранение профилей пользователей в Tencent Cloud Chat и в бизнес-бэкенде.

| Элемент | Chat | Бизнес-бэкенд |
| --- | --- | --- |
| Ёмкость хранилища | Поддерживает автоматическое масштабирование | Поддерживает ограниченную ёмкость и сложно масштабировать |
| Профили пользователей | Поддерживает стандартные и пользовательские поля с ограничениями на длину и имена полей | Настраиваемые, более гибкие |
| Чтение/запись профилей | Поддерживает простые в использовании сервисные API и рекомендации | Требует самостоятельной разработки |
| API | Требует частоту вызовов API не более 200 раз/секунду | Позволяет разрабатывать возможности вызовов API и другие возможности по мере необходимости |
| Безопасность | Поддерживает удалённое восстановление после аварии и развёртывание в нескольких регионах | Требует самостоятельного обслуживания |

Помимо возможностей хранения и чтения/записи профилей, Chat имеет следующие преимущества благодаря своему сервису размещения профилей пользователей:

1. Chat предоставляет удалённое восстановление после аварии, развёртывание в нескольких регионах и возможности автоматического масштабирования. Таким образом, вы полностью избавлены от сложных процессов обработки, таких как простои серверов, репликация первичного-вторичного с несколькими копиями и масштабирование ёмкости.
2. Chat предоставляет распространённые в промышленности бизнес-процессы обработки, о которых вам не нужно беспокоиться о бизнес-логике профилей пользователей.
3. Chat предоставляет профессиональные процессы эксплуатации и команды, гарантирующие качество обслуживания 99,99% в год и помогающие вам предоставлять услуги, известные своей стабильностью.
4. Chat предоставляет простые в использовании сервисные API и легкодоступные рекомендации с премиум-сервисами на протяжении всего процесса.

#### Методы хранения профилей пользователей Chat

Схема хранения Chat включает хранение профилей пользователей и возможности чтения/записи. Ниже описано, как Chat хранит профиль пользователя, профиль друга и данные расширения. Все данные хранятся в формате `Key-Value`. `Key` имеет тип `String` и поддерживает только прописные и строчные буквы, цифры и подчёркивания. `Value` поддерживает следующие типы:

| Тип | Описание |
| --- | --- |
| uint64_t | Целое число (не поддерживается пользовательскими полями.) |
| string | Строка. Длина строки не может превышать 500 байт. |
| bytes | Буфер. Длина буфера не может превышать 500 байт. |
| string array | Массив строк. Каждая строка не может превышать 500 байт. Доступно только для поля `Tag_SNS_IM_Group` списка друзей. |

- **Профиль пользователя**: Профиль пользователя включает стандартные и пользовательские поля. Для пользовательских полей см. раздел данных расширения ниже. Для стандартных полей, поддерживаемых в настоящее время Chat, см. [Стандартные поля профиля](https://intl.cloud.tencent.com/document/product/1047/33520).
- **Профиль друга**: Профиль друга включает стандартные и пользовательские поля друзей. Список контактов Chat поддерживает до 3000 друзей. Стандартные поля друзей описаны следующим образом:

| Имя поля | Тип | Описание |
| --- | --- | --- |
| Tag_SNS_IM_Group | Array | Список друзей |
| Tag_SNS_IM_Remark | String | Примечания к другу |
| Tag_SNS_IM_AddSource | String | Источник запроса на добавление в друзья |
| Tag_SNS_IM_AddWording | String | Содержание запроса на добавление в друзья |
| Tag_SNS_IM_AddTime | Integer | Временная метка запроса на добавление в друзья |

Дополнительную информацию см. в разделе [Стандартные поля друзей](https://intl.cloud.tencent.com/document/product/1047/33521).

- **Пользовательский профиль**: Для подачи заявки на пользовательские поля профиля администратор приложения может войти в [консоль Chat](https://console.trtc.io/chat/login-message), найти ваше приложение и перейти к **Конфигурация функций**. После отправки заявки пользовательские поля профиля вступят в силу в течение пяти минут.
- Дополнительную информацию об управлении профилями пользователей см. в разделе [Управление профилями](https://intl.cloud.tencent.com/document/product/1047/33520).
- Дополнительную информацию об управлении пользовательским профилем цепочки отношений см. в разделе [Пользовательские поля друзей](https://intl.cloud.tencent.com/document/product/1047/33521).

#### Ограничения хранения профилей пользователей Chat

- **Ограничения функций сервиса**
- Хранение данных: Каждая строка или буфер не может превышать 500 байт.
- Пользовательские поля: Ключевые слова пользовательских полей должны состоять из букв английского алфавита с длиной не более 8 байт. Значения пользовательских полей не могут превышать 500 байт.
- Цепочка отношений друзей: Каждый пользователь может иметь до 3000 друзей.
- **Ограничения, связанные с API**
- Управление учётными записями: За один раз можно импортировать до 100 имён пользователей, а за один запрос можно запросить статус до 500 пользователей.
- Другая частота вызовов: До 200 раз в секунду.

Дополнительную информацию об ограничениях использования см. в разделе [Ограничения использования](https://intl.cloud.tencent.com/document/product/1047/34381).

### Системы электронной почты

Системы электронной почты в наши дни почти обязательны в играх. Электронные письма могут содержать текстовые сообщения, а также вложения электронной почты, такие как игровые предметы и награды. Электронное письмо может быть отправлено одному пользователю или может быть отправлено в виде группового письма для распределения наград за событие. Ниже описано, как функции системы электронной почты реализуются с различных аспектов, таких как получение игроками электронных писем, список электронных писем, количество непрочитанных электронных писем, отправка электронных писем всем участникам и период действия электронного письма.

#### Получение и отправка электронной почты

- **Игроки получают электронные письма**: Когда системные электронные письма успешно отправляются и игроки находятся в сети, игроки могут надлежащим образом получать системные электронные письма. Игроки могут получать исторические или последние электронные письма, получив список сообщений переписки электронной почты. Они также могут добавлять или удалять прослушиватель для получения новых сообщений всех типов (включая текстовые, пользовательские и мультимедийные сообщения). Примеры кода для Unity выглядят следующим образом:

```
// Настройка прослушивателя событий получения сообщенияTencentIMSDK.AddRecvNewMsgCallback((List<Message> messages, string user_data)=>{foreach(Message message in messages){  foreach (Elem elem in message.message_elem_array)  {    // Есть следующий элемент    if (elem.elem_type == TIMElemType.kTIMElem_Text)    {       string text = elem.text_elem_content;    }  }}})// Прослушивание обратного вызова `RecvNewMsgCallback` для получения сообщений// Для остановки получения сообщений вызовите `RemoveRecvNewMsgCallback` для удаления прослушивателя. Этот шаг является необязательным и может выполняться по мере необходимости.
```

Дополнительную информацию см. в разделе [Unity — получение сообщения](https://intl.cloud.tencent.com/document/product/1047/48573).

- **Система отправляет электронные письма**: Система может отправлять системные электронные письма пользователям через различные серверные API, которые описаны ниже:

| API | Характеристики | Сценарии применения |
| --- | --- | --- |
| [Отправка персональных сообщений одному пользователю](https://intl.cloud.tencent.com/document/product/1047/34919) | Отправляет сообщение указанному аккаунту. Отправитель, отображаемый получателю, это не администратор, а учётная запись, указанная администратором. | Отправка сообщений конкретному пользователю, например отправка игровой награды пользователю |
| [Отправка персональных сообщений нескольким пользователям](https://intl.cloud.tencent.com/document/product/1047/34920) | Персональное сообщение может быть отправлено до 500 пользователям за один раз, а максимальная частота вызовов составляет 200 раз в секунду. | Отправка сообщений конкретным пользователям без необходимости создания группы получателей. Если количество получателей велико, вам нужно отправлять сообщения партиями. |
| [Отправка обычных сообщений в группе](https://intl.cloud.tencent.com/document/product/1047/34959) | При отправке обычных сообщений группе вам нужно добавить всех получателей в одну группу. | Отправка обычных сообщений большому количеству пользователей (до 100 000 пользователей на одну группу сообщества) |
| [Отправка всем пользователям](https://intl.cloud.tencent.com/document/product/1047/37165) | Отправляет сообщения всем пользователям в приложении. Вы можете указать теги и атрибуты пользователей для отправки сообщений. | Отправка сообщений всем пользователям в приложении или большому количеству пользователей с конкретными характеристиками, такими как письма с кампаний |

> **Примечание:**Отправка всем пользователям доступна только для редакции Pro ãPro Plus или Enterprise. Для её использования необходимо [приобрести редакцию Pro ãPro Plus или Enterprise](https://www.tencentcloud.com/document/product/1047/34577), перейти на [консоль](https://console.tencentcloud.com/im/login-message), выбрать **Конфигурация функций** > **Вход и сообщения** > **Отправка всем пользователям** и включить функцию.

Ниже приведен образец базового запроса на отправку обычного сообщения в группе:

```
{    "GroupId": "@TGS#2C5SZEAEF",    "Random": 8912345, // Случайное число. Если случайные числа двух сообщений одинаковы в течение пяти минут, они считаются одним сообщением. #### Список электронной почты

Хранение списка исторической электронной почты эквивалентно хранению сообщений. Оно состоит из хранения исторических персональных сообщений и хранения исторических групповых сообщений. Поскольку групповой чат требует как минимум двух пользователей, вы можете создать группу, содержащую учётную запись, указанную администратором, и пользователя, получающего электронное письмо, для хранения.

> **Примечание:** Для редакций Free Trial и Standard период хранения составляет семь дней. Для редакции Pro период хранения составляет 30 дней. Для редакций Pro Plus и Enterprise период хранения составляет 90 дней. Редакции Pro и Pro Plus и Enterprise позволяют продлить период хранения. Вы можете войти в [консоль](https://console.tencentcloud.com/im), чтобы изменить соответствующую конфигурацию.
> Продление периода хранения исторических сообщений — платная услуга стоимостью. Дополнительную информацию о счётах см. в разделе [Цены на услуги стоимостью](https://www.tencentcloud.com/zh/document/product/1047/34350).

Когда сеть в норме, будут получены последние облачные данные; когда сеть нарушена, SDK возвращает локально хранившиеся исторические сообщения. Поддерживается разбиение по страницам. Ниже приведен пример кода Unity для получения списка исторических электронных писем:

#### Количество непрочитанных электронных писем

Запись пользователя-системного письма эквивалентна переписке в чате. Tencent Cloud Chat предоставляет функцию подсчёта непрочитанных сообщений переписки для напоминания пользователям о непрочитанных сообщениях. Когда пользователь переходит в переписку и возвращается в список переписок, счётчик непрочитанных сообщений очищается. Ниже приведен пример кода Unity:

```

Дополнительную информацию см. в разделе [Unity — количество непрочитанных сообщений переписки](https://intl.cloud.tencent.com/document/product/1047/48846).

#### Отправка писем всем участникам

Отправка писем всем участникам — это отправка писем всем игрокам в игре. Tencent Cloud Chat предоставляет функцию отправки всем участникам на стороне сервера. Ниже приведен пример кода:

```
https://adminapisgp.im.qcloud.com/v4/all_member_push/im_push?usersig=xxx&identifier=admin&sdkappid=88888888&random=99999999&contenttype=json
```

Ниже приведен пример запроса:

```
{    "From_Account": "admin",    "MsgRandom": 56512,    "MsgLifeTime": 120, // Автономное хранилище на 120 с (две минуты)    "MsgBody":[        {        "MsgType": "TIMTextElem",        "MsgContent":{            "Text": "привет, красавица"            }        }    ]}
```

Функция отправки сообщений всем участникам позволяет установить период автономного хранилища, так что даже если некоторые пользователи не в сети, они могут получать сообщения в течение указанного периода автономного хранилища. Настройте `MsgLifeTime` (единица измерения: секунда), чтобы указать период автономного хранилища: максимальный период составляет 604 800 секунд (семь дней). Значение по умолчанию — `0`, что указывает на то, что сообщения не хранятся автономно.

Дополнительную информацию об отправке всем пользователям см. в разделе [Отправка всем пользователям](https://intl.cloud.tencent.com/document/product/1047/37166).

#### Период действия электронного письма

Для редакций Free Trial и Standard период хранения составляет семь дней. Для редакции Pro период хранения составляет 30 дней. Для редакций Pro Plus и Enterprise период хранения составляет 90 дней. Редакции Pro и Pro Plus и Enterprise позволяют продлить период хранения. Дополнительную информацию о хранении исторических писем см. в разделе [Параметры периода хранения исторических сообщений](https://intl.cloud.tencent.com/document/product/1047/34419).
Кроме того, при отправке сообщения на стороне сервера вы можете установить `MsgLifeTime` для указания периода автономного хранилища сообщения (до семи дней). Если `MsgLifeTime` равен `0`, это означает, что сообщение отправляется только онлайн-пользователям и не хранится автономно. Дополнительную информацию см. [здесь](https://intl.cloud.tencent.com/document/product/1047/34919).

### Временная комплектация команд

Временная комплектация команд необходима в сетевых многопользовательских играх. Ниже описаны сценарии формирования команд и способы получения информации о команде фоном и членами команды.

#### Сценарии на основе команд

- **Создание команды перед началом игры**: Когда первый игрок входит в игру, сервер автоматически создаёт группу и может быть указано максимальное количество членов группы. Если запрос указывает владельца группы или членов группы, указанные владелец группы или члены группы будут автоматически добавлены в группу при создании группы. Ниже приведен пример URL запроса:

```
https://adminapisgp.im.qcloud.com/v4/group_open_http_svc/create_group?sdkappid=88888888&identifier=admin&usersig=xxx&random=99999999&contenttype=json
```

Ниже приведен базовый пример запроса:

```
{  "Owner_Account": "leckie", // UserId владельца группы (необязательно)  "Type": "Public", // Тип группы: Private, Public, ChatRoom, AVChatRoom или Community  "Name": "TestGroup", // Имя группы (обязательно)  "MaxMemberCount":5 // Максимальное количество членов группы (необязательно)}
```

> **Примечание:** Приложение поддерживает до 100 000 групп. За дополнительные группы требуется определённая плата, как описано в разделе [Цены](https://intl.cloud.tencent.com/document/product/1047/34350). Дополнительную информацию о создании группы на стороне сервера см. в разделе [Создание группы](https://intl.cloud.tencent.com/document/product/1047/34895).

- **Добавление членов группы**: Если новые игроки входят в игру после создания группового чата, вам нужно добавить новых игроков в существующую группу. Ниже приведен пример URL запроса:

```
https://adminapisgp.im.qcloud.com/v4/group_open_http_svc/add_group_member?sdkappid=88888888&identifier=admin&usersig=xxx&random=99999999&contenttype=json
```

Ниже приведен пример запроса:

```
{  "GroupId": "@TGS#2J4SZEAEL", // Группа, в которую нужно добавить членов (обязательно)  "MemberList": [ // За один раз можно добавить до 300 членов.      {          "Member_Account": "tommy" // ID члена, который нужно добавить (обязательно)      },      {          "Member_Account": "jared"  }]}
```

Дополнительную информацию см. в разделе [Добавление членов группы](https://intl.cloud.tencent.com/document/product/1047/34921).

- **Webhook для успешного формирования команды**: Если максимальное количество членов группы установлено при создании группы, игра может начаться только когда все требуемое количество членов находится в группе. Когда получено `webhook после заполнения группы`, игра может быть запущена. Ниже приведен пример URL запроса:

```
https://www.example.com?SdkAppid=$SDKAppID&CallbackCommand=$CallbackCommand&contenttype=json&ClientIP=$ClientIP&OptPlatform=$OptPlatform
```

Ниже приведен пример запроса:

```
{  "CallbackCommand": "Group.CallbackAfterGroupFull", // Команда webhook  "GroupId": "@TGS#2J4SZEAEL" // ID группы}
```

Дополнительную информацию см. в разделе [После заполнения группы](https://intl.cloud.tencent.com/document/product/1047/34376).

- **Уведомление о присоединении нового члена к группе**: Когда новый игрок входит в игру (групповой чат), система отправляет `webhook после присоединения пользователя к группе`, чтобы уведомить других членов групп

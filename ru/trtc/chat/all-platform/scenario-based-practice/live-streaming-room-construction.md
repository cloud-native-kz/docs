# Конструирование комнаты прямого эфира

По мере того как прямая трансляция стала повсеместной, все больше предприятий и разработчиков строят свои собственные платформы прямого эфира. В этом процессе они должны работать со сложными требованиями, такими как отправка и получение потока прямого эфира, транскодирование в прямом эфире, захват экрана в прямом эфире, микширование потока прямого эфира, чат в прямом эфире, взаимодействие в комнате прямого эфира (лайки, подарки и совместное вещание) и управление статусом комнаты прямого эфира. В этом документе в качестве примеров используются продукты Tencent Cloud ([Tencent Chat](https://trtc.io/document/45912?platform=web&product=chat&menulabel=uikit) и [CSS](https://console.tencentcloud.com/live/livestat)) для описания реализации комнаты прямого эфира, а также возможные проблемы и соображения, дающие представление о бизнесе и требованиях прямого эфира.

Во время прямого эфира администратор может отправлять сообщения в комнату прямого эфира и отключать (удалять) несоответствующих пользователей, что можно сделать через [API-интерфейсы сервера Chat](https://trtc.io/document/34620?product=chat&menulabel=restfulapi). ## 2. Руководство по разработке функций комнаты прямого эфира

### Статус комнаты прямого эфира

Комната прямого эфира имеет следующие статусы:

| № | Статус комнаты прямого эфира |
| --- | --- |
| 1 | Должны быть начаты |
| 2 | В прямом эфире |
| 3 | Приостановлено |
| 4 | Завершено |
| 5 | Воспроизведено в обратном порядке |

Статусы комнаты прямого эфира имеют следующие характеристики:

| № | Характеристики |
| --- | --- |
| 1 | Пользователи в комнате прямого эфира должны быть уведомлены в реальном времени об изменении статуса комнаты прямого эфира. |
| 2 | Новые пользователи в комнате прямого эфира должны получить текущий статус комнаты прямого эфира. |

Предусмотрены две схемы реализации, анализ их плюсов и минусов приведен ниже.

| Схема реализации | Плюсы | Минусы |
| --- | --- | --- |
| Серверная часть бизнеса поддерживает статус комнаты прямого эфира и использует API сервера Chat для отправки [пользовательских групповых сообщений](https://intl.cloud.tencent.com/document/product/1047/34959) для уведомления пользователей о статусе. | Когда статус комнаты прямого эфира нужно получать часто, хранение статуса на серверной части бизнеса сокращает частоту вызова SDK Chat по сравнению с хранением статуса в профиле группы Chat. Эта схема реализации делает возможным получение статуса комнаты прямого эфира, когда SDK Chat не интегрирован. | Серверная часть бизнеса должна предоставить дополнительный модуль для чтения и записи статуса комнаты прямого эфира. При получении данных комнаты прямого эфира более вероятны исключения, так как данные поступают как с серверной части бизнеса, так и с профиля группы Chat. Пользовательские сообщения могут быть потеряны при отправке. Когда имеется большое количество сообщений, низкоприоритетные сообщения будут отброшены в первую очередь, что влияет на отображение статуса комнаты прямого эфира. Поэтому мы рекомендуем использовать пользовательские сообщения с высоким приоритетом. |
| Вы можете сохранить статус комнаты прямого эфира через пользовательское поле группы или атрибут группы и уведомить пользователей группы через [обратный вызов для изменения атрибута группы](https://intl.cloud.tencent.com/document/product/1047/48175) клиентского SDK. | Вам не нужно предоставить дополнительный модуль для чтения и записи статуса комнаты прямого эфира. Теоретически обратный вызов для изменения атрибута группы не будет потерян. Данные комнаты прямого эфира получаются из профиля группы, который служит единым источником данных, уменьшая исключения. | Вам нужно часто получать профиль группы в модуле с высокой посещаемостью, что увеличивает нагрузку на Chat. Если модуль SDK Chat не интегрирован, вам нужно вызвать SDK сервера Chat на серверной части бизнеса для получения профиля группы, и количество вызовов ограничено. |

На основе приведенного выше анализа мы рекомендуем комбинировать две схемы для поддержки статуса комнаты прямого эфира:

1. В комнате прямого эфира используйте схему 1 для получения статуса комнаты прямого эфира.
2. Вне комнаты прямого эфира используйте схему 2 для получения статуса комнаты прямого эфира.
3. После изменения статуса комнаты прямого эфира на серверной части бизнеса используйте [API сервера Chat](https://intl.cloud.tencent.com/document/product/1047/34621) для своевременной синхронизации данных с профилем группы Chat ([атрибут группы](https://intl.cloud.tencent.com/document/product/1047/44188) и [пользовательское поле группы](https://intl.cloud.tencent.com/document/product/1047/34962)).

### Выбор типа группы

Раздел чата пользователя в сценариях прямого эфира имеет следующие характеристики:

1. Пользователи часто присоединяются и покидают группу, и информация о разговоре в группе (количество непрочитанных и `lastMessage`) не требует управления.
2. Пользователи могут присоединиться к группе без утверждения.
3. Пользователи отправляют сообщения, не беспокоясь об истории чата.
4. Обычно в группе большое количество участников.
5. Информация о членах группы не требует сохранения.

Поэтому вы можете выбрать **аудиовидеогруппы (AVChatRoom)** как **тип группы** для комнаты прямого эфира на основе [характеристик группы](https://intl.cloud.tencent.com/document/product/1047/33529) Chat.

Аудиовидеогруппы IM (AVChatRooms) имеют следующие функции:

- **Поддержка интерактивных сценариев прямого эфира с неограниченным количеством участников группы.**
- Сообщения (системные уведомления группы) могут быть отправлены всем онлайн-участникам.
- Пользователи могут войти в группу непосредственно без требования утверждения администратора.

> **Примечание:** IM SDK для веб-сайтов позволяет пользователям присоединяться только к одной аудиовидеогруппе (AVChatRoom) одновременно. Если пользователь входит в клиент и входит в комнату прямого эфира A, многоклиентский вход включен в [консоли](https://console.trtc.io/chat/login-message), и пользователь входит в другой клиент и входит в комнату прямого эфира B, пользователь будет удален из комнаты прямого эфира A.

### Объявление в комнате прямого эфира

Объявление в комнате прямого эфира (тема) необходимо для каждой комнаты прямого эфира и может быть просмотрено пользователями после входа в комнату. Кроме того, членов аудиовидеогруппы необходимо уведомлять об изменениях объявления в реальном времени. Вы можете сохранить объявление комнаты прямого эфира на серверной части бизнеса или в профиле группы IM, как и статус комнаты прямого эфира; однако есть некоторые дополнительные вещи, на которые требуется ваше внимание:

1. Поля **Введение группы** и **Объявление группы** **профиля группы** могут содержать до **240** и **300** байт соответственно.
2. Поля **Введение группы** и **Объявление группы** **профиля группы** поддерживают только тип строки. Для создания объявления с изображениями и текстом вы можете использовать тип строки JSON, и изображения должны быть доступными в интернете URL-адресами.
3. Если объявление устанавливается через редактор в веб-сети, точнее говоря, оно содержит тег HTML, оно может быть не проанализировано на Native.

Объявление комнаты прямого эфира может быть установлено через [API-интерфейсы сервера](https://intl.cloud.tencent.com/document/product/1047/34620) или атрибут группы в клиентском SDK. ### Приоритет сообщения

Комната прямого эфира характеризуется большим количеством сообщений пользователей. Каждый пользователь может часто отправлять сообщения. Когда количество отправленных сообщений достигает порога контроля частоты IM, серверная часть IM будет отбрасывать некоторые сообщения для обеспечения стабильности системы. Необходимо применить такой порог, так как сообщения менее читаемы, когда клиент получает их слишком много.

Групповые сообщения делятся на три уровня по приоритету. Серверная часть доставляет сообщения с высоким приоритетом в первую очередь. Поэтому выбирайте соответствующие приоритеты в зависимости от важности сообщений.
Ниже приведены приоритеты от высокого к низкому:

| Приоритет | Описание |
| --- | --- |
| High | Высокий приоритет |
| Normal | Средний приоритет |
| Low | Низкий приоритет |

Сообщения отбрасываются в соответствии со следующей политикой:

Ограничение контроля частоты на основе количества ограничивает максимальное количество сообщений, отправляемых в секунду в одной группе. Значение по умолчанию — 40 сообщений в секунду. Когда количество отправленных сообщений превышает лимит, серверная часть будет сначала доставлять сообщения с более высоким приоритетом, сообщения с одинаковым приоритетом доставляются случайным образом.

Сообщение, которое было ограничено контролем частоты, не доставляется и не сохраняется в истории сообщений, но отправителю будет возвращен ответ об успехе. [Обратный вызов перед доставкой группового сообщения](https://intl.cloud.tencent.com/document/product/1047/34374) активируется, но [обратный вызов после доставки группового сообщения](https://intl.cloud.tencent.com/document/product/1047/34375) не активируется.

Поэтому необходимо установить приоритеты сообщений для комнаты прямого эфира.

Сообщения в комнате прямого эфира имеют приоритет на основе их важности следующим образом:

1. Сообщения вознаграждения и подарков, которые должны видеть все пользователи.
2. Текстовые, аудио- и видеосообщения.
3. Сообщения с лайками.

> **Примечание:** В целом, сообщения, отправленные клиентскими пользователями, отображаются на экране независимо от их приоритета. Когда имеется большое количество сообщений, для пользователей приемлемо потерять небольшое количество сообщений от других пользователей.

# Список разговоров

## Обзор

После входа пользователя в приложение список последних разговоров можно отобразить, чтобы упростить поиск целевого разговора.

Функции списка разговоров включают получение списка разговоров и обработку обновлений списка разговоров. В этом разделе описывается реализация таких функций.

## Базовое получение

Вы можете вызвать `getConversationList` ([Java](https://im.sdk.qcloud.com/doc/en/classcom_1_1tencent_1_1imsdk_1_1v2_1_1V2TIMConversationManager.html#a1bb5ba2beecb4f68146e7f664124fd8b) / [Swift](https://im.sdk.qcloud.com/doc/en/swift_V2TIMManager+Conversation.html#v2timmanager.getconversationlist(nextseq:count:succ:fail:)) / [Objective-C](https://im.sdk.qcloud.com/doc/en/categoryV2TIMManager_07Conversation_08.html#af94d9d44e90da448a395e6d92b4e512e) / [C++](https://im.sdk.qcloud.com/doc/en/classV2TIMConversationManager.html#a05675f2f0c00aedc2af7a2cd6cf2eb6b)) для получения списка разговоров. Этот API получает локально кэшированные разговоры. Если какой-либо разговор на сервере будет обновлён, SDK автоматически синхронизирует обновление и уведомит вас через обратный вызов `V2TIMConversationListener`.

Разговоры пользователя возвращаются в список, содержащий объекты `V2TIMConversation`. SDK сортирует разговоры в соответствии со следующими правилами:

- Начиная с версии v5.5.892, полученные разговоры сортируются по `orderKey` по умолчанию. Чем больше значение `orderKey` разговора, тем выше позиция разговора в списке. Поле `orderKey` — это целое число, которое увеличивается при активации разговора при отправке/получении сообщения, установке черновика или закреплении разговора в верхней части.
- В версиях ранее v5.5.892 полученные разговоры сортируются по `lastMessage` -> `timestamp` разговора по умолчанию. Чем больше значение `timestamp`, тем выше позиция разговора в списке.

> **Внимание** В некоторых случаях `lastMessage` разговора может быть пусто, например, когда сообщения в разговоре удалены. Если вы используете SDK ранее версии v5.5.892, необходимо обработать исключение при сортировке разговоров по `lastMessage`. Рекомендуется обновить SDK до версии v5.5.892 или выше и сортировать разговоры по `orderKey`.

Вы можете использовать `getConversationList` для реализации однократного получения или постраничного получения, как описано ниже.

### Однократное получение

Однократное получение подходит для сценариев с небольшим количеством разговоров (до 100). Вы можете установить `count` получения в `INT_MAX` (что, как правило, больше, чем количество разговоров).

### Постраничное получение

Если количество разговоров велико, постраничное получение рекомендуется для улучшения эффективности загрузки и экономии трафика сети. Рекомендуемое количество разговоров, получаемых за одну страницу, составляет до 100.

Указания:

1. При первом вызове `getConversationList` установите `nextSeq` в `0` (что указывает на получение списка разговоров с начала) и `count` в `50` (что указывает на получение 50 объектов разговоров за один раз).
2. После успешного первого получения списка разговоров обратный вызов `V2TIMConversationResult` для `getConversationList` будет содержать `nextSeq` (поле для следующего получения) и `isFinish` (указывающий, завершено ли получение разговоров).
  - Если `isFinished` имеет значение `true`, все разговоры получены.
  - Если `isFinished` имеет значение `false`, есть больше разговоров, которые можно получить. Это не означает, что следующая страница списка разговоров будет получена сразу же. В обычном программном обеспечении постраничное получение часто срабатывает при операции прокрутки.
3. Когда пользователь продолжает прокручивать список разговоров вверх, если есть больше разговоров, которые можно получить, вы можете продолжить вызов API `getConversationList` и передать параметры `nextSeq` (значение берётся из объекта `V2TIMConversationResult`, возвращённого последним получением) и `count` для следующего получения.
4. Повторяйте шаг 3 до `isFinished` = `true`.

## Продвинутое получение

Если базовый API, описанный выше, не может удовлетворить ваши потребности в получении списка разговоров, вы можете использовать продвинутый API `getConversationListByFilter` ([Java](https://im.sdk.qcloud.com/doc/en/classcom_1_1tencent_1_1imsdk_1_1v2_1_1V2TIMConversationManager.html#abf71156b8b6423e98943e25a77dc1967)/ [Swift](https://im.sdk.qcloud.com/doc/en/swift_V2TIMManager+Conversation.html#v2timmanager.getconversationlistbyfilter(filter:nextseq:count:succ:fail:)) / [Objective-C](https://im.sdk.qcloud.com/doc/en/categoryV2TIMManager_07Conversation_08.html#ac1b77eedff7f2f8742a873cf766daec9)/[C++](https://im.sdk.qcloud.com/doc/en/classV2TIMConversationManager.html#a7a7e159591628f2004e75d16fa0e55af)). Здесь `V2TIMConversationListFilter` разговоров получается, как описано ниже:

| Атрибут | Определение | Описание |
| --- | --- | --- |
| type | Тип разговора (`0` означает отсутствие фильтрации по этому атрибуту) | Индивидуальный или групповой разговор. |
| conversationGroup | Имя группы разговоров (пусто означает отсутствие фильтрации по этому атрибуту) | Это не название группы, а название группы разговоров. Дополнительную информацию см. в разделе [Группа разговоров](https://www.tencentcloud.com/document/product/1047/48854). |
| markType | Тип тега разговора (`0` означает отсутствие фильтрации по этому атрибуту) | Дополнительную информацию см. в разделе [Тег разговора](https://intl.cloud.tencent.com/document/product/1047/48853). |
| hasUnreadCount | Разговор содержит количество непрочитанных сообщений | true: возвращать разговоры с количеством непрочитанных сообщений; false: возвращать все разговоры. |
| hasGroupAtInfo | Разговор содержит групповые сообщения @ | true: возвращать разговоры, содержащие групповые сообщения @; false: возвращать все разговоры. |

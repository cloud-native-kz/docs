# Исторические сообщения

## Обзор

- API для получения исторических сообщений находится в классе `TencentImSDKPlugin.v2TIMManager.getMessageManager()`.
- В дополнение к поддержке получения исторических личных и групповых сообщений предоставляется расширенный API для получения сообщений по последовательности, начальной точке или диапазону времени.
- Возможно получение как локальных, так и облачных исторических сообщений.

> **Примечание:** Когда исторические сообщения получаются из облака и обнаруживается сетевое исключение, SDK вернет локально сохраненные исторические сообщения.

Локально сохраненные исторические сообщения не подлежат ограничениям по времени, но те, которые хранятся в облаке, подлежат следующим ограничениям по времени:

- Пробная версия: период бесплатного хранения составляет 7 дней и не может быть продлен.
- Профессиональная версия: период бесплатного хранения составляет 30 дней и может быть продлен.
- Профессиональная Plus версия: период бесплатного хранения составляет 90 дней и может быть продлен.
- Корпоративная версия: период бесплатного хранения составляет 90 дней и может быть продлен.

> **Примечание:** Продление периода хранения исторических сообщений является платной услугой. Вы можете войти в [консоль Chat](https://console.trtc.io/chat), чтобы изменить соответствующую конфигурацию. Информацию о расценках см. в разделе [Расценки на дополнительные услуги](https://intl.cloud.tencent.com/document/product/1047/34350). Сообщения с богатым контентом (такие как изображения, файлы и аудио) имеют такие же периоды хранения, как исторические сообщения.

## Получение исторических личных сообщений

Вызовите API `getC2CHistoryMessageList` ([подробности](https://pub.dev/documentation/tencent_cloud_chat_sdk/latest/manager_v2_tim_message_manager/V2TIMMessageManager/getC2CHistoryMessageList.html)), чтобы получить исторические личные сообщения.

При нормальном состоянии сети будут получены последние данные облака; при наличии сбоев SDK вернет локально сохраненные исторические сообщения.

Если вы хотите получить только локальные исторические сообщения, см. раздел [Расширенный API](#advance).

Этот API поддерживает получение по страницам. Дополнительные сведения см. в разделе [Получение по страницам](#advance_page).

Пример кода:

```
// Получить исторические личные сообщения
// Установите `lastMsgID` на `null` для первого получения
// `lastMsgID` может быть ID последнего сообщения в возвращаемом списке сообщений для второго получения.
TencentImSDKPlugin.v2TIMManager.getMessageManager().getC2CHistoryMessageList(
        userID: "userId",
        count: 10,
        lastMsgID: null,
);
```

## Получение исторических групповых сообщений

Вызовите API `getGroupHistoryMessageList` ([подробности](https://pub.dev/documentation/tencent_cloud_chat_sdk/latest/manager_v2_tim_message_manager/V2TIMMessageManager/getGroupHistoryMessageList.html)), чтобы получить исторические групповые сообщения.

При нормальном состоянии сети будут получены последние данные облака; при наличии сбоев SDK вернет локально сохраненные исторические сообщения.

Если вы хотите получить только локальные исторические сообщения, см. раздел [Расширенный API](#advance).

Этот API поддерживает получение по страницам. Дополнительные сведения см. в разделе [Получение по страницам](#advance_page).

> **Примечание:** Могут быть получены только исторические сообщения групп совещаний (Meeting). Дополнительные сведения об ограничениях на групповые сообщения см. в разделе [Система групп](https://intl.cloud.tencent.com/document/product/1047/33529). Этот API не применяется к аудиовидеогруппам (AVChatRoom), так как их сообщения не хранятся в облаке или локальной базе данных.

Пример кода:

```
// Получить исторические групповые сообщения
// Установите `lastMsgID` на `null` для первого получения
// `lastMsgID` может быть ID последнего сообщения в возвращаемом списке сообщений для второго получения.
TencentImSDKPlugin.v2TIMManager
      .getMessageManager()
      .getGroupHistoryMessageList(
        groupID: "groupID",
        count: 10,
        lastMsgID: null,
      );
```

## Расширенные возможности

### Расширенный API

Если обычные API выше не могут удовлетворить ваши потребности в получении исторических сообщений, вы можете использовать расширенный API `getHistoryMessageList` ([подробности](https://pub.dev/documentation/tencent_cloud_chat_sdk/latest/manager_v2_tim_message_manager/V2TIMMessageManager/getHistoryMessageList.html)).

В дополнение к получению исторических личных и групповых сообщений этот API поддерживает следующие расширенные возможности:

- Установка источника для получения сообщений: получение из локальной базы данных или облака
- Указание последовательности для получения сообщений: получение в обратном хронологическом порядке или в хронологическом порядке
- Указание типа сообщения для локального получения: текстовое, изображение, аудио, видео, файл, эмодзи, подсказка группы, комбинированное или пользовательское сообщение.

Прототип API:

```
    // Получить исторические личные сообщения
    // Установите `lastMsgID` на `null` для первого получения
    // `lastMsgID` может быть ID последнего сообщения в возвращаемом списке сообщений для второго получения.
    V2TimValueCallback<List<V2TimMessage>> getHistoryMessageListRes =
        await TencentImSDKPlugin.v2TIMManager
            .getMessageManager()
            .getHistoryMessageList(
      getType: HistoryMsgGetTypeEnum.V2TIM_GET_LOCAL_OLDER_MSG, // Источник и последовательность получения сообщений
      userID: "userID", // Чтобы получить личные сообщения с определенным пользователем, необходимо указать только `userID`.
      groupID: "groupID", // Чтобы получить групповые сообщения из определенной группы, необходимо указать только `groupID`.
      count: 10, // Количество полученных сообщений
      lastMsgID: null, // ID первого сообщения для получения
      // Это поле можно использовать только для групповых чатов.
      // Если `lastMsgSeq` установлен как начальная точка для сообщения, сообщение будет включено в возвращаемый список сообщений.
      // Если указаны оба `lastMsg` и `lastMsgSeq`, SDK будет использовать `lastMsg`.
      // Если ни `lastMsg`, ни `lastMsgSeq` не указаны, начальная точка для получения будет определена на основании установки `getTimeBegin`. Если да, заданный диапазон будет использован как начальная точка; если нет, последнее сообщение будет использовано как начальная точка.
      lastMsgSeq: -1,
      messageTypeList: [], // Используется для фильтрации атрибутов информации исторических сообщений. Если это поле остается пустым, будут получены все атрибуты.
    );
    if (getHistoryMessageListRes.code == 0) {
      // Успешно получено
    }
```

Параметры:

| Параметр | Описание | Действителен для личного чата | Действителен для группового чата | Обязательный | Замечания |
| --- | --- | --- | --- | --- | --- |
| getType | Источник и последовательность получения сообщений, которые можно установить на **локальный/облачный** и **обратный хронологический порядок/хронологический порядок** соответственно. | ДА | ДА | ДА | Когда источник получения установлен на облако, локальный список сообщений и список облачных сообщений будут объединены и возвращены. Если нет сетевого подключения, будет возвращен локальный список сообщений. |
| userID | Указанный ID пользователя, с которым получаются исторические личные сообщения | ДА | НЕТ | НЕТ | Чтобы получить личные сообщения, необходимо указать только `userID`. |
| groupID | Указанный ID группы, из которой получаются исторические групповые сообщения | НЕТ | ДА | НЕТ | Чтобы получить групповые сообщения, необходимо указать только `groupID`. |
| count | Количество сообщений за одно получение | ДА | ДА | ДА | Рекомендуется установить значение `20` для клиентов iOS/Android; в противном случае скорость получения может быть затронута. Для веб-клиентов максимально допустимое значение составляет `15`. |
| lastMsgID | ID последнего сообщения, указывающий сообщение, начиная с которого получаются исторические сообщения | ДА | ДА | НЕТ | Может использоваться как для личных, так и для групповых чатов. Если оно установлено как начальная точка для получения сообщений, сообщение не будет включено в возвращаемый список сообщений. Если оно остается пустым, последнее сообщение в разговоре будет использовано как начальная точка для получения. |
| lastMsgSeq | `seq` последнего сообщения, указывающий сообщение, начиная с которого получаются исторические сообщения | НЕТ | ДА | НЕТ | Может использоваться только для групповых чатов. Если оно установлено как начальная точка для получения сообщений, сообщение будет включено в возвращаемый список сообщений. Если указаны оба lastMsg и lastMsgSeq, SDK будет использовать lastMsg. Если ни lastMsg, ни lastMsgSeq не указаны, начальная точка для получения будет определена на основании установки getTimeBegin. Если да, заданный диапазон будет использован как начальная точка; если нет, последнее сообщение будет использовано как начальная точка. |

### Получение по страницам

- Получение по страницам может быть реализовано с использованием `lastMsg` и `count`. Для первого получения `lastMsg` может быть оставлен пустым, что указывает на то, что последнее сообщение в возвращаемом списке сообщений будет использовано как `lastMsg` для получения данных следующей страницы.
- Когда `lastMsg` остается пустым, SDK по умолчанию вернет исторические сообщения, начиная с последнего сообщения.
- Когда `lastMsg` установлен, он не включается в возвращаемый список сообщений.
- Возвращаемый список сообщений отображает сообщения в обратном хронологическом порядке.

> **Примечание:** Чтобы избежать влияния на скорость получения исторических сообщений, рекомендуется установить `count` на `20` для разбиения на страницы. Рекомендуется не использовать `lastMsgSeq` для последующего получения, так как соответствующее сообщение будет включено в возвращаемый список сообщений.

### Получение только локальных сообщений

Установите `getType` для получения только локальных сообщений:

- Когда `getType` установлен на `V2TIM_GET_LOCAL_OLDER_MSG`, локально сохраненные сообщения будут получены в обратном хронологическом порядке.
- Когда `getType` установлен на `V2TIM_GET_LOCAL_NEWER_MSG`, локально сохраненные сообщения будут получены в хронологическом порядке.

Следующий пример кода демонстрирует процесс получения 20 личных сообщений из локальной базы данных в обратном хронологическом порядке, начиная с последнего сообщения:

```
TencentImSDKPlugin.v2TIMManager.getMessageManager().getHistoryMessageList(
        count: 10,
        getType: HistoryMsgGetTypeEnum.V2TIM_GET_LOCAL_OLDER_MSG,
        userID: "userID",
);
```

### Получение сообщений после перенаправления на групповое @ сообщение

Следующий пример кода демонстрирует процесс нажатия на @ панель, перенаправления на групповое @ сообщение и получения первых 20 сообщений раньше и позже этого сообщения для отображения:

```
// Получить `последовательность` группового @ сообщения
int atSequence = 1081;
// Получить групповое @ сообщение и более ранние сообщения
  TencentImSDKPlugin.v2TIMManager.getMessageManager().getHistoryMessageList(
        count: 20, // Получить 20 сообщений
        getType: HistoryMsgGetTypeEnum.V2TIM_GET_CLOUD_OLDER_MSG, // Получить сообщения раньше группового @ сообщения
        lastMsgSeq: atSequence,// Начать получение с группового @ сообщения, которое включено в список получения
        groupID: "groupID" // Получить групповые сообщения
      );
// Получить сообщения позже группового @ сообщения
TencentImSDKPlugin.v2TIMManager.getMessageManager().getHistoryMessageList(
        count: 20, // Получить 20 сообщений
        getType: HistoryMsgGetTypeEnum.V2TIM_GET_CLOUD_NEWER_MSG, // Получить сообщения позже группового @ сообщения
        lastMsgSeq: atSequence,// Начать получение с группового @ сообщения, которое включено в список получения
        groupID: "groupID" // Получить групповые сообщения
      );
```

## Часто задаваемые вопросы

### 1. Что делать, если в журнале отображается "total count of request cloud message exceed max limit" при получении исторических сообщений?

1. Когда `getType` установлен на получение облачных исторических сообщений, а количество сообщений для получения установлено на `x`, SDK получит `x` сообщений из облака.
2. SDK отфильтрует недействительные сообщения, такие как удаленные сообщения и те, которые не относятся к текущему пользователю.
3. Если в облаке слишком много недействительных исторических сообщений, SDK будет выполнять получение с разбиением на несколько страниц.
Чтобы обеспечить стабильность и надежность системы, SDK будет выполнять не более трех автоматических получений с разбиением на страницы. По истечении этого лимита в журнале будет отображено сообщение "total count of request cloud message exceed max limit".

Чтобы минимизировать влияние такого механизма ограничения на уровень приложения, вы можете принять следующие меры для уменьшения недействительных сообщений:

- Вы можете использовать онлайн-сообщения, то есть установить `onlineUserOnly` на `YES/true` при отправке сообщений.
- Для групповых сообщений вы можете использовать целевые групповые сообщения для указания получателя.

### 2. Что делать, если сообщения "теряются" при получении облачных исторических сообщений?

1. Получить `count` сообщений из локальной базы данных.
2. Получить `count` сообщений из облака и отфильтровать недействительные сообщения, такие как удаленные сообщения. Если число меньше `count`, будет запущено получение с разбиением на страницы.
3. Объединить локальные и облачные сообщения и обновить информацию, такую как статус сообщения.
4. Вернуть `count` сообщений из объединенного списка сообщений.

Вообще говоря, потеря сообщений указывает на то, что на шаге 2 получено слишком много недействительных сообщений, что запускает [механизм ограничения в вопросе 1](#qa1) и приводит к недостаточному количеству сообщений, фактически полученных из облака.
Рекомендуется исправить эту проблему, как указано в [вопросе 1](#qa1).

### 3. Что делать, если информация о членах группы, такая как визитная карточка группы, не обновляется в реальном времени при получении исторических сообщений?

- При получении исторических групповых сообщений SDK напрямую вернет информацию о членах группы в момент создания сообщений и не будет обновлять ее в реальном времени.

Если вы хотите получить последнюю информацию о членах группы, вы можете использовать API `getGroupMembersInfo` ([подробности](https://pub.dev/documentation/tencent_cloud_chat_sdk/latest/manager_v2_tim_group_manager/V2TIMGroupManager/getGroupMembersInfo.html)).

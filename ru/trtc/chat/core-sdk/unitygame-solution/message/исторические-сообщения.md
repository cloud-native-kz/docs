#  Исторические сообщения

## Описание функции

- API для получения исторических сообщений — это `MsgGetMsgList` ([Подробнее](https://comm.qq.com/im/doc/unity/en/api/MessageApi/MsgGetMsgList.html)).
- Поддерживает получение исторических сообщений «один-на-один» и групповых сообщений.
- Можно получать как локальные, так и облачные исторические сообщения.

> **Примечание.** Когда исторические сообщения получаются из облака и обнаруживается сетевое исключение, SDK вернёт локально сохранённые исторические сообщения.

Локально сохранённые исторические сообщения не имеют ограничений по времени, но сообщения, сохранённые в облаке, имеют следующие ограничения:

- Бесплатная версия: период бесплатного хранения составляет 7 дней и не может быть продлён.
- Pro-версия: период бесплатного хранения составляет 30 дней и может быть продлён.
- Pro Plus-версия: период бесплатного хранения составляет 90 дней и может быть продлён.
- Enterprise-версия: период бесплатного хранения составляет 90 дней и может быть продлён.

> **Примечание.** Продление периода хранения исторических сообщений является платной услугой. Вы можете войти в [консоль IM](https://console.tencentcloud.com/im) для изменения соответствующей конфигурации. Подробнее о тарификации см. [Цены](https://intl.cloud.tencent.com/document/product/1047/34350). Мультимедийные сообщения (такие как изображения, файлы и аудиозаписи) имеют одинаковые периоды хранения с историческими сообщениями.

## Получение исторических групповых сообщений

Этот API поддерживает получение по страницам. Подробнее см. раздел [Получение по страницам](#advance_page).

> **Внимание.** Можно получать только исторические сообщения групп встреч (Meeting). Подробнее об ограничениях групповых сообщений см. раздел [Система групп](https://intl.cloud.tencent.com/document/product/1047/33529). Этот API не применяется к аудио-видео группам (AVChatRoom), так как их сообщения не хранятся на облачном сервере маршрутизации или в локальной базе данных.

### Получение по страницам

- Получение по страницам можно реализовать, используя `msg_getmsglist_param_last_msg` и `msg_getmsglist_param_count`. При первом получении `msg_getmsglist_param_last_msg` можно оставить пустым, что означает, что последнее сообщение из возвращённого списка сообщений будет использовано как `msg_getmsglist_param_last_msg` для получения данных следующей страницы.
- Когда `msg_getmsglist_param_last_msg` оставлен пустым, SDK по умолчанию вернёт исторические сообщения, начиная с последнего сообщения.
- Когда `msg_getmsglist_param_last_msg` установлен, он не включается в возвращённый список сообщений.
- Возвращённый список сообщений отображает сообщения в обратном хронологическом порядке.

> **Примечание.** Чтобы избежать влияния на скорость получения исторических сообщений, рекомендуется установить `msg_getmsglist_param_count` в значение `20` для разбиения по страницам. Не рекомендуется использовать `msg_getmsglist_param_last_msg_seq` для последующего получения, так как соответствующее сообщение будет включено в возвращённый список сообщений.

## Часто задаваемые вопросы

### 1. Что делать, если при получении исторических сообщений в журнале отображается сообщение «total msg_getmsglist_param_count of request cloud message exceed max limit»?

1. Когда `getType` установлен для получения облачных исторических сообщений и количество получаемых сообщений установлено в `x`, SDK получит `x` сообщений из облака.
2. SDK отфильтрует недействительные сообщения, такие как удалённые сообщения и сообщения, не относящиеся к текущему пользователю.
3. Если в облаке слишком много недействительных исторических сообщений, SDK выполнит несколько получений по страницам.
Для обеспечения стабильности и надёжности системы SDK выполнит максимум три автоматических получения по страницам. После превышения этого предела в журнале будет отображено сообщение «total msg_getmsglist_param_count of request cloud message exceed max limit».

Чтобы минимизировать влияние такого механизма ограничений на уровень бизнеса, вы можете принять следующие меры для сокращения недействительных сообщений:

- Вы можете использовать онлайн-сообщения, то есть установить `message_is_online_msg` в значение `YES/true` при отправке сообщений.
- Для групповых сообщений вы можете использовать адресованные групповые сообщения для указания получателя.

### 2. Что делать, если сообщения «теряются» при получении облачных исторических сообщений?

1. Получите `msg_getmsglist_param_count` сообщений из локальной базы данных.
2. Получите `msg_getmsglist_param_count` сообщений из облака и отфильтруйте недействительные сообщения, такие как удалённые. Если количество меньше `msg_getmsglist_param_count`, будет запущено получение по страницам.
3. Объедините локальные и облачные сообщения и обновите информацию, такую как статус сообщения.
4. Вернитесь к `msg_getmsglist_param_count` сообщениям из объединённого списка сообщений.

Как правило, потеря сообщений указывает на то, что на шаге 2 получено слишком много недействительных сообщений, что вызывает [механизм ограничений в вопросе 1](#qa1) и приводит к недостаточному количеству фактически полученных из облака сообщений.
Рекомендуется исправить эту проблему в соответствии с инструкциями в разделе [вопрос 1](#qa1). Если проблема сохраняется, свяжитесь с нами для получения помощи.

### 3. Что делать, если информация о членах группы, такая как карточка имени группы, не обновляется в реальном времени при получении исторических сообщений?

- При получении исторических групповых сообщений SDK напрямую вернёт информацию о членах группы на момент создания сообщений и не будет обновлять её в реальном времени.

Если вы хотите получить последнюю информацию о членах группы, вы можете использовать API `GroupGetMemberInfoList` ([Подробнее](https://comm.qq.com/im/doc/unity/en/api/GroupApi/GroupGetMemberInfoList.html)).

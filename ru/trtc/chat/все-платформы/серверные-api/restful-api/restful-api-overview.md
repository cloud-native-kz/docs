# Обзор RESTful API

RESTful API — это API управления HTTP, которые предоставляют бэкенду приложения точку входа управления. Дополнительные сведения о поддерживаемых Chat RESTful API см. в разделе [Список RESTful API](https://www.tencentcloud.com/document/product/1047/34621).

Помимо RESTful API, консоль приложения также поддерживает простое управление данными и отправку сообщений один-к-одному/группе. Разработчики могут управлять, просматривать и тестировать данные в консоли. Хотя RESTful API менее удобны для пользователя, они предоставляют мощные возможности управления.

По соображениям безопасности RESTful API поддерживаются только через HTTPS.

## Предварительные требования

Перед вызовом RESTful API выполните следующие операции:

1. Создайте приложение в консоли Chat. Дополнительные сведения см. в разделе [Создание и обновление приложения](https://www.tencentcloud.com/document/product/1047/34577).
2. Назначьте учетную запись администратора приложению. Дополнительные сведения см. в разделе **Настройка администраторов учетных записей** в [Базовая конфигурация](https://www.tencentcloud.com/document/product/1047/34540).

> **Внимание** Чтобы избежать ненужных ошибок вызовов, используйте учетную запись администратора для вызова RESTful API.

## Метод вызова

### URL запроса

Формат URL RESTful API выглядит следующим образом:

```
https://xxxxxx/$ver/$servicename/$command?sdkappid=$SDKAppID&identifier=$identifier&usersig=$usersig&random=99999999&contenttype=json
```

Описание и значения параметров приведены ниже (имена параметров и значения параметров чувствительны к регистру):

| Параметр | Описание | Значение |
| --- | --- | --- |
| https | Протокол запроса | Протокол запроса — HTTPS, метод запроса — POST. |
| xxxxxx | Выделенное имя домена | Китай: console.tim.qq.com (резервный: adminapi.my-imcloud.com) Сингапур: adminapisgp.im.qcloud.com (резервный: adminapisgp.my-imcloud.com) Сеул: adminapikr.im.qcloud.com (резервный: adminapikr.my-imcloud.com) Токио: adminapijpn.im.qcloud.com (резервный: adminapijpn.my-imcloud.com) Франкфурт: adminapiger.im.qcloud.com (резервный: adminapiger.my-imcloud.com) Кремниевая долина: adminapiusa.im.qcloud.com (резервный: adminapiusa.my-imcloud.com) Джакарта: adminapiidn.im.qcloud.com (резервный: adminapiidn.my-imcloud.com) |
| ver | Номер версии протокола | Всегда `v4`. |
| servicename | Имя внутреннего сервиса. Разные значения `servicename` соответствуют разным типам сервисов. | Пример: для `v4/im_open_login_svc/account_import` `im_open_login_svc` — это `servicename`. Дополнительные сведения см. в разделе [Список RESTful API](https://www.tencentcloud.com/document/product/1047/34621). |
| command | Слово команды. Этот параметр используется вместе с параметром `servicename` для определения конкретной функции сервиса. | Пример: для `v4/im_open_login_svc/account_import` `account_import` — это `command`. Дополнительные сведения см. в разделе [Список RESTful API](https://www.tencentcloud.com/document/product/1047/34621). |
| sdkappid | ID приложения, полученный в консоли Chat | Вы можете получить SDKAppID при применении доступа к SDK Chat. |
| identifier | Имя пользователя, которое должно быть учетной записью администратора приложения при вызове RESTful API | Дополнительные сведения см. в разделе **Администратор приложения** в [Аутентификация входа](https://www.tencentcloud.com/document/product/1047/33517). |
| usersig | Пароль, соответствующий имени пользователя. | Дополнительные сведения см. в разделе [Генерация UserSig](https://www.tencentcloud.com/document/product/1047/34385). |
| random | Параметр, используемый для идентификации текущего запроса | Случайное 32-битное целое число без знака в диапазоне от 0 до 4 294 967 295 |
| contenttype | Формат запроса | Всегда `json`. |

> **Примечание** Когда серверное приложение вызывает RESTful API, `identifier` должна быть учетной записью администратора приложения. Приложение может генерировать `UserSig` для учетной записи администратора каждый раз при вызове RESTful API или генерировать фиксированный `UserSig` для повторного использования с периодом действия.

### Формат тела HTTP запроса

RESTful API поддерживают только метод POST, а его тело запроса находится в формате JSON. Дополнительные сведения о формате тела запроса см. в подробном описании каждого API.

Обратите внимание, что тело запроса POST не может быть пустым. Даже когда в теле запроса не требуется никакая информация, тело запроса по-прежнему должно содержать пустой объект JSON (`{}`).

### Код возврата HTTP

Возвращаемый код состояния HTTP для RESTful API всегда 200, за исключением ошибки сети (например, ошибка 502). Конкретный код ошибки и сообщение об ошибке содержатся в теле ответа HTTP.

### Формат тела ответа HTTP

Тело ответа RESTful API находится в формате JSON и имеет следующие элементы:

```
{    "ActionStatus": "OK",     "ErrorInfo": "",     "ErrorCode": 0    // Другое содержимое ответа RESTful API}
```

Тело ответа должно содержать поля `ActionStatus`, `ErrorInfo` и `ErrorCode`. Эти три поля описаны следующим образом:

| Поле | Тип | Описание |
| --- | --- | --- |
| ActionStatus | String | Результат запроса. `OK`: успешно. `FAIL`: не удалось. Если запрос не удается, причина отказа отображается в поле `ErrorInfo`. |
| ErrorInfo | String | Причины отказа |
| ErrorCode | Integer | Код ошибки. `0`: успешно. Другие значения: не удалось. Дополнительные сведения см. в разделе [Коды ошибок](https://intl.cloud.tencent.com/document/product/1047/34348). |

В следующем примере показано, как использовать RESTful API для [получения всех групп в приложении](https://intl.cloud.tencent.com/document/product/1047/34960).

HTTPS запрос:

```
POST /v4/group_open_http_svc/get_appid_group_list?usersig=xxx&identifier=admin&sdkappid=88888888&random=99999999&contenttype=json HTTP/1.1Host: console.tim.qq.comContent-Length: 22{    "Limit": 2}
```

```

```

## Часто задаваемые вопросы

### Истекло время ожидания RESTful API запроса и не было получено никакого ответа.

1. Период тайм-аута, установленный для RESTful API в бэкенде Chat, составляет три секунды. Вам необходимо указать период тайм-аута больше трех секунд.
2. Выполните `telnet console.tim.qq.com 443` для проверки доступности порта сервиса.
3. Выполните `curl -I https://console.tim.qq.com` для проверки кода состояния 200.
4. Проверьте, использует ли машина приватный или открытый DNS-сервер. Если машина использует приватный DNS-сервер, убедитесь, что исходящий трафик DNS-сервера находится в том же регионе, что и исходящий IP-адрес машины интернет-провайдера.
5. Рекомендуется использовать режим «постоянное соединение + пул соединений».

> **Примечание** Рекомендуется использовать постоянное соединение RESTful API для подключения к SDK. Причина в том, что установление постоянных HTTPS соединений требует много времени, поскольку каждый запрос требует служебных данных TCP+TLS handshake.
> Для сценариев, когда используется стандартная библиотека HTTP: для HTTP 1.0 необходимо указать заголовок запроса `Connection: keep-alive`; для HTTP 1.1 постоянные соединения поддерживаются по умолчанию; для сценариев, где HTTPS запросы инкапсулируются на основе TCP, TCP соединения могут быть повторно использованы для отправки и получения запросов.

# API для отправки сообщений всем пользователям

## Обзор функции отправки сообщений всем пользователям

Сервис отправки сообщений всем пользователям представляет собой группу REST API, реализованных на основе архитектуры связи Chat. Этот сервис используется для удовлетворения требований приложения по отправке сообщений, включая отправку сообщений всем пользователям, отправку сообщений по тегам и отправку сообщений по атрибутам. Клиенты могут получать отправляемые сообщения, используя функции онлайн и оффлайн отправки (фоновые уведомления Android и APNs) SDK. Сервис групповой отправки также поддерживает оффлайн хранение сообщений, помогая операторам более эффективно достичь своих целей.

> **Примечание:** Учетная запись может получать сообщения, отправленные всем пользователям, только после входа в систему или ручного импорта. Функция отправки всем пользователям доступна только для пользователей с учетными записями Pro Edition, Pro Plus Edition или Enterprise Edition. См. [Заявку на изменение конфигурации](https://intl.cloud.tencent.com/document/product/1047/44322) для подачи запроса на эту функцию. Функция будет включена **через 48 часов** после одобрения вашего приложения.

## Основные возможности

- Отправка сообщений всем пользователям приложения.
- Отправка сообщений по пользовательским тегам.
- Отправка сообщений по пользовательским атрибутам.

## Преимущества

Сервис групповой отправки предоставляется на основе клиентского SDK Chat и бэкэнда Chat, что гарантирует возможности обмена сообщениями и доступность системы.

- Доставка сообщений гарантирована на основе Chat SDK, и сервис предоставляет простые возможности трансляции сообщений для приложений.
- Для каждого пользователя можно установить до 10 атрибутов отправки. Каждый атрибут можно устанавливать отдельно без влияния на другие.
- Сервис отправляет сообщения на основе нескольких атрибутов, связанных логикой AND или OR.
- Сервис отправляет сообщения на основе нескольких тегов, связанных логикой AND или OR.
- Сервис отправляет сообщения только онлайн пользователям и сохраняет сообщения оффлайн до 7 дней (отзыв и роуминг не поддерживаются).
- Вы можете настраивать сообщения.
- Вы можете указать учетную запись отправителя.

## Сценарии использования

### Отправка сообщений всем пользователям

Пример 1: игровое приложение планирует предоставить специальные предложения на Рождество и должно отправить уведомление всем пользователям. В этом случае сервис групповой отправки может повысить эффективность. Кроме того, чтобы информировать больше пользователей об этой акции, вы можете установить период сохранения оффлайн сообщений. Таким образом, даже если некоторые пользователи не в сети при отправке сообщения, они все равно смогут получить сообщение при входе в сеть в течение этого периода сохранения. Это повышает качество и эффективность акции.
Пример 2: приложение для прямых трансляций планирует запустить крупномасштабное маркетинговое мероприятие в прямом эфире и должно отправить уведомление всем пользователям. В этом случае вы можете отправить уведомление всем пользователям, начиная за 7 дней до мероприятия, и установить период сохранения оффлайн сообщений на 7 дней. Когда мероприятие начнется, вы можете отправить уведомление всем пользователям снова, не указывая период сохранения оффлайн. Таким образом, все онлайн пользователи смогут получить уведомление и присоединиться к прямому эфиру, привлекая больше пользователей.

### Отправка сообщений пользователям по пользовательским тегам

Например, финансовый продукт планирует отправить услугу финансового планирования пользователям, которые следят за "акцией A" или "акцией B". В этом случае вы можете отправить сообщения по тегам:

1. Когда пользователь следит за "акцией A" или "акцией B", добавьте соответствующий тег пользователю, вызвав API для добавления тегов.
2. Когда пользователь перестает следить за "акцией A" или "акцией B", удалите соответствующий тег пользователя, вызвав API для удаления тегов.
3. Установите условие отправки на "акцию A" или "акцию B" (используя функцию TagsOr) в API для отправки сообщений. Таким образом, все пользователи, которые следят за "акцией A" или "акцией B", смогут получить сообщения.

### Отправка сообщений по пользовательским атрибутам

Например, пользователи игры классифицируются на не-членов, обычных членов, золотых членов и платиновых членов. Предположим, оператор планирует отправить мероприятие платиновым членам в Шэньчжэне. В этом случае они могут отправить сообщение по атрибутам:

1. Установите имена атрибутов приложения. В этом примере пользователи игры имеют два атрибута: уровень членства и город. Вы можете установить атрибут 0 как уровень членства и атрибут 1 как город.
2. Когда уровень членства пользователя изменяется (например, при истечении членства или когда пользователь покупает членство), измените атрибут уровня членства пользователя, вызвав API для установки пользовательских атрибутов. Например, когда пользователь покупает золотое членство, установите атрибут уровня членства пользователя на "золотой член".
3. Когда город, где находится пользователь, изменяется, измените атрибут города пользователя, вызвав API для установки пользовательских атрибутов. Например, когда город меняется с Пекина на Шанхай, установите атрибут города пользователя на "Шанхай".
4. Во время мероприятия вы можете вызвать API отправки и установить атрибут уровня членства на платина и атрибут города на Шэньчжэнь. Затем все платиновые члены в Шэньчжэне получат отправленное сообщение.

> **Примечание:** В этом сценарии отправка по атрибутам предпочтительнее отправки по тегам. Если вы отправляете сообщение по тегу, когда город пользователя меняется с Шэньчжэня на Гуанчжоу, приложение должно удалить тег "Шэньчжэнь" пользователя и затем добавить тег "Гуанчжоу", что требует двух вызовов API. Однако если вы отправляете сообщение по атрибуту, требуется только один вызов API для изменения атрибута города пользователя.

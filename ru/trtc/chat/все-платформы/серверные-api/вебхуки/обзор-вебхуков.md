# Обзор вебхуков

## Обзор

Чтобы предоставить вам детальный контроль над функциями приложения, Chat предоставляет мощные вебхуки бесплатно. Вебхуки используют режим постоянного соединения по умолчанию. Вебхук означает, что бэкенд Chat отправляет запрос на бэкенд приложения до или после возникновения события. Это позволяет бэкенду приложения синхронизировать данные, если необходимо, или вмешаться в последующую обработку события. Дополнительную информацию о вебхуках, поддерживаемых Chat, см. в разделе [Список команд вебхуков](https://intl.cloud.tencent.com/document/product/1047/34355).

Вебхук отправляется на бэкенд приложения с помощью HTTP/HTTPS запроса, и бэкенд приложения должен обработать запрос вебхука Chat и предоставить ответ как можно скорее. Возьмём в качестве примера событие вебхука [До отправки сообщения группе](https://intl.cloud.tencent.com/document/product/1047/34374). Перед отправкой сообщения бэкенд Chat отправляет запрос вебхука на бэкенд приложения и определяет, должно ли сообщение быть отправлено на основе результата вебхука. На основе вебхука приложение может синхронизировать сообщение. На следующей диаграмме показан процесс вебхука.

## Классификация вебхуков

Вебхуки можно разделить на четыре типа в соответствии с их функциями:

- Вебхуки статуса онлайна
- Вебхуки цепочки отношений
- Вебхуки сообщений один-на-один
- Вебхуки группы

Вебхуки можно разделить на два типа по процессу:

- Вебхук до возникновения действия: целью этого типа вебхука является возможность для бэкенда приложения вмешаться в логику обработки события. Chat определит последующий поток обработки на основе кода возврата вебхука. Например, вебхук перед отправкой сообщения группе относится к этому типу вебхука.
- Вебхук после возникновения действия: целью этого типа вебхука является возможность для бэкенда приложения реализовать существенную синхронизацию данных. Chat игнорирует коды возврата таких вебхуков. Например, вебхук после того, как участник покидает группу, относится к этому типу вебхука.

## Протокол вебхука

Вебхуки основаны на протоколах HTTP/HTTPS. Бэкенд приложения должен предоставить URL вебхука в Chat, и Chat использует запрос POST для инициирования запроса вебхука на бэкенд приложения. При инициировании запроса вебхука Chat добавляет следующие параметры в конец URL, предоставленного бэкенд приложения:

| Параметр | Описание |
| --- | --- |
| SdkAppid | ID приложения, назначенный Chat |
| CallbackCommand | Команда вебхука |
| contenttype | Необязательно. Значение обычно является строкой JSON. |
| ClientIP | IP адрес клиента |
| OptPlatform | Платформа клиента. В зависимости от типа платформы доступны следующие значения: `RESTAPI` (запросы отправляются с использованием API RESTful) и `Web` (запросы отправляются с использованием Web SDK), `Android`, `iOS`, `Windows`, `macOS`, `iPad` и `Unknown` (запросы отправляются с использованием неизвестного устройства). |

> **Примечание:** В вебхуке `State.StateChange` используется "IOS" (все прописные буквы), а в других вебхуках используется "iOS" (первая буква строчная). Пожалуйста, выполняйте обработку совместимости при использовании.

Конкретное содержимое вебхука включено в пакет HTTP запроса. Подробности см. в примерах вебхуков ниже.

## Примеры вебхуков

Пример запроса вебхука:

```
POST /?SdkAppid=888888&CallbackCommand=Group.CallbackAfterNewMemberJoin&contenttype=json&ClientIP=$ClientIP&OptPlatform=$OptPlatform HTTP/1.1Host: www.example.comContent-Length: 337{    "CallbackCommand": "Group.CallbackAfterNewMemberJoin",     "GroupId": "@TGS#2J4SZEAEL",     "Type": "Public",     "JoinType": "Apply",     "Operator_Account": "leckie",     "NewMemberList": [        {            "Member_Account": "jared"        },         {            "Member_Account": "tommy"        }    ]}
```

## Период ожидания вебхука и повторные попытки

Период ожидания вебхуков Chat на бэкенд приложения составляет две секунды.

До возникновения события вебхуки не повторяются. После возникновения события вебхуки по умолчанию не повторяются, и вы можете настроить, следует ли повторять вебхуки при истечении времени ожидания.

Чтобы обеспечить высокий процент успеха вебхука, приложения третьих сторон должны быстро обрабатывать вебхуки. Например, приложение может отправить ответ вебхука, а затем обработать конкретную бизнес-логику.

## Политика обработки времени ожидания вебхука до возникновения события

Если вебхук истечет до возникновения события, политика по умолчанию — доставить сообщение.

Вы также можете настроить политику обработки времени ожидания вебхука перед возникновением события в консоли. Например, когда происходит истечение времени ожидания вебхука перед отправкой сообщения группе, вы можете указать, следует ли доставить сообщение.

## Соображения безопасности

Chat поддерживает как HTTP, так и HTTPS вебхуки. Для HTTPS вебхуков вам необходимо настроить сертификат, выданный CA на WebServer бэкенда приложения.

Соответствующие проблемы безопасности следующие:

1. HTTP передает данные в открытом виде, и конфиденциальность данных не может быть гарантирована. Поэтому рекомендуется использовать HTTPS.
2. Невозможно определить, действительно ли запрос вебхука исходит от Chat.

Для безопасности источника запроса мы предоставляем решение **Аутентификация вебхука**:

1. Настройте URL вебхука и включите вебхук в консоли.
2. Во время настройки URL вебхука включите аутентификацию и настройте токен аутентификации. Затем подпись (`Sign`) и временная метка подписания (`RequestTime`) будут добавлены к URL запроса вебхука. Алгоритм подписания: **Sign=sha256(TokenRequestTime)**.
3. Бэкенд приложения аутентифицирует запрос вебхука. Он использует SHA256 для расчета и проверки подписи на основе локального токена аутентификации и временной метки подписания (`RequestTime`) в URL вебхука.

```
Пример алгоритма подписания:Token=xxxxyyyyRequestTime=1669872112Sign=sha256(xxxxyyyy1669872112)=17773bc39a671d7b9aa835458704d2a6db81360a5940292b587d6d760d484061Webhook URL=URL&Sign=17773bc39a671d7b9aa835458704d2a6db81360a5940292b587d6d760d484061&RequestTime=1669872112
```

## Распространённые причины сбоев вебхуков

Если возникает сбой вебхука, проверьте, есть ли проблема с настроенным сервисом вебхука в соответствии со следующим контрольным списком:

| Признак сбоя вебхука | Возможная причина |
| --- | --- |
| Доступ к URL вебхука истёк | Chat не может завершить разрешение DNS. В этом случае проверьте, является ли доменное имя действительным в общественной сети. Например, если хост вебхука — http://notexist.com, Chat не может завершить разрешение DNS, потому что это доменное имя не существует. Chat не может получить доступ к IP адресу, настроенному в URL вебхука. В этом случае проверьте, доступен ли этот IP адрес из общественной сети. Например, если хост вебхука — http://10.0.0.1, Chat не может получить доступ к этому IP адресу, потому что доменное имя является приватным IP адресом приложения. Сбой возникает из-за политики брандмауэра сервиса вебхука приложения. В этом случае проверьте конфигурацию брандмауэра. Например, сбой вебхука происходит, если сервер вебхука приложения отклоняет все запросы, поступающие на порт 80. |
| Доступ запрещён сервисом вебхука | Chat может получить доступ к хосту, но соединение не установлено. В этом случае проверьте, был ли WebServer правильно запущен. Например, сбой вебхука произойдёт, когда WebServer сервера вебхука приложения не был запущен или когда конфигурация портов неправильная. |
| Ошибка конфигурации HTTPS сертификата сервиса вебхука | Это может произойти, когда тип вебхука — HTTPS (или взаимная аутентификация HTTPS). Chat может получить доступ к серверу вебхука приложения, но определяет, что сертификат, настроенный на WebServer приложения, является недействительным. В этом случае проверьте, что сертификат HTTPS правильно настроен. |
| Ошибка конфигурации взаимной аутентификации HTTPS сервиса вебхука | Это может произойти, когда тип вебхука — взаимная аутентификация HTTPS. Chat проверяет, что сертификат, настроенный на сервере вебхука приложения, является действительным, но сервер вебхука приложения не может проверить сертификат Chat. **Примечание:** Метод двусторонней аутентификации HTTPS был снят с производства, и мы рекомендуем вам использовать более удобный метод **Аутентификация вебхука**. |
| HTTP код возврата сервиса вебхука не равен 200 | Запрос вебхука успешен, но HTTP код возврата в пакете ответа не равен 200. |
| Пакет ответа вебхука не может быть разобран | Пакет ответа вебхука не находится в формате JSON. |

# Обзор RESTful API

RESTful API является частью интерфейса управления HTTP для серверной части конференции TRTC, предоставляя разработчикам упрощенную точку входа для управления.

В целях безопасности RESTful API доступен только через интерфейс HTTPS.

## Предварительные условия

Для вызова RESTful API необходимо приобрести или получить необходимый план для RoomSdk.

> **Предупреждение:** Новая версия RESTful API прошла значительную оптимизацию и модернизацию архитектуры серверной части, предоставляя расширенные возможности. К сожалению, это означает, что в настоящее время она не может взаимодействовать с версией 1.x SDK. Поэтому при интеграции последнего RESTful API убедитесь, что используется клиентский SDK версии 2.0 или выше.

## Метод вызова

### URL запроса

Формат URL для RESTful API следующий:

```
https://console.tim.qq.com/$ver/$servicename/$command?sdkappid=$SDKAppID&identifier=$identifier&usersig=$usersig&random=99999999&contenttype=json
```

Значения и описание каждого параметра приведены ниже (названия параметров и их значения чувствительны к регистру):

| Параметр | Значение | Получение значения |
| --- | --- | --- |
| https | Протокол запроса | Протокол запроса — HTTPS, метод запроса — POST |
| console.tim.qq.com | Доменное имя запроса | Зафиксировано как `console.tim.qq.com` |
| ver | Номер версии протокола | Зафиксировано как `v4` |
| servicename | Имя внутреннего сервиса, разные имена сервисов соответствуют разным типам сервисов | Пример: `v4/room_engine_http_srv/create_room`, где `room_engine_http_srv` — это `servicename`. Дополнительную информацию см. в [списке интерфейсов REST API](https://www.tencentcloud.com/document/product/647/60702#). |
| command | Слово `command`, в сочетании с `servicename`, используется для обозначения конкретной бизнес-функции | Пример: `v4/room_engine_http_srv/create_room`, где `create_room` — это `command`. Дополнительную информацию см. в [списке интерфейсов REST API](https://www.tencentcloud.com/document/product/647/60702#). |
| sdkappid | Идентификатор приложения, доступный в консоли Chat | Получается при подаче заявки на интеграцию |
| identifier | Имя пользователя, при вызове RESTful API должна быть учетная запись администратора приложения | См. [Администратор приложения](https://www.tencentcloud.com/document/product/1047/33517#app-.E7.AE.A1.E7.90.86.E5.91.98) |
| usersig | Пароль, соответствующий имени пользователя | См. [Генерирование UserSig](https://www.tencentcloud.com/document/product/1047/34385) |
| random | Определяет параметр случайного числа для текущего запроса | 32-битное целое число без знака, диапазон от 0 до 4294967295 |
| contenttype | Формат запроса | Фиксированное значение: `json` |

> **Примечание:** При вызове RESTful API идентификатор сервера приложения должен быть учетной записью администратора приложения. Приложение может генерировать UserSig для учетной записи администратора при каждом вызове RESTful API или генерировать фиксированный UserSig для повторного использования, но обратите особое внимание на период действия UserSig.

### Формат тела HTTP запроса

RESTful API поддерживает только метод POST, и его тело запроса находится в формате JSON. Для конкретных форматов тела см. подробное описание каждого API.
Особенно важно отметить, что тело POST не может быть пусто. Даже если протокол не требует передачи информации, необходимо включить пустой объект JSON, а именно `{}`.

### Код возврата HTTP

Если не происходит сетевая ошибка (например, ошибка 502), результат вызова RESTful API всегда составляет 200. Фактический код ошибки и сообщение об ошибке вызова API возвращаются в теле HTTP ответа.

### Формат тела HTTP ответа

Тело ответа RESTful API также находится в формате JSON и его формат соответствует следующим характеристикам:

```
{    "ActionStatus": "OK",     "ErrorInfo": "",     "ErrorCode": 0,    "RequestId": "Id-70e312f1de024af5a36714b7b71da224-O-Seq-63504"    // Другое содержимое ответа REST API}
```

Тело ответа должно содержать четыре атрибута: ActionStatus, ErrorInfo, ErrorCode, RequestId. Их значения следующие:

| Поле | Тип | Описание |
| --- | --- | --- |
| ActionStatus | String | Результат обработки запроса. OK для успеха, FAIL для ошибки. Если это FAIL, ErrorInfo предоставит причину ошибки. |
| ErrorInfo | String | Причина ошибки |
| ErrorCode | Integer | Код ошибки. 0 для успеха, другие значения для ошибки. Для конкретных причин см. [таблицу кодов ошибок](https://www.tencentcloud.com/document/product/647/57276#). |
| RequestId | String | Код ошибки. 0 для успеха, другие значения для ошибки. Для конкретных причин см. [таблицу кодов ошибок](https://www.tencentcloud.com/document/product/647/57276#). |

## Часто задаваемые вопросы

### Есть ли вероятность истечения времени ожидания запросов RESTful API без получения ответа?

1. Время ожидания RESTful API серверной части Room установлено на 3 с, время ожидания вызывающей стороны должно быть больше 3 с.
2. Используйте `telnet console.tim.qq.com 443` чтобы подтвердить, может ли быть подключен порт сервиса.
3. Используйте `curl -I https://console.tim.qq.com` для простого теста, проверяя, составляет ли код состояния 200.
4. Убедитесь, является ли конфигурация DNS сервера машины внутренним DNS сервером или публичным DNS сервером. Если это внутренний DNS сервер, убедитесь, что сетевой выход DNS сервера и регион ISP IP адреса сетевого выхода машины совпадают.
5. Рекомендуется для бизнес-вызывающих использовать паттерн **Long Connection + Connection Pool** (долгоживущее соединение + пул соединений).

> **Примечание:** Из-за высокого времени на установку коротких HTTPS соединений каждый запрос несет нагрузку на рукопожатие TCP + TLS, поэтому рекомендуется для RESTful API использовать долгоживущие соединения.
> В сценарии использования стандартных библиотек HTTP: для HTTP1.0 необходимо указать заголовок запроса Connection: keep-alive; для HTTP1.1 долгоживущие соединения поддерживаются по умолчанию. В сценарии инкапсуляции HTTPS запросов на основе TCP, TCP соединение может быть переиспользовано для отправки и получения запросов.

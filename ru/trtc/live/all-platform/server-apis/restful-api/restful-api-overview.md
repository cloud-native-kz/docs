# Обзор RESTful API

RESTful API — это интерфейс управления HTTP для Live Streaming Room SDK. Его основная функция — предоставить разработчикам простую точку входа для управления. В целях безопасности RESTful API предлагает только интерфейсы HTTPS.

## Предварительные требования

Для вызова RESTful API необходимо приобрести или получить необходимые пакеты Bundle для LiveSdk.

## Способ вызова

### URL запроса

Формат URL RESTful API следующий:

```
https://console.tim.qq.com/$ver/$servicename/$command?sdkappid=$SDKAppID&identifier=$identifier&usersig=$usersig&random=99999999&contenttype=json
```

Значения и описание каждого параметра приведены ниже (названия параметров и их значения чувствительны к регистру):

| Параметр | Значение | Получение значения |
| --- | --- | --- |
| https | Протокол запроса | Протокол запроса — HTTPS, метод запроса — POST |
| console.tim.qq.com | Домен запроса | Фиксировано как `console.tim.qq.com` |
| ver | Номер версии протокола | Фиксировано как `v4` |
| servicename | Имя внутреннего сервиса, разные servicename соответствуют разным типам сервисов | Пример: `v4/live_engine_http_srv/create_room`, где `room_engine_http_srv` — это `servicename`. Для получения дополнительной информации обратитесь к [списку RESTful API](https://www.tencentcloud.com/document/product/647/63399) |
| command | Слово команды, используется в комбинации с servicename для идентификации конкретной функции бизнеса | Пример: `v4/live_engine_http_srv/create_room`, где `create_room` — это `command`. Для получения дополнительной информации обратитесь к [списку RESTful API](https://www.tencentcloud.com/document/product/647/63399) |
| sdkappid | Идентификатор приложения, полученный из консоли Chat | Получено при подаче заявки на интеграцию |
| identifier | имя пользователя, при вызове RESTful API должна быть учётная запись администратора приложения | См. [Администратор приложения](https://www.tencentcloud.com/document/product/1047/33517) |
| usersig | пароль, соответствующий имени пользователя | См. [Генерирование UserSig](https://www.tencentcloud.com/document/product/1047/34385) |
| random | Идентифицирует параметр случайного числа для текущего запроса | 32-битное целое число без знака, в диапазоне от 0 до 4294967295 |
| contenttype | Формат запроса | Фиксированное значение `json` |

> **Примечание:** При вызове REST API идентификатор сервера приложения должен быть учётной записью администратора приложения. Приложение может генерировать UserSig для учётной записи администратора каждый раз при вызове REST API или может генерировать фиксированный UserSig для повторного использования, но обратите особое внимание на период действия UserSig.

### Формат тела HTTP запроса

REST API поддерживают только метод POST, а тело запроса находится в формате JSON. О конкретном формате тела см. в подробном описании каждого API.
Особенно важно, что тело POST не может быть пустым. Даже если протокол не требует передачи какой-либо информации, должен быть включен пустой объект JSON, а именно `{}`.

### HTTP код возврата

Если только не происходит ошибка сети (например, ошибка 502), результат вызова REST API всегда равен 200. Реальные коды ошибок вызова API и сообщения об ошибках возвращаются в теле HTTP ответа.

### Формат тела HTTP ответа

Тело ответа RESTful API также находится в формате JSON и его формат соответствует следующим характеристикам:

```
{    "ActionStatus": "OK",     "ErrorInfo": "",     "ErrorCode": 0,    "RequestId": "Id-70e312f1de024af5a36714b7b71da224-O-Seq-63504"    // Other response content of REST API}
```

Тело ответа должно содержать свойства ActionStatus, ErrorInfo, ErrorCode, RequestId. Их значения следующие:

| Поле | Тип | Описание |
| --- | --- | --- |
| ActionStatus | String | Результат обработки запроса. OK означает успех, FAIL означает ошибку. Если это FAIL, ErrorInfo предоставит причину ошибки |
| ErrorInfo | String | Ошибка |
| ErrorCode | Integer | Код ошибки. 0 означает успех, остальные означают ошибку. Для получения конкретных причин обратитесь к [таблице кодов ошибок](https://www.tencentcloud.com/document/product/1047/34348) |
| RequestId | String | Код ошибки. 0 означает успех, остальные означают ошибку. Для получения конкретных причин обратитесь к [таблице кодов ошибок](https://www.tencentcloud.com/document/product/1047/34348) |

## Часто задаваемые вопросы

### Может ли быть возможность истечения тайм-аута запросов REST API без получения ответа?

1. Тайм-аут для интерфейса REST Live бэкенда установлен на 3 секунды. Параметр тайм-аута вызывающей стороны должен быть больше 3 секунд.
2. `telnet console.tim.qq.com 443` — чтобы подтвердить, может ли быть подключён порт сервиса.
3. Используйте `curl -I https://console.tim.qq.com` для простого проверки, правда ли код состояния равен 200.
4. Подтвердите, что в конфигурации DNS сервера машины используется внутренний DNS сервер или общедоступный DNS сервер. Если это внутренний DNS сервер, убедитесь, что сетевое исходящее соединение DNS сервера и область ISP исходящего IP адреса сети машины совпадают.
5. Рекомендуется для деловых вызывающих сторон использовать модель "Long Connection + Connection Pool".

> **Примечание:** Из-за высокого потребления времени на установление HTTPS коротких соединений, каждый запрос влечёт за собой накладные расходы на квитирование TCP и TLS. Поэтому рекомендуется получать доступ к REST API с длительными соединениями. В сценариях стандартной библиотеки HTTP: HTTP 1.0 требует указания заголовка запроса Connection: keep-alive, HTTP 1.1 поддерживает длительные соединения по умолчанию. В сценариях, где запросы HTTPS инкапсулированы на основе TCP, вы можете повторно использовать TCP соединения для отправки и получения запросов.

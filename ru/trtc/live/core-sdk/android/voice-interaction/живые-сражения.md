# Живые сражения

`AtomicXCore` предоставляет два основных модуля: `CoHostStore` для **кросс-комнатных соединений** и `BattleStore` для **PK-сражений**. Данное руководство объясняет, как интегрировать эти модули для включения функций соединения и PK в приложение голосового чата.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4b488f1dc9fc11f087ad52540099c741.png)

## Основные концепции

| **Основная концепция** | **Тип** | **Основные обязанности и описание** |
| --- | --- | --- |
| **CoHostStore** | `abstract class` | Управляет жизненным циклом соединений между хостами. Предоставляет API соединения, такие как `requestHostConnection` и `acceptHostConnection`. |
| **BattleStore** | `abstract class` | Обрабатывает сигнализацию PK-сражений и управление состоянием в разных комнатах, включая отслеживание статуса PK. |
| **LiveListStore** | `abstract class` | Получает список активных прямых трансляций (`fetchLiveList`). Используйте это для обнаружения комнат, доступных для соединения. |
| **LiveSeatStore** | `abstract class` | Управляет состояниями мест в комнате, включая кросс-комнатные места во время соединения. Отслеживает события присоединения/отключения пользователя и статус микрофона. |
| **CoHostLayoutTemplate** | `enum` | Определяет шаблоны расположения мест для кросс-комнатных соединений, например `HOST_STATIC_VOICE_6V6`. |
| **BattleConfig** | `data` | Настраивает детали PK-сражения, такие как продолжительность (`duration`) и требуется ли вторичный ответ (`needResponse`). |

## Предварительные условия

Интегрируйте [Быстрый старт](https://www.tencentcloud.com/document/product/647/74681) и убедитесь, что вы реализовали функции **Трансляция хоста** и **Просмотр аудиторией**.

## Хост инициирует соединение

### Шаг 1: Получить список активных трансляций

Чтобы инициировать соединение или PK, хост должен получить текущий список прямых комнат и выбрать целевого хоста.

**Реализация:** Вызовите `LiveListStore.shared().fetchLiveList`.

> **Примечание:** Если целевая комната отсутствует в возвращаемом списке, убедитесь, что isPublicVisible в TUILiveInfo установлена в true при запуске трансляции. Этот API поддерживает пагинацию. Используйте cursor из callback onSuccess для последующих запросов. Пустой cursor означает, что все комнаты получены. Для оптимальной производительности установите pageCount равным 20.

```
import io.trtc.tuikit.atomicxcore.api.live.LiveListStore
import io.trtc.tuikit.atomicxcore.api.CompletionHandler

fun fetchLiveListForCoHost() {
    val cursor = "" // Initial fetch
    val pageCount = 20
    LiveListStore.shared().fetchLiveList(cursor, pageCount, object : CompletionHandler {
        override fun onSuccess() {
            println("Live room list fetched successfully")
            // Update your invitation panel with LiveListStore.liveState.liveList
            // inviteAdapter.submitList(LiveListStore.liveState.liveList)
        }

        override fun onFailure(code: Int, desc: String) {
            println("Failed to fetch live room list: $code")
        }
    })
}
```

### Шаг 2: Отправить приглашение для совместной трансляции

Когда хост выбирает другого хоста, используйте `CoHostStore.requestHostConnection` для отправки приглашения соединения.

> **Примечание:** AtomicXCore поддерживает до **9** взаимосвязанных комнат, каждая с до **50** пользователями. Этот интерфейс поддерживает массовые приглашения, которые вы можете использовать непрерывно; чтобы проверить, было ли приглашение в одну прямую комнату успешным, необходимо прослушивать callback `onSuccess`. API поддерживает поля расширения. Они доставляются целевому хосту через `onCoHostRequestReceived` в `CoHostListener`. Используйте это для передачи бизнес-логики, такой как "начать PK немедленно".

```
import io.trtc.tuikit.atomicxcore.api.live.CoHostLayoutTemplate
import io.trtc.tuikit.atomicxcore.api.CompletionHandler

val targetHostLiveID = "target_host_room_id"
val layoutTemplate = CoHostLayoutTemplate.HOST_STATIC_VOICE_6V6
val timeout = 10 // seconds
val extraInfo = "" // e.g., direct PK flag

coHostStore.requestHostConnection(
    targetHostLiveID,
    layoutTemplate,
    timeout,
    extraInfo,
    object : CompletionHandler {
        override fun onSuccess() {
            println("Connection request sent. Awaiting response...")
        }

        override fun onFailure(code: Int, desc: String) {
            println("Failed to send invitation: $desc")
        }
    })
```

### Шаг 3: Целевой трансляционный хост получает приглашение

Целевой хост получает приглашение через `onCoHostRequestReceived` в `CoHostListener`. Отобразите подсказку UI, например "Хост приглашает вас присоединиться к соединению".

```
private val coHostListener = object : CoHostListener() {
    override fun onCoHostRequestReceived(
        inviter: SeatUserInfo, extensionInfo: String
    ){
        // Show invitation dialog
    }
}
```

### Шаг 4: Целевой хост принимает или отклоняет приглашение

В диалоговом окне приглашения целевой хост может принять или отклонить. Вызовите `acceptHostConnection` или `rejectHostConnection` соответственно.

```
private val coHostStore = CoHostStore.create(liveID)

private val coHostListener = object : CoHostListener() {
    override fun onCoHostRequestReceived(
        inviter: SeatUserInfo, extensionInfo: String
    ){
        // Accept connection
        coHostStore.acceptHostConnection(fromHostLiveID, null)
        // Or reject
        // coHostStore.rejectHostConnection(fromHostLiveID, null)
    }
}
```

### Шаг 5: Отрендерить UI-макет совместной трансляции

После установления соединения прослушивайте `coHostStatus` в `CoHostStore.coHostState` и `seatList` в `LiveSeatStore.liveSeatState` для динамического обновления макета мест.

```
import io.trtc.tuikit.atomicxcore.api.live.CoHostStatus
import io.trtc.tuikit.atomicxcore.api.live.LiveSeatStore
import io.trtc.tuikit.atomicxcore.api.live.SeatInfo
import kotlinx.coroutines.flow.collect

private val liveSeatStore = LiveSeatStore.create(liveID)

coHostStore.coHostState.coHostStatus.collect { status ->
    if (status == CoHostStatus.CONNECTED) {
        // Render connection layout (e.g., 6v6)
    }
}

liveSeatStore.liveSeatState.seatList.collect { seatList: List<SeatInfo> ->
    // Update avatars, mute status, etc.
    // Use seatInfo.userInfo.liveID to distinguish local/remote users.
    // Use seatInfo.userInfo.microphoneStatus for mute icon.
}
```

### Шаг 6: Пригласить дополнительных трансляционных хостов во время совместной трансляции

Вы можете пригласить больше хостов даже после установления соединения, вызвав `requestHostConnection` снова.

```
import io.trtc.tuikit.atomicxcore.api.live.CoHostLayoutTemplate
import io.trtc.tuikit.atomicxcore.api.CompletionHandler

val targetHostLiveID = "target_host_room_id"
val layoutTemplate = CoHostLayoutTemplate.HOST_STATIC_VOICE_6V6
val timeout = 10
val extraInfo = ""

coHostStore.requestHostConnection(
    targetHostLiveID,
    layoutTemplate,
    timeout,
    extraInfo,
    object : CompletionHandler {
        override fun onSuccess() {
            println("Connection request sent. Awaiting response...")
        }

        override fun onFailure(code: Int, desc: String) {
            println("Failed to send invitation: $desc")
        }
    })
```

### Шаг 7: Выход из совместной трансляции

Чтобы отключиться, вызовите `CoHostStore.exitHostConnection`. Остальные хосты должны соответственно обновить макет своих мест.

```
public void onExitButtonClicked() {
    val coHostStore = CoHostStore.create(liveInfo.liveID)
    coHostStore.exitHostConnection(object : CompletionHandler{
        override fun onSuccess(){
            // Handle successful exit
        }

        override fun onFailure(code: Int, desc: String){
            // Handle error
        }
    })
}
```

## Трансляционный хост инициирует PK

Трансляционные хосты могут начать PK-сражения, используя два основных потока, в зависимости от состояния комнаты перед началом PK:

| **Режим PK** | **Состояние комнаты (перед PK)** | **Состояние комнаты (после PK)** |
| --- | --- | --- |
| Сценарий 1: PK после соединения | Обе стороны подключены | Возвращается в состояние соединения |
| Сценарий 2: Соединение с режимом PK | Обе стороны не подключены | Возвращается в состояние соединения |

### Сценарий 1: PK после соединения

AtomicXCore позволяет хостам запустить ограниченное по времени PK-сражение после подключения. Типичный UI-поток:

1. Отправить приглашение соединения; приглашенный хост принимает.
2. Хосты нажимают на PK, чтобы начать ограниченное по времени сражение. Победитель определяется полученными подарками.
3. Целевой хост получает подсказку PK и может принять.
4. UI входит в режим PK и отображает прогресс-бар (на основе подарков, лайков и т.д.).
5. Когда PK заканчивается, отображаются победитель/проигравший и начинается штрафной период.
6. После PK состояние возвращается в режим соединения.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4b475a6ec9fc11f087ad52540099c741.png)

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4ad26a5ec9fc11f0aedb525400454e06.png)

#### Шаг 1: Инициировать приглашение PK

Чтобы начать PK-сражение между подключенными хостами, используйте `BattleStore.requestBattle`. Настройте продолжительность PK, требуется ли ответ, и любую информацию расширения.

> **Примечание:** Продолжительность PK по умолчанию составляет 5 минут, если не установлена в `BattleConfig`. Чтобы начать PK немедленно, установите `needResponse` на `false`. Если `needResponse` равно `false`, установите `timeout` на 0. Если целевой хост не ответит в течение timeout, оба хоста получат `onBattleRequestTimeout`. PK можно повторить.

```
val targetUserID = "target_host_user_id"
val config = BattleConfig(
    duration = 30, // seconds
    needResponse = true,
    extensionInfo = "")

battleStore.requestBattle(
    config,
    listOf(targetUserID),
    timeout = 10, // seconds
    completion = null)
```

#### Шаг 2: Целевой трансляционный хост получает приглашение PK

Целевой хост получает запрос PK через `onBattleRequestReceived` в `BattleListener`. Используйте `needResponse` в `battleInfo.config`, чтобы определить, нужно ли выдавать подсказку пользователю.

- Если `needResponse` равно false, войдите прямо в состояние PK.
- Если true, покажите диалог и войдите в PK при принятии.

```
private val mBattleListener: BattleListener = object : BattleListener() {
    override fun onBattleRequestReceived(
        battleId: String,
        inviter: SeatUserInfo,
        invitee: SeatUserInfo,
    ) {
        super.onBattleRequestReceived(battleInfo, inviter, invitee)
        val userId = inviter.userId
        val userName = inviter.userName
        val needResponse = battleInfo.config.needResponse
        // Show PK prompt if needed
    }
}
```

#### Шаг 3: Войти в состояние PK

Когда `onBattleStarted` срабатывает в `BattleListener`, отрендерите прогресс-бар PK в вашем UI.

- Если `needResponse` равно `false`, PK запускается автоматически.
- Если `needResponse` равно `true`, PK запускается после того, как все приглашенные хосты вызовут `acceptBattle`.
- Хосты, присоединяющиеся к PK, получают `onUserJoinBattle`.
- Все участники PK получают `onBattleStarted`.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4b859b66c9fc11f09e745254007c27c5.png)

#### Шаг 4: Завершить состояние PK

PK заканчивается либо когда истекает продолжительность, либо когда все хосты выходят.

1. **PK-время истекает:** Все хосты получают `onBattleEnded`.
2. **Ручной выход:** Любой хост может вызвать `exitBattle`. Остальные хосты остаются в PK и получают `onUserExitBattle`. Если все хосты выходят, PK заканчивается рано.

```
val battleStore: BattleStore = BattleStore.create(liveID)
battleID = battleStore.battleState.currentBattleInfo.value?.battleId

battleStore.exitBattle(battleID, null)
```

После завершения PK используйте `onBattleEnded` для отображения результатов и определения победителя/проигравшего.

> **Примечание:** После завершения PK сеанс возвращается в состояние кросс-комнатной совместной трансляции. Чтобы завершить совместную трансляцию после PK, вызовите `exitHostConnection`.

### Сценарий 2: Совместная трансляция с режимом PK

**AtomicXCore** поддерживает прямое инициирование PK во время прямой сессии, без предварительной совместной трансляции. После завершения PK сеанс возвращается в состояние прямой комнаты.

1. Трансляционный хост нажимает `PK`, открывает панель с активными трансляционными хостами (см. Шаг 1 выше).
2. Выбирает трансляционного хоста и отправляет приглашение PK.
3. Целевой трансляционный хост получает приглашение и может принять или отклонить.
4. При принятии PK запускается автоматически. Очки накапливаются через подарки или лайки.
5. После завершения PK победитель определяется по очкам.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4b47decec9fc11f0a4a55254001c06ec.png)

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/4ad25df0c9fc11f087ad52540099c741.png)

> **Примечание:** Этот рабочий процесс требует как CoHostStore, так и BattleStore, с `needResponse` в BattleConfig установленным на `false`.

#### Шаг 1: Отправить запрос совместной трансляции

Этот шаг аналогичен Шагам 1 и 2 в "Хост инициирует соединение". Используйте поле `extensionInfo` в `requestHostConnection` для указания режима PK для различия UI.

- Если `extraInfo` равно `{"withPK:true"}`, покажите "Хост приглашает вас на PK".
- Если `extraInfo` равно `{"withPK:false"}`, покажите "Хост приглашает вас присоединиться к кросс-комнатному соединению".

```
import io.trtc.tuikit.atomicxcore.api.live.CoHostLayoutTemplate
import io.trtc.tuikit.atomicxcore.api.CompletionHandler

val targetHostLiveID = "target_host_room_id"
val layoutTemplate = CoHostLayoutTemplate.HOST_STATIC_VOICE_6V6
val timeout = 10
val extraInfo = "withPK:true"

coHostStore.requestHostConnection(
    targetHostLiveID,
    layoutTemplate,
    timeout,
    extraInfo,
    object : CompletionHandler {
        override fun onSuccess() {
            println("Connection request sent. Awaiting response...")
        }

        override fun onFailure(code: Int, desc: String) {
            println("Failed to send invitation: $desc")
        }
    })
```

#### Шаг 2: Целевой трансляционный хост получает запрос и подтверждает

Целевой хост получает `onCoHostRequestReceived` и может принять или отклонить.

```
private val coHostListener = object : CoHostListener() {
    override fun onCoHostRequestReceived(
        inviter: SeatUserInfo, extensionInfo: String
    ){
        // Show invitation dialog
    }
}
```

Примите или отклоните, используя:

```
private val coHostStore = CoHostStore.create(liveID)

private val coHostListener = object : CoHostListener() {
    override fun onCoHostRequestReceived(
        inviter: SeatUserInfo, extensionInfo: String
    ){
        coHostStore.acceptHostConnection(fromHostLiveID, null)
        // Or reject
        // coHostStore.rejectHostConnection(fromHostLiveID, null)
    }
}
```

#### Шаг 3: Инициировать PK автоматически

Когда целевой трансляционный хост принимает, немедленно вызовите `BattleStore.requestBattle` с `needResponse: false` для запуска PK.

```
override fun onCoHostRequestAccepted(invitee: SeatUserInfo) {
        coHostStore.acceptHostConnection(invitee.liveID, null)
        val directPKConfig = BattleConfig(
            duration = 30,
            needResponse = false,
            extensionInfo = ""
        )
        battleStore.requestBattle(
            directPKConfig,
            listOf(invitee.userID),
            timeout = 0,
            completion = null
        )
    }
```

## Обновление очков PK и определение победителя/проигравшего

### Обновление очков PK через REST API

В типичных **сценариях PK между живыми хостами** стоимость подарков, полученных хостом, связана с очками PK (например, когда зритель отправляет подарок "Ракета", очки PK хоста увеличиваются на 500 пунктов). Вы можете реализовать обновление очков PK в реальном времени, используя наш REST API.

> **Примечание:** Система очков PK в LiveKit-бэкэнде использует чистое числовое вычисление и накопление. Вы должны рассчитать очки PK в соответствии с вашей собственной бизнес-логикой перед вызовом API обновления. См. следующие примеры расчета очков PK: Тип подарка — правило расчета очков — пример. Базовый подарок — стоимость подарка × 5 — подарок в 10 юаней → 50 пунктов. Промежуточный подарок — стоимость подарка × 8 — подарок в 50 юаней → 400 пунктов. Продвинутый подарок — стоимость подарка × 12 — подарок в 100 юаней → 1200 пунктов. Специальный эффект подарка — фиксированный высокий балл — подарок в 520 юаней → 1314 пунктов

#### Поток вызова REST API

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/1cd5aeb2c9fd11f0a7775254005ef0f7.png)

#### Описание ключевого процесса

1. **Получить статус PK:**
  - **Конфигурация callback:** Настройте [PK Status Callback](https://www.tencentcloud.com/document/product/647/68260), чтобы LiveKit-бэкэнд активно уведомлял вашу систему при начале или завершении PK.
  - **Активный запрос:** Ваш бэкэнд может вызвать API [PK Status Query](https://www.tencentcloud.com/document/product/647/68256) в любое время, чтобы проверить текущий статус PK.
2. **Расчет очков PK:** Ваш бэкэнд рассчитывает прирост очков PK в соответствии с вашими бизнес-правилами.
3. **Обновление очков PK:** Ваш бэкэнд вызывает API [Update PK Score](https://www.tencentcloud.com/document/product/647/68255) для обновления очков PK в LiveKit-бэкэнде.
4. **LiveKit-бэкэнд синхронизирует клиент:** Бэкэнд автоматически синхронизирует обновленные очки PK со всеми клиентами.

#### Вовлеченные REST API-точки

| **API** | **Описание функции** | **Пример запроса** |
| --- | --- | --- |
| Active API - Query PK Status | Проверить, находится ли текущая комната в PK | [Пример](https://www.tencentcloud.com/document/product/647/68256) |
| Active API - Update PK Score | Обновить рассчитанные очки PK | [Пример](https://www.tencentcloud.com/document/product/647/68255) |
| Callback Configuration - PK Start Callback | Получить уведомление в реальном времени при начале PK | [Пример](https://www.tencentcloud.com/document/product/647/68260) |
| Callback Configuration - PK End Callback | Получить уведомление в реальном времени при завершении PK | [Пример](https://www.tencentcloud.com/document/product/647/68261) |

### Определение победителя

В сценариях PK очки обычно основаны на полученных подарках, что способствует конкуренции и участию. Модуль PK в RoomEngine поддерживает обновления очков и трансляции. Свяжите вашу логику подарков с этой системой.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/d50c20c9cc3d11f0afdc52540044a08e.png)

Когда ваша биллинговая система подтверждает транзакцию подарка, вызовите [Modify PK Score](https://www.tencentcloud.com/document/product/647/68255) для обновления очков PK хоста. Все пользователи в PK-комнатах видят обновленный `battleScore` в `BattleState`. Обновляйте свой прогресс-бар PK при каждом изменении `battleScore`.

```
private fun observeBattleScore() {
    val job = lifecycleScope.launch {
        battleStore?.battleState?.battleScore?.collect { battleScore ->
            // Update PK progress bar here
        }
    }
    collectJobs.add(job)
}
```

## Часто задаваемые вопросы

#### Какое максимальное количество комнат и пользователей поддерживается для соединения или PK?

До 9 комнат могут быть подключены одновременно, с 50 пользователями на комнату.

#### Как я реализую штрафной период после проигрыша в PK?

После получения callback `onBattleEnded` определите победителя, используя `battleScore` из `BattleState`. Соединение остается активным после PK. Реализуйте пользовательский 30-секундный штрафной период, затем отключитесь после обратного отсчета.

#### Почему приглашение на соединение не достигло целевого хоста?

- Убедитесь, что **targetHostLiveId** верен и целевая комната активно транслируется.
- Проверьте сетевое соединение; сигнализация приглашения имеет timeout по умолчанию в 30 секунд.

#### Что происходит, если хост отключается или приложение падает во время соединения или PK?

CoHostStore и BattleStore имеют встроенное обнаружение heartbeat и timeout. Если участник выходит неожиданно, другая сторона уведомляется через события, такие как `onCoHostUserLeft` или `onUserExitB

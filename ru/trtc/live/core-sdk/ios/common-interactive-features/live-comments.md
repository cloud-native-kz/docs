# Живые комментарии

Этот документ содержит полное руководство для разработчиков iOS по быстрой интеграции богатой функциональностью и высокопроизводительной системы оверлея живого чата (barrage/danmaku) в приложение потокового вещания с использованием модуля `BarrageStore` из фреймворка **AtomicXCore**.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/17934b0bc6e111f087ad52540099c741.png)

## Основные возможности

`BarrageStore` обеспечивает полное решение barrage для приложения потокового вещания. Основные возможности включают:

- Получение и отображение сообщений barrage в прямой трансляции.
- Отправка текстовых сообщений barrage для взаимодействия со зрителями.
- Отправка пользовательских бизнес-сообщений barrage для поддержки сложных сценариев, таких как подарки и лайки.
- Вставка системных подсказок в локальный список сообщений (например, "Добро пожаловать XX в прямую трансляцию").

## Основные концепции

| **Основная концепция** | **Тип** | **Основная ответственность и описание** |
| --- | --- | --- |
| [Barrage](https://tencent-rtc.github.io/TUIKit_iOS/documentation/atomicxcore/barrage) | struct | Представляет модель данных для одного сообщения barrage. Содержит всю необходимую информацию, такую как отправитель (sender), содержание сообщения (textContent или data), тип сообщения (messageType) и прочее. |
| [BarrageState](https://tencent-rtc.github.io/TUIKit_iOS/documentation/atomicxcore/barragestate) | struct | Представляет текущее состояние модуля barrage. Основное свойство messageList — это массив [Barrage], содержащий все сообщения barrage в хронологическом порядке для текущей прямой трансляции. Служит источником данных для отрисовки пользовательского интерфейса. |
| [BarrageStore](https://tencent-rtc.github.io/TUIKit_iOS/documentation/atomicxcore/barragestore) | class | Основной класс управления функциональностью barrage. Используйте его для отправки сообщений (sendTextMessage, sendCustomMessage) и подпишитесь на его свойство state, чтобы получать все обновления сообщений barrage. |

### Шаг 2: Инициализация и прослушивание Barrage

Получите экземпляр `BarrageStore`, связанный с `liveId` текущей прямой трансляции, и установите подписчика для получения последнего полного списка сообщений barrage в реальном времени.

```
import Foundation
import AtomicXCore // Импортируйте основную библиотеку
import Combine     // Для реактивного программирования

class BarrageManager {
    private let liveId: String
    private let barrageStore: BarrageStore
    private var cancellables = Set<AnyCancellable>()
    
    // Издатель полного списка сообщений для подписки слоя пользовательского интерфейса
    let messagesPublisher = PassthroughSubject<[Barrage], Never>()
    
    init(liveId: String) {
        self.liveId = liveId
        // 1. Получить одиночный экземпляр BarrageStore для заданного liveId
        self.barrageStore = BarrageStore.create(liveID: liveId)
        // 2. Начать прослушивание сообщений barrage сразу после инициализации
        subscribeToBarrageUpdates()
    }
    
    private func subscribeToBarrageUpdates() {
        barrageStore.state
            .subscribe() // Подпишитесь на все изменения BarrageState
            .receive(on: DispatchQueue.main) // Убедитесь в основном потоке для безопасности пользовательского интерфейса
            .sink { [weak self] barrageState in
                // 3. Передайте обновленный список сообщений на уровень пользовательского интерфейса
                // Примечание: это обеспечивает полный список, включая все исторические сообщения
                self?.messagesPublisher.send(barrageState.messageList)
            }
            .store(in: &cancellables) // Управляйте жизненным циклом подписки
    }
}
```

### Шаг 3: Отправка текстовой строки Barrage

Используйте метод `sendTextMessage` для трансляции простого текстового сообщения всем пользователям в прямой трансляции.

```
extension BarrageManager {
    /// Отправить текстовое сообщение barrage
    func sendTextMessage(text: String) {
        // Предотвратить отправку пустых сообщений
        guard !text.isEmpty else {
            print("Содержимое barrage не может быть пустым")
            return
        }
        
        // Отправить сообщение, используя основной API
        barrageStore.sendTextMessage(text: text, extensionInfo: nil) { result in
            // Обработайте результат в обратном вызове, например, показать подсказку пользователю
            switch result {
            case .success:
                print("Текстовая строка barrage '\\(text)' успешно отправлена")
            case .failure(let error):
                print("Ошибка отправки текстовой строки barrage: \\(error.localizedDescription)")
            }
        }
    }
}
```

#### Параметры API

| **Параметр** | **Тип** | **Описание** |
| --- | --- | --- |
| `text` | `String` | Текстовое содержимое для отправки. |
| `extensionInfo` | `[String: String]?` | Дополнительная информация расширения для бизнес-кастомизации. |
| `completion` | `CompletionClosure?` | Обратный вызов после завершения отправки, содержащий результат успеха или отказа. |

### Шаг 4: Отправка пользовательской строки Barrage

Отправить сообщение, содержащее пользовательскую бизнес-логику, такую как подарки, лайки или команды геймификации. Формат содержимого этого сообщения определяется бизнес-слоем. Получатель должен анализировать и обрабатывать его на основе `businessId` и `data`.

```
extension BarrageManager {
    /// Отправить пользовательскую строку barrage, например, для отправки подарков
    func sendGiftMessage(giftId: String, giftCount: Int) {
        // 1. Определить узнаваемый бизнесом идентификатор
        let businessId = "live_gift"
        
        // 2. Кодировать бизнес-данные как строку JSON
        let giftData: [String: Any] = ["gift_id": giftId, "gift_count": giftCount]
        guard let jsonData = try? JSONSerialization.data(withJSONObject: giftData),
              let jsonString = String(data: jsonData, encoding: .utf8) else {
            print("Ошибка кодирования данных подарка как JSON")
            return
        }
        
        // 3. Отправить пользовательское сообщение, используя основной API
        barrageStore.sendCustomMessage(businessID: businessId, data: jsonString) { result in
            switch result {
            case .success:
                print("Сообщение о подарке (пользовательская строка barrage) успешно отправлено")
            case .failure(let error):
                print("Ошибка отправки сообщения о подарке: \\(error.localizedDescription)")
            }
        }
    }
}
```

#### Параметры API

| **Параметр** | **Тип** | **Описание** |
| --- | --- | --- |
| `businessId` | `String` | Уникальный идентификатор бизнеса, например, "live_gift", используется получателем для различения разных пользовательских сообщений. |
| `data` | `String` | Бизнес-данные, обычно строка в формате JSON. |
| `completion` | `CompletionClosure?` | Обратный вызов после завершения отправки. |

### Шаг 5: Вставка локального сообщения подсказки

Вставить локальное сообщение в список сообщений текущего пользователя. Это сообщение не отправляется другим пользователям в прямой трансляции. Используйте это для системных приветствий, предупреждений или советов по действиям.

```
extension BarrageManager {
    /// Вставить приветственную подсказку в локальный список сообщений
    func showWelcomeMessage(for user: LiveUserInfo) {
        // 1. Создать сообщение Barrage
        var welcomeTip = Barrage()
        welcomeTip.messageType = .text // Повторно использовать текстовый тип для отображения
        welcomeTip.textContent = "Добро пожаловать \\(user.userName) в прямую трансляцию!"
        // отправитель может быть пустым или установлен на идентификатор системного пользователя
        
        // 2. Вставить сообщение в локальный список
        barrageStore.appendLocalTip(message: welcomeTip)
    }
}
```

#### Параметры API

| **Параметр** | **Тип** | **Описание** |
| --- | --- | --- |
| message | Barrage | Объект сообщения для локальной вставки. SDK добавляет это сообщение в messageList в BarrageState. |

### Шаг 6: Управление правами говорения пользователей (отключение звука и включение звука)

Как хост или администратор, вы можете управлять правами чата пользователей в прямой трансляции, чтобы поддерживать здоровую среду сообщества.

#### Отключение звука/включение звука отдельного пользователя

Используйте метод `disableSendMessage` в `LiveAudienceStore` для отключения или включения звука пользователя. Это состояние сохраняется даже если пользователь повторно присоединится к прямой трансляции.

```
import AtomicXCore

// 1. Получить экземпляр LiveAudienceStore для текущей прямой трансляции
let audienceStore = LiveAudienceStore.create(liveID: "your_live_id")

// 2. Укажите ID пользователя и статус отключения звука
let userIdToMute = "user_id_to_be_muted"
let shouldDisable = true // true для отключения звука, false для включения

// 3. Вызвать API для выполнения операции
audienceStore.disableSendMessage(userID: userIdToMute, isDisable: shouldDisable) { [weak self] result in
    guard let self = self else { return }
    switch result {
    case .success:
        print("\\(shouldDisable ? "Отключен звук" : "Включен звук") пользователя \\(userIdToMute) успешно")
    case .failure(let error):
        print("Операция не удалась: \\(error.localizedDescription)")
    }
}
```

#### Включение/отключение глобального отключения звука

Для отключения звука всех пользователей в прямой трансляции (обычно исключая хоста) обновите информацию прямой трансляции через `LiveListStore`.

```
import AtomicXCore

// 1. Получить одиночный экземпляр LiveListStore
let liveListStore = LiveListStore.shared

// 2. Получить информацию о текущей прямой трансляции и изменить статус глобального отключения звука
var currentLiveInfo = liveListStore.state.value.currentLive
currentLiveInfo.isMessageDisable = true // true для включения глобального отключения звука, false для отключения

// 3. Вызвать API обновления и указать флаг изменения
liveListStore.updateLiveInfo(currentLiveInfo, modifyFlag: .isMessageDisable) { result in
    switch result {
    case .success:
        print("Статус глобального отключения звука успешно обновлен")
    case .failure(let error):
        print("Операция не удалась: \\(error.localizedDescription)")
    }
}
```

## Дополнительные возможности: оптимизация производительности в сценариях высокой нагрузки

После внедрения функций barrage с `BarrageStore` используйте следующие стратегии для обеспечения плавного и стабильного пользовательского опыта в сценариях потокового вещания с высокой нагрузкой. Этот раздел содержит стратегии оптимизации и примеры кода для трех основных бизнес-сценариев.

### Сценарий 1: Обработка "штормов barrage" в популярных прямых трансляциях

#### Описание сценария

Во время популярных событий большое количество зрителей может хлынуть в прямую трансляцию с обновлением barrages при десятках сообщений в секунду.

#### Техническая проблема

SDK возвращает полный список barrage с высокой частотой. Если вы вызываете `tableView.reloadData()` при каждом обновлении, основной поток будет заблокирован интенсивной раскладкой и отрисовкой пользовательского интерфейса, вызывая снижение производительности пользовательского интерфейса.

#### Оптимизация: пакетная обработка и debouncing

Не реагируйте на каждое обновление данных. Вместо этого установите пороговое значение времени (например, 300 миллисекунд). Только обновляйте пользовательский интерфейс, если этот порог прошел с момента последнего обновления. Это сокращает десятки вызовов `reloadData()` в секунду всего до 3-4, значительно улучшая плавность.

#### Пример кода

Создайте класс `BarrageUIManager` с внутренним буфером и таймером для пакетного обновления данных `UITableView`.

```
class BarrageUIManager {
    private var latestMessageList: [BarrageViewModel]?
    private var refreshTimer: Timer?
    private weak var tableView: UITableView?
    private var dataSource: [BarrageViewModel] = []
    
    init(tableView: UITableView) {
        self.tableView = tableView
        // Проверяйте каждые 0,3 секунды, нужно ли обновление
        self.refreshTimer = Timer.scheduledTimer(withTimeInterval: 0.3, repeats: true) { [weak self] _ in
            self?.refreshUIIfNeeded()
        }
    }
    
    /// Часто вызывается извне, передавая последний полный список
    func update(with fullList: [BarrageViewModel]) {
        self.latestMessageList = fullList
    }
    
    private func refreshUIIfNeeded() {
        // Проверить, есть ли новые данные для обновления
        guard let newList = self.latestMessageList, let tableView = self.tableView else { return }
        self.latestMessageList = nil // Очистить флаг, чтобы избежать повторных обновлений
        
        // Обновить источник данных и обновить пользовательский интерфейс
        self.dataSource = newList
        tableView.reloadData()
    }
    
    deinit {
        refreshTimer?.invalidate()
    }
}
```

### Сценарий 2: Обеспечение стабильности памяти для длительных прямых трансляций

#### Описание сценария

Ваше приложение может потребоваться поддерживать часовое или даже круглосуточное непрерывное потоковое вещание, такое как игровое потоковое вещание или медленные прямые трансляции. Приложение должно оставаться стабильным и избежать сбоев из-за длительной работы.

#### Техническая проблема

SDK возвращает полный `messageList`, который растет неограниченно во время длительных прямых трансляций. Даже если уровень пользовательского интерфейса ограничивает обновления, массив большого размера уровня данных будет продолжать потреблять память, в конечном итоге вызывая сбой приложения.

#### Оптимизация: циклический буфер с фиксированной емкостью

Убедитесь, что источник данных содержит только ограниченное количество сообщений. Независимо от размера полного списка, возвращаемого SDK, сохраняйте только последнюю часть для отображения.

#### Пример кода

После получения полного списка от SDK сохраните только последние 500 сообщений (или другое определенное число) для обновления пользовательского интерфейса.

```
class BarrageUIManager {
    private let capacity: Int = 500 // Сохраняйте только последние 500 сообщений
    
    // ... (другой код, как выше) ...
    
    private func refreshUIIfNeeded() {
        guard let fullList = self.latestMessageList, let tableView = self.tableView else { return }
        self.latestMessageList = nil 
        
        // Сохраняйте только последние N сообщений
        let cappedList = Array(fullList.suffix(self.capacity))
        self.dataSource = cappedList
        tableView.reloadData()
    }
}
```

### Сценарий 3: Отрисовка сложных стилей Barrage с уровнями пользователей и значками

#### Описание сценария

Для повышения атмосферы прямого вещания и выделения платящих пользователей сообщения barrage могут включать богатые визуальные элементы, такие как имена пользователей, значки уровня пользователя, значки фанатов и содержимое сообщения.

#### Техническая проблема

Отрисовка представлений с несколькими изображениями и текстами, пользовательскими шрифтами или сложными раскладками требует больше времени, чем отрисовка простого текста. Высокочастотная отрисовка этих сложных представлений в списке увеличивает рабочую нагрузку основного потока, вызывая отставание при прокрутке списка.

#### Оптимизация: асинхронная отрисовка

Переместите процесс отрисовки представлений с основного потока на фоновый поток. Основной поток должен обрабатывать только окончательное отображение уже отрисованного растрового изображения, значительно сокращая его вычислительную нагрузку.

#### Пример кода

В пользовательской `UITableViewCell` включите свойство `drawsAsynchronously` слоя, чтобы позволить системе выполнять задачи рисования на фоновом потоке.

```
import UIKit

// В пользовательской ячейке barrage
class BarrageCell: UITableViewCell {
    // ... (объявления UILabel, UIImageView и других подпредставлений)
    
    override init(style: UITableViewCell.CellStyle, reuseIdentifier: String?) {
        super.init(style: style, reuseIdentifier: reuseIdentifier)
        // Включить асинхронную отрисовку
        self.layer.drawsAsynchronously = true
    }
    
    required init?(coder: NSCoder) {
        fatalError("init(coder:) has not been implemented")
    }
    
    func configure(with viewModel: BarrageViewModel) {
        // Установить текст метки, изображения и т.д.
        // С включенной асинхронной отрисовкой отрисовка происходит на фоновом потоке при возможности
    }
    
    // Продвинутый вариант: полностью ручная отрисовка
    override func draw(_ rect: CGRect) {
        // Для более точного контроля вручную рисуйте текст и изображения здесь, используя Core Graphics
        // Объедините генерирование UIImage на фоновом потоке, а затем нарисуйте его в методе draw основного потока для полностью асинхронной отрисовки
    }
}
```

## Часто задаваемые вопросы

### Помимо базовых текстовых barrage, мы также хотим реализовать более богатые стили, такие как "цветные barrage" и "barrage с подарками". Как мы можем это достичь?

Вы можете реализовать эти функции, используя пользовательские сообщения с `sendCustomMessage`.

**Шаги реализации**

1. **Определить структуру данных**: Сотрудничайте с вашими командами клиента и сервера для определения структуры JSON для пользовательских сообщений. Например, цветная barrage может быть определена как:

```
{
  "type": "colored_text",
  "text": "Это цветной пулеметный комментарий!",
  "color": "#FF5733"
}
```

2. **Отправитель**: При отправке сериализуйте эту структуру JSON в строку и отправьте через параметр `data` метода `sendCustomMessage`. businessID может быть установлен на уникальный идентификатор для вашего использования, такой как `barrage_style_v1`.
3. **Получатель**: При получении сообщения barrage проверьте, является ли его messageType `BarrageType.CUSTOM` и совпадает ли `businessID`. Если да, разберите строку данных (обычно как JSON) и отрисуйте пользовательский стиль пользовательского интерфейса на основе разобранных данных (таких как `color`, `text`).

### Если я вызову BarrageStore.create(liveID = "some_id") в разных классах или файлах, это создаст несколько экземпляров и вызовет путаницу?

Нет. Внутренний механизм **AtomicXCore** гарантирует, что при условии передачи одного и того же liveID вы всегда получаете один и тот же экземпляр `BarrageStore` для этой прямой трансляции. Вам не нужно вручную управлять одиночками.

### Почему я не вижу отправленное мной сообщение в списке сообщений после вызова sendTextMessage?

**Шаги устранения неполадок:**

1. **Проверить обратный вызов завершения**: Метод `sendTextMessage` предоставляет обратный вызов завершения. Проверьте, указывает ли обратный вызов на успех или отказ. Если он неудачен, сообщение об ошибке будет указывать проблему (например, "Вы отключены", "Ошибка сети" и т.д.).
2. **Подтвердить время подписки**: Убедитесь, что вы подписались на `barrageStore.barrageState.messageList` после того, как сеанс прямого вещания для соответствующего liveID начался. Если вы начнете прослушивание перед присоединением к прямой трансляции, вы можете пропустить некоторые сообщения.
3. **Проверить liveID**: Убедитесь, что liveID, используемый при создании экземпляра BarrageStore, присоединении к прямой трансляции и отправке сообщений, полностью совпадает, включая чувствительность к регистру.
4. **Проблемы с сетью**: Убедитесь, что сетевое соединение устройства нормальное. Отправка сообщений зависит от сетевого соединения.

### Как новые зрители могут видеть исторические сообщения barrage, отправленные до того, как они присоединились к прямой трансляции?

`AtomicXCore` поддерживает получение исторических сообщений чата, но вам нужно включить эту функцию в **консоли сервера**. После конфигурации SDK автоматически обрабатывает все — дополнительный код клиента не требуется.

#### **Шаг 1: Конфигурация в консоли Live**

1. Войдите в [**Console > Live > Configuration**](https://console.trtc.io/live/configuration) и выберите целевое приложение в верхней части страницы.
2. На странице Live Configuration выберите **View Past Messages** и установите Previous Messages Viewable, чтобы указать количество сообщений (максимум 50), которые могут видеть новые зрители.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/5ca7e8c2e20c11f095b0525400e889b2.png)

#### **Шаг 2: Автоматическое получение на стороне клиента**

После завершения приведенной выше конфигурации **не требуются изменения** в коде клиента.

Когда новый пользователь присоединяется к прямой трансляции, `AtomicXCore` автоматически получает настроенное количество исторических сообщений чата в фоновом режиме. Эти сообщения доставляются на уровень пользовательского интерфейса через канал подписки `BarrageState` так же, как сообщения в реальном времени. Ваше приложение будет получать и отображать эти исторические сообщения чата так же, как сообщения прямого вещания.

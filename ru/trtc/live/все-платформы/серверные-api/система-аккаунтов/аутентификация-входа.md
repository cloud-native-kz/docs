# Аутентификация входа

## Обзор аутентификации входа

Live и Instant Messaging (Chat) используют одну и ту же систему аутентификации входа. Instant Messaging (Chat) разработан на основе системы мгновенного обмена сообщениями QQ. Мы извлекли общие модули QQ и интегрировали их для создания удобного в интеграции Chat SDK и серверной службы.

Chat SDK можно рассматривать как QQ без пользовательского интерфейса. Интеграция Chat SDK в приложение похожа на интеграцию ядра QQ.

Перед использованием Chat для обмена сообщениями необходимо войти. Если вы входите в QQ с помощью ID и пароля QQ, то для входа в Chat SDK необходимо использовать определённое имя пользователя (`UserID`) и пароль (`UserSig`).

- **UserID**: Ранее известен как Identifier. Это имя пользователя для входа в Chat. По сути, это ID пользователя в вашем приложении.
- Например, если пользователь в вашем приложении имеет ID 27149, вы можете использовать 27149 в качестве UserID для входа в Chat.
- **UserSig**: Это пароль, с помощью которого пользователь входит в Chat. Это данные, полученные после того, как App Server использует ключ для шифрования информации, такой как UserID. Подробнее см. [Генерирование UserSig](https://www.tencentcloud.com/document/product/647/69883#).

> **Внимание**: Серверная часть Chat полностью доверяет UserSig. Чтобы избежать влияния на ваши данные и бизнес, вы должны обеспечить безопасность приватного ключа. UserSig, созданные API Chat SDK по умолчанию, действительны в течение 180 дней. Разработчики могут использовать API с параметром validity для настройки периода действия. Разработчики должны получить новый UserSig от сервера разработчика до истечения срока действия исходного UserSig. Подробнее о Chat SDK для генерирования UserSig см. [Генерирование UserSig](https://www.tencentcloud.com/document/product/647/69883#).

## Администратор приложения

Некоторые службы Chat требуют прав администратора, например вызов RESTful API, расформирование группы и отправка сообщений всем членам группы. По сравнению с обычными учётными записями роль администратора приложения имеет наивысший уровень привилегий:

- Он имеет более высокие права доступа на чтение. Например, он может получить все группы в приложении и любую информацию о любой группе.
- Он имеет более высокие права на выполнение операций. Например, он может отправлять сообщения любому пользователю и добавлять или удалять членов любой группы.

Администраторов приложения можно задать только в консоли. Подробнее о процедуре см. [Настройка администраторов учётных записей](https://www.tencentcloud.com/document/product/647/69881#).

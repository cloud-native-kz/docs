# Часто задаваемые вопросы

## Проблемы, связанные с ушным мониторингом

### В сценариях караоке часто используется мониторинг в наушниках. Как включить функцию мониторинга звука в наушниках?

```
[[trtcCloud getAudioEffectManager] enableVoiceEarMonitor:YES];
```

### Что делать, если после включения функции мониторинга звука в наушниках эффект отсутствует?

Из-за высокой аппаратной задержки беспроводных наушников попробуйте предложить ведущему использовать проводные наушники на уровне пользовательского интерфейса. При этом учитывайте, что не все телефоны могут обеспечить отличный эффект мониторинга звука после включения этой функции. TRTC SDK уже заблокировал этот эффект на некоторых телефонах с плохой производительностью мониторинга звука.

### Является ли задержка мониторинга звука слишком высокой?

Проверьте, используете ли вы беспроводные наушники. Из-за высокой аппаратной задержки беспроводных наушников по возможности используйте проводные наушники.

Кроме того, вы можете попытаться улучшить высокую задержку мониторинга звука, включив аппаратный мониторинг звука через экспериментальный интерфейс setSystemAudioKitEnabled. В настоящее время для устройств Huawei и Vivo SDK использует аппаратный мониторинг звука по умолчанию, а для других устройств используется программный мониторинг звука по умолчанию.

```
// Включить аппаратный мониторинг звукаNSDictionary *jsonDic = @{                            @"api": @"setSystemAudioKitEnabled",                            @"params": @{@"enable": @(1)}                         };NSData *jsonData = [NSJSONSerialization dataWithJSONObject:jsonDic options:NSJSONWritingPrettyPrinted error:nil];NSString *jsonString = [[NSString alloc] initWithData:jsonData encoding:NSUTF8StringEncoding];[trtcCloud callExperimentalAPI:jsonString];
```

## Проблемы синхронизации времени NTP

### Напоминание: **«Синхронизация времени NTP завершена, но результат может быть неточным»**

Синхронизация времени NTP успешна, но отклонение может быть более 30 мс, что отражает плохую сетевую среду клиента и постоянный скачок времени туда-сюда (rtt jitter).

### Напоминание: **«Ошибка в AddressResolver: адрес не связан с именем хоста»**

Сбой синхронизации времени NTP может быть вызван временными аномалиями в разрешении DNS локального оператора связи в текущей сетевой среде. Пожалуйста, попробуйте позже.

## Рекомендации по тестированию скорости сети

Сценарии онлайн-караоке предъявляют высокие требования к сети для пользователей, особенно в случае пения в реальном времени в дуэте. Для хорошего опыта караоке необходима высокое качество и стабильная сетевая среда. Поэтому рекомендуется перед входом в комнату провести тест скорости сети для пользователя и дать напоминание на уровне пользовательского интерфейса пользователям, которые не соответствуют требованиям к сети, запретив им присоединяться к комнате караоке или участвовать в дуэте.

Инициирование теста скорости сети с помощью TRTC SDK:

```
TRTCSpeedTestParams *speedTestParams = [[TRTCSpeedTestParams alloc] init];speedTestParams.sdkAppId = SDK_APP_ID;speedTestParams.userId = userId;speedTestParams.userSig = userSig;// Если фактическая полоса пропускания выше ожидаемого значения, результат теста — ожидаемое значение;// если фактическая полоса пропускания ниже ожидаемого значения, результат теста — фактическое значение полосы пропускания.speedTestParams.expectedDownBandwidth = 3000; // Ожидаемая нисходящая полоса пропускания, от 10 до 5000 кбитspeedTestParams.expectedUpBandwidth = 3000; // Ожидаемая восходящая полоса пропускания, от 10 до 5000 кбит[trtcCloud startSpeedTest:speedTestParams];
```

> **Примечание:** Ожидаемая восходящая полоса пропускания, от 10 до 5000 кбит. Пожалуйста, проводите тест скорости сети перед входом в комнату. Тестирование скорости сети в комнате повлияет на нормальные эффекты передачи аудио и видео, а из-за слишком большого количества помех результаты теста скорости сети также будут неточными.

Обратный вызов результата теста скорости сети TRTC SDK:

```
- (void)onSpeedTestResult:(TRTCSpeedTestResult *)result {    NSString *tquality = @"Unknown";    switch (result.quality) {        case TRTCQuality_Unknown:            tquality = @"Unknown";            break;        case TRTCQuality_Excellent:            tquality = @"The current network is excellent";            break;        case TRTCQuality_Good:            tquality = @"The current network is good";            break;        case TRTCQuality_Poor:            tquality = @"The current network is poor";            break;        case TRTCQuality_Bad:            tquality = @"The current network is bad";            break;        case TRTCQuality_Vbad:            tquality = @"The current network is very bad";            break;        case TRTCQuality_Down:            tquality = @"The current network does not meet TRTC\\`s minimum requestments";            break;        default:            break;    }    if (result.success) {        [mTextTestResult addObject:@"test successfullï¼\\n"];        [mTextTestResult addObject:[NSString stringWithFormat:@"IP addressï¼%@ \\n", result.ip]];        [mTextTestResult addObject:[NSString stringWithFormat:@"uplink packet loss rateï¼%f \\n", result.upLostRate]];        [mTextTestResult addObject:[NSString stringWithFormat:@"downlink packet loss rateï¼%f \\n", result.downLostRate]];        [mTextTestResult addObject:[NSString stringWithFormat:@"network lantencyï¼%u ms \\n", result.rtt]];        [mTextTestResult addObject:[NSString stringWithFormat:@"downlink bandwidthï¼%ld kbps \\n", (long)result.availableDownBandwidth]];        [mTextTestResult addObject:[NSString stringWithFormat:@"uplink bandwidthï¼%ld kbps \\n", result.availableUpBandwidth]];        [mTextTestResult addObject:[NSString stringWithFormat:@"downlink bandwidthï¼%@ \\n", tquality]];    } else {        [mTextTestResult addObject:@"test successfullï¼\\n"];        [mTextTestResult addObject:[NSString stringWithFormat:@"errMsgï¼%@ \\n", result.errMsg]];    }}
```

## Присоединение к дуэту в процессе пения

Решение для пения в реальном времени в дуэте теоретически не имеет ограничений на количество участников дуэта, поддерживая одновременное участие нескольких человек в дуэте, а также присоединение к дуэту в процессе пения.

Ниже приведены ключевые действия для присоединения к дуэту в процессе пения:

- Синхронизация времени NTP
- Включение экспериментального интерфейса дуэта
- Вход в комнату и начало потокового вещания на микрофон
- Получение сигнала дуэта, получение ресурсов аккомпанемента и согласованного времени дуэта
- Расчет разницы между согласованным временем и текущим временем, предварительная загрузка и поиск аккомпанемента
- Начало участия в дуэте и синхронизация прогресса аккомпанемента и прогресса текста песни в реальном времени

Для получения подробной информации о процессе реализации и реализации кода вышеуказанных ключевых действий, пожалуйста, обратитесь к документации по [синхронизации песни](https://www.tencentcloud.com/zh/document/product/647/57025), [синхронизации текста песни](https://www.tencentcloud.com/zh/document/product/647/57028), [синхронизации вокала](https://www.tencentcloud.com/zh/document/product/647/57032).

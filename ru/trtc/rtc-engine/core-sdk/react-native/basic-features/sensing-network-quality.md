# Определение качества сетевого соединения

Этот документ в основном описывает, как оценить качество текущего сетевого соединения.

## Определение качества сетевого соединения во время звонка

TRTC предоставляет событие обратного вызова **onNetworkQuality**, которое сообщает вам о текущем качестве сетевого соединения каждые две секунды. Его параметры включают две части: localQuality и remoteQuality.

- **localQuality**: Представляет текущее качество вашего сетевого соединения, разделённое на 6 уровней: Excellent, Good, Poor, Bad, VeryBad и Down.
- **remoteQuality**: Представляет качество сетевого соединения удалённых пользователей. Это массив, в котором каждый элемент представляет качество сетевого соединения удалённого пользователя.

| Качество | Имя | Описание |
| --- | --- | --- |
| 0 | Unknown | Не определено |
| 1 | Excellent | Текущее сетевое соединение отличное. |
| 2 | Good | Текущее сетевое соединение хорошее. |
| 3 | Poor | Текущее сетевое соединение среднее. |
| 4 | Bad | Текущее сетевое соединение плохое, могут возникнуть заметные задержки и латентность в вызове. |
| 5 | VeryBad | Текущее сетевое соединение очень плохое. TRTC может лишь едва поддерживать соединение, но не может гарантировать качество связи. |
| 6 | Down | Текущее сетевое соединение не удовлетворяет минимальным требованиям TRTC, и невозможно выполнять обычные видео и аудио вызовы. |

Вам нужно только прослушать callback onNetworkQuality в TRTC и предоставить соответствующие уведомления в интерфейсе:

```
// Create a TRTC instance.const trtcCloud = TRTCCloud.sharedInstance();// Listen to the onNetworkQuality callback and perceive changes in the current network status.const onRtcListener = useCallback((type: TRTCCloudListener, params: any) => {  if (type === TRTCCloudListener.onNetworkQuality) {     console.log("local user", params.localQuality);     params.remoteQuality.forEach((user: any) => {       console.log("remote user", "id=" + user.userId + ",quality=" + user.quality);     });  }}// Register a callbacktrtcCloud.registerListener(onRtcListener);
```

## Определение качества сетевого соединения перед звонком

### Принцип тестирования скорости

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/774df7df257211f0a62e525400454e06.png)

- Принцип тестирования скорости заключается в том, что SDK отправляет пакет зондирующих пакетов на узел сервера, а затем вычисляет качество возвращённых пакетов и уведомляет результаты тестирования скорости через API обратного вызова.
- Результаты тестирования скорости будут использоваться для оптимизации последующей стратегии выбора сервера SDK. Поэтому рекомендуется выполнить тестирование скорости перед первым вызовом пользователя. Это поможет выбрать наиболее подходящий сервер. Кроме того, если результаты теста неудовлетворительны, вы можете предложить пользователю с помощью выделяющегося в интерфейсе элемента выбрать лучшую сеть.
- Результат тестирования скорости ([TRTCSpeedTestResult](https://liteav.sdk.qcloud.com/doc/product/trtc/dart/api/en/trtc_cloud_def/TRTCSpeedTestResult-class.html)) включает следующие поля:

| Поле | Значение | Описание |
| --- | --- | --- |
| success | Успешно ли выполнено | Выполнен ли тест успешно. |
| errMsg | Сообщение об ошибке. | Подробная информация об ошибке тестирования полосы пропускания. |
| ip | IP-адрес сервера | IP-адрес тестирующего сервера |
| quality | Оценка качества сетевого соединения | Качество сетевого соединения, рассчитанное алгоритмом оценки, чем меньше потери и меньше RTT, тем выше оценка. |
| upLostRate | Коэффициент потери пакетов в восходящем направлении | Диапазон [0 - 1.0]. Например, 0.3 означает, что при отправке 10 пакетов на сервер, 3 могут быть потеряны. |
| downLostRate | Коэффициент потери пакетов в нисходящем направлении | Диапазон [0 - 1.0]. Например, 0.2 означает, что при получении 10 пакетов от сервера, 2 могут быть потеряны. |
| rtt | Задержка сетевого соединения | Время в пути туда и обратно (RTT) между SDK и сервером. Чем меньше это значение, тем лучше. Нормальное значение находится между 10 мс и 100 мс. |
| availableUpBandwidth | Полоса пропускания восходящего направления | Предполагаемая полоса пропускания восходящего направления, единица: кбит/с. -1 указывает на недопустимое значение. |
| availableDownBandwidth | Полоса пропускания нисходящего направления | Предполагаемая полоса пропускания нисходящего направления, единица: кбит/с. -1 указывает на недопустимое значение. |

### Как провести тестирование скорости

1. Вы можете запустить функцию тестирования скорости через `startSpeedTest` в TRTCCloud.

```
// Call startSpeedTest to start the speed test.const trtcCloud = TRTCCloud.sharedInstance();const param: TRTCSpeedTestParams = {  expectedDownBandwidth: 10, // Expected downstream bandwidth (kbps, value ranges from 10 to 5000, no test when it is 0)  expectedUpBandwidth: 10,   // Expected uplink bandwidth (kbps, value ranges from 10 to 5000, no test when it is 0).  scene: 1,                  // 1: Delay test. 2: Delay and bandwidth test. 3: Online chorus testing.  sdkAppId: 0,               // Application identifier  userId: "",                // User identity  userSig: "",               // User signature  }trtcCloud.startSpeedTest(param);const onRtcListener = useCallback((type: TRTCCloudListener, params: any) => {  if (type === TRTCCloudListener.onSpeedTestResult) {     console.log('Speed test result:', params);     setSpeedTestResult(params);  }}
```

2. Результат тестирования скорости будет возвращён через функцию обратного вызова onSpeedTestResult.

```
// Return results through the onSpeedTestResult callback functionconst trtcCloud = TRTCCloud.sharedInstance();const onRtcListener = useCallback((type: TRTCCloudListener, params: any) => {  if (type === TRTCCloudListener.onSpeedTestResult) {     console.log('Speed test result:', params);     setSpeedTestResult(params);  }}trtcCloud.registerListener(onRtcListener);
```

| Типы перечисления | Описание |
| --- | --- |
| availableDownBandwidth | Полоса пропускания нисходящего направления (кбит/с, -1: недопустимое значение). |
| availableUpBandwidth | Полоса пропускания восходящего направления (кбит/с, -1: недопустимое значение). |
| downJitter | Дрожание пакетов нисходящего направления (мс), указывающее на стабильность передачи данных в текущей сетевой среде для пользователя. Чем меньше это значение, тем лучше. Нормальный диапазон значений: 0 мс - 100 мс. -1 означает, что тестирование скорости не смогло успешно измерить допустимое значение. Обычно дрожание сети WiFi немного выше, чем в сетях 4G/5G. |
| downLostRate | Коэффициент потери пакетов в нисходящем направлении. Значение находится в диапазоне [0 - 1.0]. Например, 0.2 означает, что при получении 10 пакетов с сервера, 2 могут быть потеряны. |
| errMsg | Информация об ошибке тестирования полосы пропускания. |
| ip | IP-адрес сервера. |
| quality | Качество сетевого соединения, рассчитанное алгоритмом оценки. Подробнее:quality = 0: не определено.quality = 1: Текущее сетевое соединение отличное.quality = 2: Текущее сетевое соединение хорошее.quality = 3: Текущее сетевое соединение среднее.quality = 4: Текущее сетевое соединение плохое.quality = 5: Текущее сетевое соединение очень плохое.quality = 6: Текущее сетевое соединение не удовлетворяет минимальным требованиям TRTC. |
| rtt | Латентность (мс): относится к времени в пути туда и обратно (RTT) от текущего устройства к серверу TRTC. Чем меньше значение, тем лучше. Нормальный диапазон значений: 10 мс - 100 мс. |
| success | Выполнен ли тест успешно. |
| upJitter | Дрожание пакетов восходящего направления (мс): относится к стабильности передачи данных в текущей сетевой среде пользователя. Чем меньше значение, тем лучше. Нормальный диапазон значений: 0 мс - 100 мс. -1 означает, что в этом тестировании скорости не удалось успешно измерить допустимое значение. Обычно дрожание сети WiFi немного выше, чем в сетях 4G/5G. |
| upLostRate | Коэффициент потери пакетов в восходящем направлении. Значение находится в диапазоне [0 - 1.0]. Например, 0.3 означает, что при отправке 10 пакетов на сервер, 3 могут быть потеряны. |

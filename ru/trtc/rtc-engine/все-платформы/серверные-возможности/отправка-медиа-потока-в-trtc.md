# Отправка медиа-потока в TRTC

## Обзор

Совместный просмотр, совместное прослушивание, совместная игра, совместное обучение… Различные виды взаимодействия, которые когда-то требовали личного контакта, теперь переходят в онлайн. Даже находясь на расстоянии тысячи миль друг от друга, друзья могут вместе смотреть фильмы, слушать музыку и общаться. Этот удивительный опыт взаимодействия в реальном времени становится популярным среди молодых людей и в настоящее время является основной функцией и основным направлением развития аудио- и видеопродуктов.

TRTC предлагает два решения для потокового вещания: [отправка онлайн-медиа-потока](https://www.tencentcloud.com/document/product/647/62716#50940aad-d90f-4473-9f46-d5dd46917653) и [RTMP-вещание с TRTC](https://www.tencentcloud.com/document/product/647/62716#3b96cc52-90be-46ee-aec0-218dcf302119), каждое со своим сценарием применения, подробно описанным ниже:

- [Отправка онлайн-медиа-потока](https://www.tencentcloud.com/document/product/647/62716#50940aad-d90f-4473-9f46-d5dd46917653) используется для **извлечения облачных онлайн-медиа-потоков (онлайн-трансляция или облачные файлы видео по запросу)** и отправки их в комнату TRTC.
- [RTMP-вещание с TRTC](https://www.tencentcloud.com/document/product/647/62716#3b96cc52-90be-46ee-aec0-218dcf302119) используется для **отправки локальных медиа-файлов и аудио- и видеоданных с устройств захвата** в комнату TRTC с использованием стандартного протокола RTMP.

> **Примечание:** Связанные сборы следующие: Разблокировка функций: функции **отправка онлайн-медиа-потока** и **RTMP-вещание с TRTC** должны быть разблокированы подпиской на **базовую** или **профессиональную версию** [пакета RTC-Engine на месяц](https://www.tencentcloud.com/document/product/647/56025#). Плата за использование: использование функции потокового вещания предполагает операции транскодирования, которые влекут затраты на транскодирование. Для получения дополнительной информации см. [Описание биллинга для MixTranscoding и Bypass Relay](https://www.tencentcloud.com/document/product/647/47631#). Расходы на аудиодорожку, понесенные ботом для потокового вещания в комнате, учитываются (Примечание: расходы, понесенные ботом в комнате за функции отправки онлайн-медиа-потока, не будут учитываться до 15 августа 2024 г. и начнут учитываться с 16 августа 2024 г.). Аудитория в комнате, подписанная на аудио- и видеопотоки, будет нести расходы на аудио- и видеовызовы. Дополнительные сведения см. в разделе [Описание биллинга для аудио- и видеодорожек](https://www.tencentcloud.com/document/product/647/42734).

### Архитектура функции

1. Пользователи создают задачи отправки онлайн-медиа-потока с помощью REST API. Эти задачи выполняются сервером ретрансляции.
2. Сервер ретрансляции извлекает онлайн-потоки или файлы видео по запросу.
3. Сервер ретрансляции отправляет полученные аудио- и видеоданные в комнату TRTC и автоматически создает виртуального якорного пользователя. Имя пользователя и номер комнаты этого пользователя указываются при создании задачи.
4. Другие клиенты TRTC могут просматривать эти потоки и использовать возможности TRTC, такие как запись и ретрансляция.

## RTMP-вещание с TRTC

TRTC поддерживает отправку **локальных медиа-файлов** и **аудио- и видеоданных с устройств захвата** в комнату TRTC с использованием стандартного протокола RTMP. Для упрощения интеграции TRTC вы можете установить [OBS](https://obsproject.com/download), FFmpeg или другие библиотеки RTMP для реализации потокового вещания с TRTC. OBS — это инструмент с открытым исходным кодом, разработанный третьей стороной для прямого вещания. Он прост в использовании, бесплатен и поддерживает OS X, Windows и Linux. OBS может использоваться в широком спектре сценариев и способен удовлетворить большинство потребностей в прямом вещании без требования дополнительных плагинов. Вы можете загрузить его последнюю версию с [веб-сайта OBS](https://obsproject.com/download?spm=a2c4g.11186623.2.15.6aac1445JPlKR8).

### Варианты использования

| Сценарий | Описание |
| --- | --- |
| Онлайн-образование | Используйте настольную версию OBS или FFmpeg для публикации учебных материалов (поддерживается большинство форматов медиа) через RTMP в комнату TRTC. Студенты в комнате могут воспроизводить поток через SDK TRTC и видеть те же учебные материалы, при этом учитель управляет прогрессом проигрывания/скоростью или переключается между разделами. Отличная синхронизация на нескольких устройствах обеспечивает лучшее качество обучения. |
| Просмотр спортивных событий | Организаторы спортивных мероприятий предоставляют контент в виде потоков RTMP. Вы можете публиковать потоки в комнаты TRTC, чтобы пользователи в комнатах могли смотреть событие с ультранизкой задержкой. Благодаря возможности взаимодействия TRTC пользователи также могут разговаривать и обмениваться видео друг с другом на протяжении всего события. |
| Другое | Вы также можете использовать функцию публикации RTMP для реализации других приложений взаимодействия в реальном времени на основе потокового вещания. |

### Архитектура

Клиент RTMP — это модуль TRTC, который может взаимодействовать с другими клиентами TRTC. Задержка взаимодействия в нормальных условиях составляет менее 600 мс. Он также может использовать возможности TRTC, такие как запись и ретрансляция. Архитектура сети показана на рисунке ниже. **Не поддерживает извлечение потока RTMP из TRTC; поддерживает только отправку потока в TRTC.**

### URL-адреса публикации и воспроизведения

#### URL-адреса публикации

```
rtmp://intl-rtmp.rtc.qq.com/push/Room ID?sdkappid=Application ID&userid=User ID&usersig=Signature
```

- Основной домен — intl-rtmp.rtc.qq.com; резервный домен — rtmp.rtc-web.com. Если возникают проблемы с разрешением DNS основного домена, вы можете использовать резервный домен.
- Для публикации RTMP `appName` — это `push`.
- Замените "Room ID", "Application ID", "User ID" и "Signature" на их фактические значения.
- Для простоты мы поддерживаем только идентификаторы комнат строкового типа. Идентификатор комнаты может содержать цифры, буквы и подчеркивания и не может превышать 64 символа.

> **Предупреждение:** Чтобы воспроизвести поток RTMP на других клиентах TRTC, при входе в комнату убедитесь, что вы используете идентификатор комнаты строкового типа.

- Дополнительную информацию о способе создания `UserSig` см. в разделе [UserSig](https://www.tencentcloud.com/document/product/647/35166). **Убедитесь, что ваша подпись находится в периоде действия**.

**Пример:**

```
rtmp://intl-rtmp.rtc.qq.com/push/hello-string-room?sdkappid=140*****66&userid=******rtmp2&usersig=eJw1jdE********RBZ8qKGRj8Yp-wVbv*mGMVZqS7w-mMDQL
```

##### Шаг 3. Сконфигурируйте выход

Поскольку RTMP не поддерживает B-кадры, установите параметры кодирования видео следующим образом, чтобы удалить B-кадры.

1. Перейдите в **Controls** > **Settings** > **Output**.
2. Выберите **Advanced** для **Output Mode**. Рекомендуемый **Keyframe Interval** — **1** или **2**. Для **CPU Usage Preset** выберите **ultrafast**. Для **Profile** выберите **baseline**. Для **Tune** выберите **zerolatency**. И для **x264 Options** введите `threads=1`, затем нажмите **OK**.

> **Предупреждение:** Вам необходимо удалить B-кадры из потоков RTMP, иначе соединение будет разорвано после отправки. Чтобы добиться этого, выберите baseline для Profile.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/685ee210c6fe11edbe95525400088f3a.png)

##### Шаг 4. Установите параметры видео

Вы можете установить разрешение видео и частоту кадров в разделе **Video** параметров **Settings** . Разрешение определяет четкость видео, показываемого членам аудитории. Чем выше разрешение, тем четче видео. Частота кадров (кадры в секунду) определяет плавность воспроизведения. Типичная частота кадров находится в диапазоне 24 кадров в секунду до 30 кадров в секунду. Воспроизведение может заикаться, если частота кадров ниже 16 кадров в секунду. Видеоигры требуют более высокой частоты кадров и имеют тенденцию заикаться при частоте кадров ниже 30 кадров в секунду.

![](https://cloudcache.intl.tencent-cloud.com/cms/backend-cms/00d7736d7b7c11ed8e7e525400463ef7.png)

#### Использование FFmpeg для публикации потоков

Для публикации потоков RMTP с использованием команд FFmpeg или других библиотек RTMP необходимо использовать полный URL, видеокодек H.264 и аудиокодек AAC. Для формата контейнера рекомендуется FLV. Для GOP рекомендуется одна или две секунды.
Конфигурация параметров FFmpeg различается в зависимости от сценария, поэтому вам необходимо иметь некоторые знания о FFmpeg, чтобы использовать его для публикации потоков. В таблице ниже перечислены некоторые распространенные команды FFmpeg. Для получения дополнительных параметров см. [документацию FFmpeg](https://ffmpeg.org/ffmpeg.html).

##### Команды FFmpeg

```
ffmpeg [global_options] {[input_file_options] -i input_url} ... {[output_file_options] output_url}
```

##### Общие параметры FFmpeg

| Параметр | Примечание |
| --- | --- |
| -re | Читает вход с собственной частотой кадров. Обычно используется для чтения локальных файлов. |

Параметры для **output_file_options** включают:

| Параметр | Примечание |
| --- | --- |
| -c:v | Библиотека видеокодирования. Рекомендуется `libx264`. |
| -b:v | Видеобитрейт. Например, `1500k` означает 1500 Кбит/с. |
| -r | Частота кадров видео. |
| -profile:v | Профиль видео. Если вы установите его на `baseline`, B-кадры не будут кодироваться. Серверная часть TRTC не поддерживает B-кадры. |
| -g | GOP (интервал кадра ключа). |
| -c:a | Библиотека аудиокодирования. Рекомендуется `libfdk_aac`. |
| -ac | Количество звуковых каналов. Допустимые значения: `1`, `2`. |
| -b:a | Аудиобитрейт. |
| -f | Формат контейнера. Установите его на `flv`. Формат контейнера FLV требуется для публикации в TRTC. |

Ниже приведен пример чтения локального файла и публикации его в TRTC (обратите внимание, что для URL-адреса требуются кавычки):

```
ffmpeg -loglevel debug -re -i sample.flv -c:v libx264 -preset ultrafast -profile:v baseline -g 30 -sc_threshold 0 -b:v 1500k -c:a libfdk_aac -ac 2 -b:a 128k -f flv 'rtmp://intl-rtmp.rtc.qq.com/push/hello-string-room?userid=rtmpForFfmpeg&sdkappid=140xxxxxx&usersig=xxxxxxxxxx'
```

## Часто задаваемые вопросы

### Ошибка потокового вещания

Общие причины

- Пакет не куплен или истек срок действия.
- Неправильная или истекшая подпись.
- Потоковое вещание с B-кадрами (на панели показывается как "поток заканчивается через одну секунду"); следует установить кодирование baseline.

Другие причины

- Если поток отправляется встроенным аппаратным устройством, URL-адрес может быть обрезан.
- Потоковое вещание с H.265; следует использовать H.264.
- Слишком большой размер части на клиенте; рекомендуется установить размер части на 1360.

### Задержка и артефакты на экране

Проверьте [панель мониторинга](https://www.tencentcloud.com/document/product/647/39070) Tencent Cloud Real-Time Communication, чтобы контролировать стабильность частоты кадров потокового вещания. Если она стабильна, проблема, вероятно, находится на стороне проигрывателя — пожалуйста, изучите проигрыватель. **Если частота кадров нестабильна**, рассмотрите следующее:

- Проверьте, находятся ли ЦП и память локального клиента потокового вещания под высокой нагрузкой. Если вы используете OBS для потокового вещания, наблюдайте за строкой состояния в нижней части для получения информации о потерянных кадрах, сети, ЦП и частоте кадров.
- Проверьте, достаточна ли локальная **пропускная способность сети**. Выполните ping по домену потокового вещания, чтобы наблюдать RTT; используйте [инструмент диагностики сети](https://itango.tencent.com/app/data/huatuo) для проверки домена потокового вещания и пропускной способности, в идеале достигнув 10 Мбит/с.
- Клиент потокового вещания может попытаться снизить битрейт и частоту кадров, чтобы уменьшить нагрузку на клиент; см. параметры OBS в основном тексте, рекомендуется установить битрейт на 1500 Кбит/с для 720p.

### Высокая задержка

- Если клиент, извлекающий поток, использует роль якоря TRTC, задержка обычно ниже, чем у роли аудитории TRTC. Если это не роль якоря, измените на нее и сравните, чтобы увидеть, есть ли улучшение.
- Локальное кодирование и сеть значительно влияют на конец потокового вещания. Попробуйте различные платформы для тестирования; если используете OBS, рассмотрите возможность потокового вещания в системе Windows; выполните ping по домену потокового вещания, чтобы наблюдать RTT.

### Поток не видна на другом конце

Конец потокового вещания использовал строковый номер комнаты, но конец извлечения использовал числовой номер комнаты; измените конец извлечения, чтобы использовать строковый номер комнаты для входа в комнату.

### Частые отключения и повторное потоковое вещание

- Дублирование имени пользователя, вызывающее взаимное исключение; убедитесь, что userid является глобально уникальным в одном sdk
